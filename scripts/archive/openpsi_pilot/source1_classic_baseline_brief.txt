## Why this source ‚Äî cluster context

The 2026-04-26 ECAN/Attention cluster pilot (Source 4 V0-1, see `scripts/archive/ecan_pilot/source4_pre_split_monorepo/findings_reconciled_crossmodel.txt`) established that **OpenPsi was a confirmed third executable ECAN consumer** at `b31c7e3` alongside the URE STI source-selection hook and Ghost/NLP-fuzzy. Specifically:

- `opencog/openpsi/action-selector.scm:63` ‚Äî `(* (cog-confidence RULE) (cog-mean RULE) (cog-av-sti RULE))` weights candidate rules by `strength √ó confidence √ó STI`
- Pickaxe `e5bae708fa5b1e6a5a18ef6c2e4ca6a40f981d98` (2017-11-09, Amen Belayneh, "Cleanup openpsi action-selector functions") ‚Äî STI weighting in OpenPsi has existed since at least 2017
- Verified by orchestrator via `git grep b31c7e3 -- opencog/openpsi/` (the V0-1 disambiguator that overturned Gemini's C++-only NO-COUPLING verdict)

This brief opens an OpenPsi-cluster archaeology pass to determine:

1. **Architectural baseline** ‚Äî What does the canonical Classic OpenPsi look like at `b31c7e3` (peak pre-AttentionBank-removal state)? Action-selection, rule firing, dynamics subsystems, AtomSpace/AttentionBank/PLN coupling.
2. **Tradition placement** ‚Äî Per the five-tradition map in `CLAUDE.md`, is OpenPsi a Tradition 2 sub-branch (like ECAN's iCog-Labs-Dev MeTTa port relative to Classic ECAN), an independent tradition, or a cross-cluster control layer? The MeTTa port question is Source 2's job; Source 1 sets the baseline.
3. **ECAN coupling depth** ‚Äî Is `cog-av-sti` rule weighting representative of OpenPsi-ECAN integration, or are there other coupling surfaces (modulators driving STI, dynamics affecting AttentionBank, etc.)?
4. **PLN coupling depth** ‚Äî Was OpenPsi a PLN consumer (rule firing via Implicator, control of inference goals), or did it use the URE only structurally? This question matters because the "missing PLN inference-control layer" narrative from the ECAN pilot may have a parallel OpenPsi story.

## Audit checklist (questions for extraction agents)

### A. Architecture inventory (must answer)

A1. **Action-selection algorithm.** Read `action-selector.scm` end-to-end at `b31c7e3`. What is the rule-selection function? What inputs (TV, STI, modulators)? What output (single rule, ranked list, sampling)? File:line citations required.

A2. **Rule firing pipeline.** Trace `OpenPsiImplicator.cc` and `OpenPsiSatisfier.cc`: how does a selected rule become an executed action? What's the role of `_satisfiability_cache`? What's the AtomSpace effect of firing? Cite specific functions.

A3. **Demand/Goal model.** Read `demand.scm` + `rule.scm`: how are demands (Psi theory drives) declared? How do they connect to rules? Is there a numeric urge/satisfaction value, or is this purely structural?

A4. **Dynamics subsystem.** Inventory `dynamics/`: what does each of `emotion.scm`, `modulator.scm`, `sec.scm`, `event.scm`, `interaction-rule.scm`, `updater.scm` do? Is the updater a real loop or a stub? Are interaction rules driving runtime state changes or just declarative templates?

A5. **C++ ‚Üî Scheme boundary.** What does `OpenPsiSCM.cc` expose to Scheme? Which C++ functions are reachable from `(use-modules (opencog openpsi))`? Cite the binding table.

A6. **Python bindings.** What does the Python wrapper (added in PR #3501, expanded in PR #3517 + PR #3655) actually expose? Is OpenPsi callable end-to-end from Python at `b31c7e3`?

### B. ECAN/STI coupling (V0-1 carry-forward)

B1. **Verify `action-selector.scm:63` STI weighting at `b31c7e3`.** Use `git show b31c7e3:opencog/openpsi/action-selector.scm` to confirm `(* (cog-confidence RULE) (cog-mean RULE) (cog-av-sti RULE))` is present. Quote the surrounding context (function definition, comment block).

B2. **Other STI/AttentionalFocus consumers.** Grep `b31c7e3` openpsi/ for ALL of: `cog-av-sti`, `cog-av-lti`, `cog-stimulate`, `cog-set-sti!`, `cog-set-lti!`, `cog-af`, `cog-af-boundary`, `bank->`, `AttentionBank`, `AttentionValue`, `getSTI`, `setSTI`, `atom_is_in_AF`. Per the ECAN Source 4 V0-1 mandatory rule, both C++ AND Scheme tokens. Report file:line for each hit.

B3. **AttentionBank-optional path.** Commit `f091e9d0d` ("Make guile modules load if (opencog attention-bank) isn't installed") and PR #3572 (`define-have-openpsi` `HAVE_OPENPSI`) imply OpenPsi can build without AttentionBank. Trace the `HAVE_OPENPSI` / attention-bank-optional logic at `b31c7e3`. Is the STI weighting silently disabled when AttentionBank is absent, or does the code still compile-link with stub AV functions?

B4. **Stimulation/decay flow.** Does OpenPsi *write* STI (via `cog-stimulate` or `cog-set-sti!`) anywhere, or only read it via `cog-av-sti`? If write-side coupling exists, what triggers it?

### C. PLN/URE coupling

C1. **URE usage.** Commit `0acdb0e77` ("Rename rule-engine by ure") signals OpenPsi was at one point coupled to the URE rule engine. At `b31c7e3`, does OpenPsi instantiate URE chainers, register rules with URE, or use URE for any inference step? Grep for `ure-`, `BackwardChainer`, `ForwardChainer`, `cog-fc`, `cog-bc` in openpsi/.

C2. **PLN inheritance.** Does OpenPsi rule firing depend on PLN-style truth values (strength √ó confidence multiplication)? The V0-1 weighting `strength √ó confidence √ó STI` is multiplicative-confidence-style ‚Äî is that PLN-derived or independent?

C3. **Implicator vs. Satisfier semantics.** OpenPsiImplicator.cc/h vs. OpenPsiSatisfier.cc/h ‚Äî what's the semantic distinction? Is Implicator more PLN-like (forward inference) and Satisfier more pattern-match (backward)?
