## Cross-model consensus (both Codex and Gemini agree)

### Architecture (A1-A6)

- Hybrid Scheme/C++ implementation: 27 files at `b31c7e3` — README/CMake, four C++ class pairs (`OpenPsiImplicator`, `OpenPsiRules`, `OpenPsiSatisfier`, `OpenPsiSCM`), seven top-level Scheme files (`action-selector`, `control`, `demand`, `main`, `openpsi`, `rule`, `utilities`), and ten files in `dynamics/` (`emotion`, `modulator`, `sec`, `event`, `interaction-rule`, `mock-interaction-rules`, `dynamics`, `updater`, `utilities`, `CMakeLists`).
- Rule structure: `ImplicationLink TV (AndLink context action) goal`. C++-side rule cache and PatternLink-based satisfiability indexing in `OpenPsiRules::add_rule`.
- Implicator/Satisfier separation: `OpenPsiImplicator::check_satisfiability` runs PatternLink search; `OpenPsiImplicator::imply` requires prior satisfiability check and instantiates the action; satisfaction caching in `OpenPsiSatisfier::grounding`.
- Dynamics subsystem: separate updater loop with modulators (arousal, valence, resolution-level, etc.), SECs (novelty, goal-relevance, pleasantness, etc.), event detection, interaction rules using `PredictiveImplication`, exponential-curve `adjust-psi-var-level`. Both models flag this as runtime-shaped but research-grade.
- Python wrapper: present in `opencog/cython/opencog/openpsi.{pyx,pxd}` with `set_action_selector` extension hook. Test `test_run_openpsi` skipped at `b31c7e3` (Codex C1-35, OpenCog issue 3532).

### PLN/URE coupling (C1-C3)

The only active URE-module import is `demand.scm` importing `(opencog ure)`, traced to commit `0acdb0e77` ("Rename rule-engine by ure") rather than active integration.

- Gemini cites `URE:attention-allocation` parameter as the coupling surface — orchestrator note: this is the same dead-config switch already audited in ECAN Source 4 (`pln-config.scm:62` set to `0`); not OpenPsi-specific evidence.

## Cross-model dissent (V1-N)

### V1-1 — OpenPsi STI runtime-coupling status (CRITICAL DISSENT)

**The headline reconciliation question. Resolution affects ECAN Source 4 V0-1 carry-forward and CLAUDE.md cluster-narrative-deciding finding #3.**

**Codex C1-2 / C1-4 / C1-5 / C1-55 / C1-60 / C1-85:**
> "The default action-selection loop is not STI-weighted... `rule-sca-weight` is the only OpenPsi-local `cog-av-sti` hit at b31c7e3... not wired into the default loop and has no packaged caller in `opencog/openpsi/`... Therefore the ECAN Source 4 phrase 'OpenPsi weights rules by STI' should be narrowed for Source 1: Classic OpenPsi exposes an STI-weighted selector helper and optional AttentionBank dependency, but the b31c7e3 default runtime path is satisfiability-based, not STI-weighted... Classification: PRESENT-AS-OPTIONAL-HELPER / NOT-DEFAULT-RUNTIME."

**Gemini Section 2 (B1-B4):**
> "Status: Strong Runtime Coupling (2017-2018 peak). Verified Formula: `action-selector.scm:63` explicitly weights rules by `(* (cog-confidence RULE) (cog-mean RULE) (cog-av-sti RULE))`."

**[Claude orchestrator, repo-verified]** Direct grep against `b31c7e3` for callers:

```
$ git grep b31c7e3 -- "rule-sca-weight"
b31c7e3:opencog/openpsi/action-selector.scm:57:(define (rule-sca-weight RULE)
b31c7e3:opencog/openpsi/action-selector.scm:59:  rule-sca-weight RULE
```

Two hits only: the `define` and the docstring self-reference. No call sites. Same pattern for `pick-from-weighted-list` and `sort-by-weight` — they exist as helpers but `pick-from-weighted-list` is itself uncalled outside its own definition. The whole weighted-selection chain (`pick-from-weighted-list` → `sort-by-weight` → arbitrary `WEIGHT-FUNC`, of which `rule-sca-weight` is one) is **dead code at `b31c7e3`** by caller analysis.

Verified default-selector path at `main.scm:54-59` (verbatim from `git show b31c7e3:opencog/openpsi/main.scm`):

```scheme
; Assign a default action-selector
(psi-set-action-selector! component
  (ExecutionOutput
    (GroundedSchema "scm: psi-get-satisfiable-rules")
    (List component))
)
```

`psi-get-satisfiable-rules` returns ALL satisfiable rules; no STI weighting, no scoring, no ranking. The default OpenPsi runtime path is satisfiability-only.

**Critical follow-up** — does ANY consumer at `b31c7e3` install a custom action-selector that uses `rule-sca-weight`? Checked `opencog/ghost/matcher.scm:284` (Ghost installs its own action-selector via `psi-set-action-selector!`):

```
$ git show b31c7e3:opencog/ghost/matcher.scm | sed -n '115,150p'
  ; Calculate the weight of the rule R [Wr]
  (define (calculate-rule-weight R)
    (define strength
      (if (> strength-weight 0)
        (* strength-weight (cog-mean R)) 1))
    (define context
      (if (> context-weight 0)
        (* context-weight
          (assoc-ref context-alist (psi-get-context R))) 1))
    (define sti
      (if (> sti-weight 0)
        (* sti-weight (cog-av-sti R)) 1))
    ...
    (define weight (* strength context sti urge))
```

Ghost installs `(GroundedSchema "scm: ghost-find-rules")` (`matcher.scm:284`) which uses Ghost's own `calculate-rule-weight` — a parallel STI-weighted formula with `strength × context × sti × urge` factors, all knob-controllable via `*-weight` parameters. **Ghost does NOT call OpenPsi's `rule-sca-weight`.**

**Resolution: Codex's PRESENT-AS-OPTIONAL-HELPER / NOT-DEFAULT-RUNTIME verdict is correct.** Gemini's "Strong Runtime Coupling (2017-2018 peak)" verdict is **wrong by methodology**: Gemini verified the formula's textual presence at line 63 but did not verify any caller chain. The same lesson as ECAN Source 3 V0-1 (pickaxe before "dead code") applies here in reverse: **caller-analysis before "actively coupled" claims**. Defined-but-uncalled code is not runtime coupling, even when the definition is syntactically valid and conceptually sound.

**Reconciled status: PRESENT-AS-OPTIONAL-HELPER / NOT-DEFAULT-RUNTIME.** Classic OpenPsi at `b31c7e3` defined an STI-weighted scoring helper (`rule-sca-weight = strength × confidence × STI`) and made `(opencog attention-bank)` an optional-load dependency, but did not invoke that helper anywhere in its own runtime path. The default OpenPsi runtime is satisfiability-only.

**The cluster-narrative correction:** OpenPsi at `b31c7e3` was an *ECAN-coupling enabler* (it provided the action-selector seam through which Ghost installed its STI-weighted selector) but not itself a default ECAN consumer. The actual STI-weighted rule selection running at runtime in 2017-2019 OpenCog was Ghost's, not OpenPsi's.

This nuance does not invalidate the ECAN Source 4 broader-OpenCog finding ("AttentionBank had real non-PLN executable consumers") — that remains true via Ghost, NLP fuzzy, and the Python web API. But it sharpens the OpenPsi-specific characterization from "weighted rules by STI" to "exposed an STI-weighting helper used by Ghost's parallel formula via the OpenPsi action-selector seam, with no OpenPsi-internal default usage".

## Cluster-position summary (Source 1)

1. **Architectural baseline locked.** Classic OpenPsi at `b31c7e3` is a real C++/Scheme runtime: rule registry + satisfiability cache + Implicator/Satisfier dispatch + Scheme component loop + dynamics subsystem (modulators, SECs, interaction rules). Not a stub.

2. **Tradition placement: Tradition 2 (OpenCog Classic Attention/Motivation lineage).** Cross-cuts ECAN through the optional STI-helper seam, PLN through TV/rule-structure idioms, Ghost through the action-selector plug. Not its own tradition.

3. **ECAN coupling: PRESENT-AS-OPTIONAL-HELPER / NOT-DEFAULT-RUNTIME.** Defined an STI-weighting formula `strength × confidence × STI` but never called it; default selector returns all satisfiable rules. The actual ECAN-driven action selection at `b31c7e3` lived in Ghost, which used OpenPsi's component plumbing without using OpenPsi's STI math.

4. **PLN/URE coupling: NONE direct.** Conceptual ancestry only — `ImplicationLink` rule structure justified by PLN-compatibility argument in README; `psi-context-weight` cites PLN conjunction formula as a local-helper basis. No URE chainer calls.

5. **Volatility surface: substantial.** README/code drift, satisfiability-cache thrashing in mid-2019, AttentionBank optional-load path added 2018, action-selector simplified by `e5bae708` 2017-11. The pinned `b31c7e3` is the eve-of-AttentionBank-removal snapshot of an actively churning subsystem.

6. **Cohort: Amen Belayneh dominant** (294 of 564 commits, 52%; both `@hansonrobotics.com` and personal addresses → Hanson/SingNet cross-org overlap). Eddie Monroe and Linas Vepstas as second/third tier. Significant 2017-period MindCloud copyright presence on C++ files.

7. **New methodology rule (Source 1 V1-1 driven):** caller-analysis before "actively coupled" claims.
