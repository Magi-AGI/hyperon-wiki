## Audit checklist (questions for extraction agents)

### A. Paper identity disambiguation (V2-2)

A1. **Identify all OpenPsi/Psi/MetaMo papers in the wiki RawData and code citations.** Cross-reference the RawData parent IDs and titles. For each paper, determine: authors, year, venue, link/source, **and whether the full text is available in RawData chunks**.

A2. **Verify the relationship between `Publications+OpenPsi Cognitive Model` (ID 3093, AGI-11 2011) and `RawData+Publications+Openpsi Zhenhua` (ID 3827, "openpsi-zhenhua.pdf").** Are these the same paper, related papers, or different? Check by reading the openpsi-zhenhua chunks (3828, 3829) and matching content/title/authors.

A3. **Locate the 2013 paper.** Codex's citation: "OpenPsi: A Novel Computational Affective Model and its Application in Video Games by Goertzel, Ben et al. (2013)". Search:
   - `RawData+Publications+OpenPsi Cognitive Model` (ID 3168) full text -- does the 2011 paper reference an in-progress 2013 follow-up?
   - `Publications+PSI Affective Dynamics` content -- does this 2011 Cognitive Systems Research paper match the 2013 title's "Computational Affective Model" framing? (The 2013 paper may be a re-titled or extended version.)
   - External web (NOT required of the audit; flag if web search would be useful).

A4. **Map "Ben's Paper" code citations to specific paper(s).** `demand.metta:150-168` says "according to Ben's Paper"; `demand-updaters.metta:1-76` says "Based on OpenPsi ben paper". Are these references to the same paper? Are the formulas in those files traceable to one of the wiki papers?

### B. Equation verification (V2-3 driver)

B1. **Locate eq. 10-13 in the cited paper.** Once A1-A3 settles which paper is the source, find equations 10, 11, 12, 13 in that paper. Quote them verbatim (LaTeX or Unicode rendering).

B2. **Compare paper eq. 10-13 to MeTTa code helpers.** For each of:
   - `ActivationHelper(c, e) = c/(c+0.5) * e/(0.05+e)`
   - `SecuringThresholdHelper(c, i) = c/(c+0.05) * i^3`  (note: `c` here is `certanity`/certainty)
   - `SelectionThresholdHelper(c) = fuzzy_equal(c, 1, 15)`

   Determine: **literal match** (same formula), **structural match** (same shape, different constants), **inspired** (same idea, different math), **divergent** (paper says X, code does Y). Cite paper page/equation number for each comparison.

B3. **Verify `calculateDemandSatisfaction` against paper.** Same comparison framework -- is the piecewise function (with `0.8 + 0.2*random()` in-range, `fuzzy_equal` at-min, `0.9 + 0.1*random()` above-max) literal/structural/inspired/divergent? **The random terms are particularly suspect** -- does any OpenPsi paper specify random-noise in demand satisfaction, or is this an engineered addition?

B4. **Verify demand-updater formulas (`demand-updaters.metta:1-76`).** Same framework -- `certaintyDemandUpdater`, `affiliationDemandUpdater`, `competenceDemandUpdater` against paper definitions of those demand types.

B5. **Hardcoded value check (Codex C2-63).** The MeTTa code hardcodes `certanity=0.5, integrity=0.5` in `modulatorUpdaterAgent`. The paper presumably has these as time-varying state, not constants. Is the hardcoding documented/justified anywhere, or is it a partial-implementation marker?

### C. Tradition placement decision (V2-3)

C1. **Based on B1-B5 evidence, classify the MeTTa port's relationship to the source paper(s).** Use the following criteria:
   - **Tradition 6: paper-faithful re-implementation** = formulas literal-match (or are deliberate corrections of paper typos), parameters time-varying as paper specifies, semantically faithful to paper's algorithm.
   - **Tradition 2 sub-branch: faithful Classic-surrogate port** = formulas match Classic OpenPsi C++/Scheme rather than the paper, even where Classic deviated from paper.
   - **Hybrid** = some formulas paper-faithful, others Classic-faithful.
   - **Independent re-derivation** = formulas in code don't match either paper or Classic; engineered for MeTTa-port purposes.

C2. **Document tradition decision rationale.** Whatever the C1 classification, provide the supporting evidence -- cite specific paper equations and specific code lines.

C3. **Cross-cluster comparison.** ECAN cluster pilot established Tradition 2 sub-branch behavior for the metta-attention port (faithful Classic-surrogate replication). How does OpenPsi's MeTTa port compare? If different, is the difference attributable to (a) different cohort priorities, (b) different paper-anchor availability, (c) different code-quality at the Classic baseline being ported, or (d) deliberate community choice to "fix" what Classic got wrong?

## Methodology rules (per CLAUDE.md standing protocols + Source 1/2 emergence)

1. **Read-only.** No working-tree modifications, no wiki writes from extraction agents.
2. **Cross-model triangulation.** Codex + Gemini extraction passes; Claude orchestrator cross-checks; reconciliation file synthesizes.
3. **Both API layers** (carried forward from ECAN Source 4 V0-1; primarily code-side rule, but applies if any Scheme/MeTTa paper-citation comments exist).
4. **Pickaxe before "dead code" claims** (carried forward from ECAN Source 3 V0-1).
5. **Caller-analysis before "actively coupled" claims** (carried forward from OpenPsi Source 1 V1-1).
6. **Bidirectional fork-divergence check** (Source 2 V2-1 emergence) -- applies if Source 3 surfaces forked or alternative paper versions.
7. **Paper-equation literal verification.** Quote paper equations verbatim (LaTeX or Unicode). Cite paper section/equation number AND code file:line. Distinguish literal-match / structural-match / inspired / divergent -- do NOT collapse the distinction.
8. **Do NOT trust code comments as source attribution.** Comments like `;; calculate demand satisfaction according to Ben's Paper` or `;; align ... with OpenPsi paper (eq. 10-13)` indicate INTENT, not verified attribution. Always match the formula against the paper text directly. If the paper text doesn't have the cited equation number or doesn't match the formula, the code comment is wrong (or the paper has been revised). Report mismatches as findings, not as paper-faithful confirmation.
8. **RawData fidelity** (per CLAUDE.md): if a paper RawData has typos/scan-OCR errors, do NOT clean them in the audit; quote verbatim and note OCR limitations separately.
9. **Verbatim quotes for critical claims.** Code file:line + paper page/equation cite for any paper-vs-code claim.
