## Three dissents (V4-N)

### V4-1. Caller-analysis time-indexing -- the 2016 Hanson master STI-default path

**Codex (winning):** The brief's pre-loaded "zero caller" expectation, inherited from Source 1 V1-1, is FALSE for Hanson master at the 2016-06-24 snapshot. In `opencog-hansonrobotics/opencog/openpsi/action-selector.scm`, the default selector at lines 121-127 and the per-demand variant at lines 170-176 both call `most-important-weighted-atoms` -- which uses `(* (tv-conf a-stv) (tv-mean a-stv) sti)` at `utilities.scm:87-90` -- WHEN `cog-af-boundary` is not 0 or 1. The non-STI fallback (`most-weighted-atoms`) is gated on the degenerate-AF-boundary condition. Per `main.scm:50-91`, `psi-step` iterates demands and calls `psi-select-rules-per-demand`, which routes to this default selector when no custom action-selector is installed. So this is a real default-runtime STI-coupling path at the 2016 Hanson snapshot, not a dead helper.

Codex pickaxe identified the lifecycle:
- `8ab0e8f81` Amen Belayneh, 2016-05-10, "add importance into the utility function" -- adds the STI default branch
- `9f2697859` Linas Vepštas, 2016-11-24, "General cleanup of the action selector" -- removes the conditional default STI branch from `psi-default-action-selector` and `psi-default-action-selector-per-demand`
- `e5bae708f` Amen Belayneh, 2017-11-08, "Cleanup openpsi action-selector functions" -- reintroduces `rule-sca-weight` as `strength * confidence * STI` in upstream, but only as a helper definition with no runtime caller

**Gemini's position:** Gemini's report does not raise this finding. Its analysis of the OpenPsi seam pattern centers on the bypassing of OpenPsi's quantitative economy in `ros-behavior-scripting` (Q6), without examining the action-selector default path in the 2016 Hanson opencog snapshot.

**Orchestrator verification:** Verified directly. `Read` of `opencog-hansonrobotics/opencog/openpsi/action-selector.scm:121-127` shows verbatim:

```scheme
(define (choose-rules)
    ; NOTE: This check is required as ecan isn't being used continuesely.
    ; Remove `most-weighted-atoms` version once ecan is integrated.
    (if (or (equal? 0 (cog-af-boundary)) (equal? 1 (cog-af-boundary)))
        (most-weighted-atoms (psi-get-all-satisfiable-rules))
        (most-important-weighted-atoms (psi-get-all-satisfiable-rules))
    )
)
```

The same conditional at lines 170-176 in `psi-default-action-selector-per-demand`. The author comment "Remove `most-weighted-atoms` version once ecan is integrated" demonstrates intent: the STI branch was the *intended default*, and the non-STI fallback was the temporary path. `Read` of `utilities.scm:87-90` confirms the STI-weighted formula. `git show --no-patch` on all three commit SHAs confirms author + date + message verbatim as Codex reported.

**Verdict:** Codex correct. The Source 1 V1-1 "enabler-not-default" framing is correct for the 2019 Source 1 baseline (singnet `b31c7e3`) but is INCORRECT for the 2016 Hanson master snapshot. Caller-analysis must be time-indexed; "called/uncalled" claims should always include the commit/date snapshot. This is a new methodology rule (Source 4 emergent rule).

**Implication for ECAN cluster pilot finding #2 in CLAUDE.md:** That finding currently states "The only realized executable hook in OpenCog history was a narrow URE forward-chainer STI source-selection mechanism: wired by Misgana Bayetta in commit `0a0b09912` (2016-03-09), unwired by Nil Geisweiller in commit `0b744dbab` (2018-10-23)..." This is now KNOWN INCOMPLETE. The OpenPsi default action-selector also had an STI-weighted path from 2016-05-10 (`8ab0e8f81`) to 2016-11-24 (`9f2697859`). Cluster-narrative addendum will be needed during cluster-close, similar in shape to ECAN Source 4 V0-1 addendum.

## Source 4 emergent methodology rules

**Rule 9 (new): Caller-analysis must be time-indexed.** "Called/uncalled" claims about a function/formula should always include the commit/date snapshot at which the analysis was performed. Source 1 V1-1's "rule-sca-weight has zero callers" is correct at the 2019 singnet `b31c7e3` snapshot but is INCORRECT at the 2016 Hanson `aec9b1f` snapshot, where the same formula was on the default action-selector path. Future archaeology should pickaxe the function definition's lifecycle (when added, when removed from callers, when reintroduced as helper) before classifying.

**Rule 10 (new, derived from V4-1): Codebase-staleness affects pickaxe interpretation.** A "stale" fork that has not pulled upstream changes is NOT equivalent to "an isolated tradition." It may simply be a frozen snapshot of a particular upstream era. Diff against `upstream/master` and check `git log fork ^upstream/master` and `git log upstream/master ^fork` BEFORE labeling a fork as a divergent tradition. (This generalizes Source 2 V2-1's bidirectional-fork-divergence rule to fork-vs-upstream-snapshot identification.)
