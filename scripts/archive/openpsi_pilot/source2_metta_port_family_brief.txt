## Why this source — cluster context (carry-forward from Source 1)

OpenPsi cluster pilot Source 1 (2026-04-27 V1-1 reconciliation, file `scripts/archive/openpsi_pilot/source1_classic_baseline/findings_reconciled_crossmodel.txt`) established the corrected baseline for Classic OpenPsi at `b31c7e3`:

> **Classic OpenPsi was an ECAN-coupling enabler, not itself a default executable ECAN consumer.** It defined an STI-weighting helper `rule-sca-weight` at `opencog/openpsi/action-selector.scm:57-64` that had **zero callers** in the OpenPsi runtime. The default component selector at `main.scm:54-59` was `psi-get-satisfiable-rules` (returns all satisfiable rules, no STI scoring). Ghost was the actual STI consumer; it installed its own STI-weighted action-selector at `opencog/ghost/matcher.scm:117-130, 284` via OpenPsi's `psi-set-action-selector!` seam — a parallel `strength × context × sti × urge` formula, not a call into OpenPsi's helper.

Source 1 also locked in additional baseline facts:
- 27-file Scheme/C++ implementation at `b31c7e3`; ~52% Amen Belayneh contribution.
- Implicator/Satisfier/Rules C++ pipeline + Scheme component loop + dynamics subsystem (modulators, SECs, interaction rules).
- Conceptual PLN/URE ancestry (rule structure, TV idioms) but no runtime invocation of either.
- README/code drift: README documents weighted-selection that doesn't exist in code; `openpsi.scm` exports symbols that aren't defined.

**Source 2 question:** What survived the MeTTa transition?

## Audit checklist (questions for extraction agents)

### A. Architecture mapping (Classic → MeTTa)

A1. **File-level mapping.** For each Classic OpenPsi file (27 files at `b31c7e3`), identify the MeTTa-port equivalent or document its absence:
   - `OpenPsiImplicator.{cc,h}` → `main/rules/implicator.metta` (verify semantic equivalence — same role of "if context satisfiable, fire action"?).
   - `OpenPsiSatisfier.{cc,h}` → ? (Classic's satisfiability checker; MeTTa equivalent unclear from file inventory).
   - `OpenPsiRules.{cc,h}` → `main/rules/rule.metta` (?).
   - `OpenPsiSCM.{cc,h}` → no equivalent (SCM bindings layer is C++↔Scheme; MeTTa port is MeTTa-native).
   - `action-selector.scm` → `mind-agents/action-planner/action-planner-v{1,2}.metta` (?). Note name change from "selector" to "planner".
   - `control.scm`, `demand.scm`, `main.scm`, `openpsi.scm`, `rule.scm`, `utilities.scm` → ?
   - `dynamics/{emotion, modulator, sec, event, interaction-rule, mock-interaction-rules, dynamics, updater, utilities}.scm` → mostly fragmented across `main/{emotion, modulator}/` and `mind-agents/{demand-updater, modulator-updater, feeling-updater, perception-updater, monitor-changes}/` — verify mapping.

A2. **Net-new structure.** Inventory what exists in the MeTTa port that has NO Classic equivalent: `psi-utilities/`, `feed-back-loop/`, `mind-agents/feeling-updater/`, `mind-agents/perception-updater/`, `mind-agents/monitor-changes/`, the multi-planner architecture (v1/v2/aStar/thompson/ts-ucb), `use-cases/{curious-agent, qwestor}/`. For each, what is its function and what motivates its existence?

A3. **Net-removed structure.** What from Classic OpenPsi has NO MeTTa-port equivalent: `OpenPsiSatisfier`, `OpenPsiSCM` C++/Scheme bindings layer, `dynamics/sec.scm` (SECs — sensitivity-evaluation-components), `dynamics/event.scm`, `dynamics/interaction-rule.scm` with `PredictiveImplication`, the satisfiability cache mechanism. What's been lost?

A4. **Type system.** `main/types.metta` exists in MeTTa port; Classic had no type module (Scheme/C++). What does the type schema constrain — rule shape, demand shape, modulator shape? Does it enforce constraints the Classic implementation only documented in comments?

### B. Action-selector seam survival (Source 1 V1-1 carry-forward)

B1. **Did the configurable selector seam survive?** Classic's distinguishing feature was the ability for clients (Ghost) to install custom selectors via `psi-set-action-selector!`. In the MeTTa port, is there a parallel pluggable selector point? If yes, where? If no, what replaced it?

B2. **Multiple planners — pluggable or hardcoded?** The MeTTa port has 5 planner variants (`action-planner-v1`, `action-planner-v2`, `aStar-action-planner`, `thompson-sampling/`, `ts-ucb/`). Are these:
   (a) Selectable at runtime via a planner-selection seam (the "enabler" survived),
   (b) Hardcoded for specific use-cases (curious-agent uses one, ping-pong uses another, etc.),
   (c) Different versions where only the latest is canonical and earlier are dead code?

B3. **STI/AttentionalFocus presence.** Per the both-API-layers methodology rule, grep the MeTTa port for: `cog-av-sti`, `cog-stimulate`, `cog-af`, `cog-set-sti!`, `cog-set-lti!`, plus MeTTa-equivalents (`get-sti`, `set-sti`, `bind-af`, `attention-bank`, `attentional-focus`, `STI`, `LTI`, `AttentionValue`). Report file:line for every hit. Source 1's caller-analysis methodology applies — is each hit a definition, a runtime invocation, a dead helper, or test-only?

B4. **AttentionBank optional-load equivalent.** Classic OpenPsi at `b31c7e3` had `(resolve-module '(opencog attention-bank) #:ensure #f)` to be loadable without AttentionBank. Does the MeTTa port have any analogous "optional ECAN" architecture, or is ECAN integration just absent entirely (which would match the Source 1 V1-1 finding that OpenPsi was an enabler not a consumer)?
