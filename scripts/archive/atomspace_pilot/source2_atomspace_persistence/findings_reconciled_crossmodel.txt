SECTION A — STORAGE NODE / API SURFACE INVENTORY
================================================================================

R2.A1 StorageNode = CANONICAL classical persistence interface.
      C++ Atomese object backed by BackingStore virtual interface;
      exposed through Scheme message/value APIs.
      Lifecycle: open / open_read_only / close / connected / create /
      destroy / erase.
      Read/write: fetch_atom, fetch_value, fetch_incoming_set,
      fetch_incoming_by_type, store_atom, store_value, store_atomspace,
      remove_atom.
      Cf. Codex C2-A1-A6. atomspace-storage StorageNode.h:42, 81-114,
      169-341; BackingStore.h:54, 76-112; persist.scm:127-1235.

R2.A2 ProxyNode family is part of the canonical ecosystem (Codex-only
      enumeration; Gemini did not surface):
      NullProxy / ReadThruProxy / SequentialReadProxy / WriteThruProxy /
      ReadWriteProxy / WriteBufferProxy / CachingProxy / DynamicDataProxy.
      Architecturally relevant: these can mirror writes, load-balance
      reads, buffer writes, cache reads, or compute dynamic values.
      Cf. Codex C2-A8, F3. storage_types.script:41-77.
      Cluster relevance: ProxyNode composition could provide the
      Decko-aware adapter layer atomspace-pgres lacks, IF combined with
      a custom backing store.

R2.A3 StorageNode is asynchronous; barrier matters.
      Cf. Codex C2-A7, F2. StorageNode.h:152.

R2.A4 FileStorageNode is the minimal local persistence reference.
      Implements only some StorageNode APIs; whole AtomSpaces can be
      read/written and individual atoms can be written; complex queries
      not supported.
      Cf. Codex C2-A9.

SECTION G — atomspace-bridge / BridgeStorageNode VERDICT
================================================================================

R2.G1 BridgeStorageNode = BRIDGE-ADAPTER (per brief vocabulary).
      Currently the closest existing Decko-applicable seam.
      Cf. consensus across Codex C2-G1-G12 and Gemini Section G.

R2.G2 CURRENT IMPLEMENTATION STATUS:
      - READ/IMPORT: implemented and functional. `cog-bridge-load-tables`
        and `cog-bridge-load-rows` Scheme primitives work.
      - WRITE/UPDATE: EMPTY STUBS. `storeAtom`, `removeAtom`, `storeValue`,
        `updateValue`, `loadValue`, `loadType`, `loadAtomSpace`,
        `storeAtomSpace` all have empty bodies.
      - README claims "should allow both the read and the update" — this
        is FUTURE INTENT, not current capability.
      Cf. Codex C2-G8, G9. SQLReader.cc:395-425 vs README.md:23-26.

R2.G3 SCHEMA-DISCOVERY MECHANISM (consensus):
      Dynamic introspection of PUBLIC schema via:
      - `pg_tables WHERE schemaname = 'public'` (Codex C2-G4)
      - `information_schema.columns` (Gemini)
      Builds SignatureLink representations of SQL tables; routes
      `fetchIncomingSet` by Atom type (PredicateNode -> table,
      VariableNode -> columns, ConceptNode/NumberNode -> joined rows).
      Cf. Codex C2-G4-G7; Gemini Section G.

R2.G4 DECKO-SPECIFIC LIMITATIONS (Codex-surfaced):
      (a) Postgres-only (Codex C2-G3) — but Decko uses Postgres, so OK.
      (b) Public-schema-only (Codex C2-G4) — Decko's schema is in
          `public` by default, so OK.
      (c) Naked PK/FK storage; punts on FK-driven atom links
          (Codex C2-G10) — Decko's relational structure (cards →
          card_actions → card_changes) would not auto-translate to
          AtomSpace links without custom mapping templates.
      (d) No Decko-specific templates exist in atomspace-bridge
          (Codex C2-G12).
      (e) `fetchIncomingByType` ignores type and delegates to
          `fetchIncomingSet` — semantic simplification (Codex C2-G7).

R2.G5 INTEGRATION VERDICT:
      atomspace-bridge is the BEST EXISTING seam for read-side Decko
      integration but requires:
      - Custom Decko-specific schema templates / mappings
      - FK-driven link generation (currently absent)
      - For write-back, implementation of the empty StorageNode write
        methods (intent stated; code absent)
      A read-only Decko-as-AtomSpace mirror via atomspace-bridge is
      achievable with engineering effort but not zero-friction.
      Cf. Codex C2-Z2, J6, L9.
