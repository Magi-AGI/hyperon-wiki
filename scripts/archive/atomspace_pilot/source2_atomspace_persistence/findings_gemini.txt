## Section C — atomspace-pgres Verdict

VERDICT: [BACKING-STORE-ONLY] (Fail for low-friction seam).

Rationale:
- SQLAtomStore.cc explicitly hardcodes:
    INSERT INTO Atoms (uuid, space, type, name, height, outgoing)
- Requires a specific "OpenCog SQL" schema.
- Provides ZERO capability for overlaying on existing PostgreSQL tables
  like Decko's.

Deprecation:
- README-design.md explicitly states: "As of 2022, the Postgres backend
  is deprecated."


## Section G — atomspace-bridge / BridgeStorageNode

The Seam:
- BridgeStorage.cc and SQLReader.cc implement a dynamic schema-discovery
  mechanism.
- Queries information_schema.columns to build SignatureLink representations
  of SQL tables.

Potential:
- This is the most promising path for "Low-Friction" integration.
- Could allow AtomSpace to "see" Decko cards as PredicateNode instances
  WITHOUT re-platforming the data.

Current Status:
- Read-only (extraction only).
- BridgeStorage::storeAtom is currently an empty stub.


## Section D — Modern Hyperon-Experimental Seams (NEW)

Two Python-native seams surfaced in hyperon-experimental that bypass the
classical C++ StorageNode entirely:

1. python/sandbox/sql_space/sql_space.py — A MeTTa-native alternative to
   the C++ StorageNode. Maps SQL tables to MeTTa grounded atoms,
   aligning with the hyperon-experimental direction.

2. python/sandbox/mork/mork.py — Bridge code for MORK substrate
   integration.

## Conclusion

The lowest-friction path is NOT atomspace-pgres, but either:
  (a) atomspace-bridge (C++ legacy, write-stubs unimplemented), or
  (b) sql_space.py (modern MeTTa-native).
