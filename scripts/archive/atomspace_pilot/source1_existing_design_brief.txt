§1. SCOPE AND DELIVERABLE TAXONOMY
----------------------------------
Source 1 is NOT a passive documentation read. It is an active claim audit.
Each substantive doc claim shall be classified into exactly one of:

  REQUIREMENT    — User/project goal to preserve unless directly contradicted
                   by Sources 2-4 evidence. Acts as the "what we must satisfy"
                   anchor for the integration architecture.

  ASSUMPTION     — Claim presented as if true but dependent on the state of
                   external code (MORK, DAS, atomspace-storage, hyperon-
                   experimental, etc.) that Sources 2-4 will verify. Tag with
                   the Source(s) that should resolve it.

  STALE CLAIM    — Already suspect from Phase 0 evidence or from prior
                   cluster pilots (PLN, ECAN, OpenPsi). Examples to expect:
                   any reference to ECAN-PLN coupling, OpenPsi paper-
                   faithfulness, or AttentionBank availability — all of which
                   were re-characterized after the doc was written. Flag with
                   the contradicting prior-pilot reconciliation.

  DESIGN OPTION  — A viable architecture branch the doc names but does not
                   commit to. Examples: StorageNode adapter vs. MCP bridge
                   vs. shadow/mirror vs. MORK FFI vs. MeTTa Spaces. Catalog
                   without choosing.

  NON-GOAL       — Explicitly out of scope, deferred, or rejected. Capture so
                   later sessions don't re-litigate.

Acceptance criterion (per Gemini's Phase 0 read of ATOMSPACE-INTEGRATION.md):
The integration is conditional on proving value over PostgreSQL alone. Doc-
declared latency budgets are <500ms target / <2s acceptable for "Fetch card."
Source 1 must surface every numeric budget, every value-over-Postgres claim,
and every "Phase 3 (Jan-Apr 2026) prototype must answer X" research question
so Sources 2-4 know what they're being graded against.

Dual-purpose deliverable (per user's explicit correction):
  (a) Wiki content correctness — every Source 1 STALE CLAIM and every
      doc-vs-wiki-card disagreement becomes a candidate wiki edit at
      cluster close, regardless of integration outcome.
  (b) Integration architecture — Source 1 outputs the requirements +
      assumptions + design-option catalog that Sources 2-4 evaluate.

§5. METHODOLOGY RULES IN FORCE
------------------------------
The OpenPsi cluster pilot established two new methodology rules that apply
to the AtomSpace pilot from Source 1 onward. Both reviewers MUST observe:

R1. PICKAXE BEFORE "DEAD" CLAIMS (ECAN Source 3 V0-1).
    For any feature/flag/function the docs describe as "deprecated", "not
    used", "legacy", or "removed", verify with `git log --all -S '<token>'`
    in the relevant repo before propagating. Source 1 may not need pickaxe
    itself but reviewers should NOT trust doc-side "deprecated" tags
    without a Source 2-4 pickaxe verification flag.

R2. CALLER-ANALYSIS MUST BE TIME-INDEXED (OpenPsi V4-1).
    "Called/uncalled" claims must include the commit/date snapshot at
    which the analysis was performed. Source 1 doc-claims that assert
    "X is currently called by Y" without a commit pin are ASSUMPTIONS,
    not REQUIREMENTS.

R3. BIDIRECTIONAL FORK-DIVERGENCE (OpenPsi V2-1, V4-2).
    Any doc claim labeling a fork as a divergent tradition or independent
    lineage shall be flagged ASSUMPTION until Source 2-4 verifies via
    `git log A..B` AND `git log B..A` AND grep both directions.

R4. C++ AND SCHEME/MeTTa BOTH (ECAN Source 4 V0-1).
    For any doc claim about API consumption in legacy OpenCog repos,
    note that verification requires grepping BOTH C++-side tokens
    (AttentionBank, get_sti, StorageNode, BackingStore) AND Scheme/MeTTa
    side (cog-av-sti, cog-stimulate, &storage, lib_spaces.metta).
    Source 1 itself only flags this; verification happens in Source 2.

R5. CODE COMMENTS / DOC CLAIMS ARE HYPOTHESES, NOT EVIDENCE
    (OpenPsi Source 3 brief #8).
    Doc statements that "X implements Y" or "the adapter will call Z"
    are ASSUMPTIONS until verified at the named code site.

Cross-model triangulation rules (carried forward from PLN/ECAN/OpenPsi):
  - One writer only: orchestrator executes wiki + file writes.
  - Cross-check every "Verified" claim against the source.
  - Reviewers may disagree; reconcile, do not vote.
  - Report dissents as V<source>-<seq> records.
