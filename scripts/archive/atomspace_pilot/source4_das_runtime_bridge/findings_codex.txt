EXECUTIVE HEADLINE

C4-1. DAS is a real distributed AtomSpace/runtime layer and is already exposed as a Hyperon MeTTa Space, but it is not the right first Decko/wiki backend seam. The safest Phase 3 prototype is READONLY-ATOMSPACE-BRIDGE: keep Decko/Rails/PostgreSQL as source of truth, build a read-only AtomSpace semantic mirror, and defer write-through DAS/MORK until delete semantics, MORK server drift, and Decko card semantics are resolved.

C4-2. DAS MorkDB remains CODE-REAL but not Decko-ready. Current Source 4 evidence confirms Source 3's split verdict: MorkDB is a DAS-side AtomDB backend over MongoDB + MORK HTTP server, but `MorkDB::delete_link` is still a hard false path, while DAS Docker still pins a stale MORK server commit.

C4-3. DAS Attention Broker is an engineering ECAN-compatible surrogate, not a literal 2009 ECAN implementation. It implements STI, contexts, exact-count Hebbian networks, rent/wages, and token spreading, but the formulas are the DAS token model, not the Classic ECAN paper equations. Label: [SURROGATE-COMPATIBLE].

C4-4. The existing Decko MCP API is a mature consumer API for cards, relations, history, render conversion, file upload, auth, and batch operations. It is the best source-of-truth API surface for extracting wiki data into any AtomSpace/DAS/MORK mirror. It is not an AtomSpace MCP.

DELIVERABLE 1 - DAS + RUNTIME AUDIT

Section A - DAS as wiki backend verdict (S4.Q1)

C4-A1. Verdict: (b) deployment-layer.

C4-A2. DAS is not "irrelevant" because it is explicitly built as a distributed AtomSpace query/runtime layer and is already integrated with Hyperon Experimental. `das/README.md:40-48` says DAS exposes a query API for large knowledge bases, has configurable persistence backends, and can be used as a special Hyperon space. `hyperon-experimental/lib/src/metta/runner/builtin_mods/das.rs:30-41` gives the MeTTa REPL usage example for `(import! &self das)`, `(new-das! ...)`, and `(match &das ...)`.

C4-A3. DAS is not the direct consumer API for Decko. Its README operational path is DAS CLI + AtomDB + Query Engine + Attention Broker + agents (`das/README.md:50-59`, `124-130`, `191-236`). The wiki already has Decko MCP endpoints for card CRUD, relations, history, render, files, trash, and auth (`mod/mcp_api/config/initializers/mcp_routes.rb:7-63`). A Decko integration should use Decko MCP/Rails/Postgres as the consumer-facing extraction/control layer, then feed DAS or an AtomSpace mirror downstream.

C4-A4. DAS is therefore a plausible later deployment/query layer once the mirror semantics are proven, but it should not replace Decko storage during Phase 3. This preserves Source 1's gated-migration requirement: Decko remains source of truth and AtomSpace must prove semantic value, latency, and rollback safety first (R1.1, R1.A1-A8, R1.B3).

Section C - Decko-MCP API surface inventory (S4.Q16)

C4-C1. Decko MCP is operationally broad enough to serve as the extraction/source-of-truth API for a read-only AtomSpace mirror. Routes include auth, JWKS, types, tags, cards, card relations, history/revision/restore, file upload, trash listing, rename, and HTML/Markdown render conversion. Evidence: `mod/mcp_api/config/initializers/mcp_routes.rb:7-63`.

C4-C2. Auth is real RS256 JWT bearer auth, not an open local script surface. `base_controller.rb:10-13` authenticates all MCP endpoints except auth/JWKS; `base_controller.rb:31-41` extracts/verifies a bearer token and finds the current account; `auth_controller.rb:10-24` documents username/password and API-key methods; `auth_controller.rb:75-120` authenticates with Decko and generates the token.

C4-C3. Card identity and type/content preservation are supported at the API level. `cards_controller.rb:65-88` creates cards by name, type, and content. `cards_controller.rb:93-120` updates content/type and supports patch mode. This aligns with Source 1 R1.A5-A6, but any AtomSpace mapping still must preserve Decko-specific semantics, not just strings.

C4-C4. Deletion and rename exist in Decko MCP and are permission-gated. `cards_controller.rb:122-132` deletes cards; `134-164` renames cards and can update referers. `cards_controller.rb:10-13` checks admin role for destroy/rename. This matters because DAS/MORK delete is not equivalent.

C4-C5. Relation endpoints give the mirror enough graph structure to reconstruct many wiki edges without scraping rendered HTML. `cards_controller.rb:166-196` lists children; `198-266` provides referers, nested_in, nests, links, and linked_by.

C4-C6. Batch operations exist and can be transactional at the Decko level. `cards_controller.rb:268-290` accepts up to 100 ops and has a transactional mode. This is useful for controlled export snapshots, but not evidence that DAS/MORK can receive equivalent transactions.

C4-C7. History/restore endpoints exist. `cards_controller.rb:292-338` returns revision history and a specific revision; `340-360` restores from trash or revision. This is a strong reason not to map Decko directly onto a backend that lacks link delete/version semantics.

C4-C8. File/Image support exists in MCP. `cards_controller.rb:414-505` uploads base64 or URL file data, writes to Decko File/Image cards, and returns file metadata. Source 1's file-card requirement (R1.A7) remains first-class and is not handled by any DAS/MORK bridge.

C4-C9. Safe query is intentionally limited to CQL-like Decko fields. `query_controller.rb:38-75` allows only name, type, content, updated_at, and created_at; `100-110` executes via `Card.search` and re-checks permissions. It is a good extraction/search endpoint, not an AtomSpace query engine.

C4-C10. Design implication: use DECKO-MCP for card extraction and change detection; do not invent an "AtomSpace MCP" until a semantic mirror proves value. Source 2 R2.3 already found the `from hyperon import Atomspace, MCP` sketch apocryphal.

Section D - DAS deep audit and Attention Broker classification (S4.Q10, S4.Q18)

C4-D1. DAS is a query engine/deployment architecture over AtomDB + agents. `das/docs/conceptual_documentation.md:16-23` describes query-engine pruning, AttentionBroker, specialized agents, and Service Bus. `README.md:40-44` describes DAS as a query API for very large knowledge bases with configurable backends.

C4-D2. AtomDB is an abstraction over storage backends, not the same as OpenCog Classic StorageNode. `conceptual_documentation.md:26-30` says AtomDB can be implemented by different backends including DBMSs and MORK. `AtomDB.h:45-65` defines add/delete/re-index methods at the API boundary.

C4-D3. Query execution is STI-prioritized through sources. `conceptual_documentation.md:64-77` says sources fetch candidate answers from AtomDB, query AttentionBroker for STI, attach STI, sort candidates by STI, and deliver higher-STI candidates first.

C4-D4. AttentionBroker stores STI outside AtomDB and per context. `conceptual_documentation.md:79-89` says STI is in [0,1], not stored as atom property, held in AttentionBroker, and can differ across contexts. It also says Hebbian links estimate co-presence probability in query answers.

C4-D5. AttentionBroker has real RPC operations and persistence. `AttentionBrokerServer.h:58-64` declares worker count and global parameters. `AttentionBrokerServer.h:95-169` exposes stimulate/correlate/asymmetric_correlate/get_importance/set_determiners/set_parameters/save_context/drop_and_load_context. `AttentionBrokerServer.cc:206-259` serializes and reloads context state.

C4-D6. AttentionBroker is not running the exact 2009 ECAN formulas. Its concrete implementation is `ExactCountHebbianUpdater` plus `TokenSpreader`. `HebbianNetworkUpdater.h:92-95` defines weights as count(A->B)/count(A). `HebbianNetworkUpdater.cc:57-75` increments symmetric correlations; `78-96` increments asymmetric correlations. This is exact-count statistical updating, not the Classic ECAN Hebbian Conjunction equation audited in the ECAN cluster.

C4-D7. Rent/wages are implemented, but as a fixed-token model. `StimulusSpreader.h:75-78` says importance is a fixed amount of tokens distributed among atoms. `StimulusSpreader.cc:54-60` collects rent as `rent_rate * importance`; `140-157` distributes wages proportional to request counts; `160-209` combines rent, alienated tokens, wages, consolidation, and one spreading cycle.

C4-D8. The spreading formula is deterministic and arity-weighted, not stochastic sigmoid diffusion. `StimulusSpreader.cc:75-82` computes spreading rate from arity ratio and lower/upper bounds, subtracts `to_spread`, and stores it. `102-109` delivers stimulus proportionally to edge count / source count normalized by sum weights. There is no tanh sigmoid, tournament, or probabilistic diffuse/no-diffuse step.

C4-D9. Defaults are fixed and aggressive. `AttentionBrokerServer.cc:10-12` sets `RENT_RATE = 0.75`, lower spreading = 0.10, upper spreading = 0.10. `AttentionBrokerServer.cc:187-197` allows runtime parameter updates through `set_parameters`.

C4-D10. Tests confirm the intended token-rent/wage mathematics. `stimulus_spreader_test.cc:23-69` tests wage distribution (2/8 -> 0.25, 1/8 -> 0.125). `stimulus_spreader_test.cc:165-189` computes expected rent, total rent/wages, updated importance, and spreading rate exactly from current constants. `attention_broker_test.cc:46-98` verifies correlated/stimulated atoms get high importance and determiners can lift another atom's importance.

C4-D11. Label for S4.Q18: [SURROGATE-COMPATIBLE]. DAS AttentionBroker is more production-oriented than the metta-attention audit surface, but its math is still a functional ECAN-compatible engineering surrogate. It should be documented as "ECAN-inspired/ECAN-compatible attention allocation for query prioritization," not "literal 2009 ECAN."

Section E - Delete-completeness verdict (S4.Q17)

C4-E1. Required hard label: [BLOCKING-INTEGRATION].

C4-E2. DAS's abstract AtomDB API includes full delete methods. `AtomDB.h:57-63` defines delete_atom/node/link and plural variants. RedisMongoDB implements them (`RedisMongoDB.h:113-119`; `RedisMongoDB.cc:996-1093`). AtomDBProxy forwards `delete_atoms` commands to the server side (`AtomDBProxy.cc:100-109`, `155-188`). RemoteAtomDB and RemoteAtomDBPeer also forward delete calls (`RemoteAtomDB.cc:285-330`, `RemoteAtomDBPeer.cc:344-395`).

C4-E3. MorkDB overrides only link delete and hard-fails it. `MorkDB.h:56-58` says delete_link is TODO until MORK supports deleting links/S-expressions. `MorkDB.cc:268-270` logs "MORKDB does not support deleting links" and returns false.

C4-E4. Tests intentionally codify the failure. `morkdb_test.cc:250-251` expects `delete_link` false, `275-276` expects `delete_links` false, and `296` repeats the false expectation. Node delete tests pass (`morkdb_test.cc:172-197`) because MorkDB inherits RedisMongoDB node deletion, but that does not solve link/S-expression cleanup.

C4-E5. There is a batch rebuild workaround, not an integration-ready delete path. `MorkDB.cc:273-316` exposes `flush_pattern` and `re_index_patterns`, clearing MORK by arity and re-adding links from MongoDB. This can repair/rebuild indexes in batch but is not transactional card delete/restore semantics and is not safe as a live Decko write-through backend.

C4-E6. Therefore Gemini's likely `[BLOCKING-INTEGRATION] / [WORKAROUND-EXISTS]` dual label should be reconciled as `[BLOCKING-INTEGRATION]` with a batch-rebuild workaround noted. The brief required one label; Decko delete/rename/trash/restore semantics need a live, per-card, consistent delete path.

DELIVERABLE 2 - INTEGRATION DESIGN MEMO

1. Executive summary (<=150 words)

Build Phase 3 as a read-only AtomSpace semantic mirror of Decko, not a write-through DAS/MORK backend. The existing Decko MCP API already exposes the source-of-truth card surface: CRUD, names/types/content, relations, history, render conversion, file upload, and auth. DAS is code-real and Hyperon-integrated, with MORK-backed AtomDB and AttentionBroker, but MorkDB link/S-expression delete is unsupported and the DAS MORK Docker path pins a stale server commit. atomspace-bridge is the closest existing SQL-to-AtomSpace seam, but its write methods are empty. The acceptance gate should therefore test semantic value over PostgreSQL under latency and integrity constraints before any storage migration. Use DAS/MORK as downstream query/substrate candidates after the read-only mirror proves value.

2. Acceptance gate recap

C4-MEMO-1. Source 1 acceptance gate remains controlling (R1.1-R1.4, R1.B3):
- Fetch card latency: target under 500ms, acceptable under 2s.
- Semantic/PLN query latency: target under 2s, acceptable under 5s, but no PLN-completeness overclaim.
- Demonstrate at least 5 semantic insights/relationships not trivially available from PostgreSQL.
- Zero export/mapping errors over repeated runs.
- Preserve Decko semantics: card identity, names, type, RichText/Markdown behavior, sections/TOC inclusions, file cards, history, permissions, delete/restore.
- Keep Decko/Rails/PostgreSQL primary during the prototype.

C4-MEMO-2. Source 2 and Source 3 narrowed the gate:
- atomspace-pgres is BACKING-STORE-ONLY and deprecated, not Decko schema-compatible (R2.1).
- atomspace-bridge is closest existing read-side seam but not write-ready (R2.2).
- DAS is already wired into Hyperon but should be evaluated as runtime/query layer (R2.4).
- MORK native is FFI/SDK/substrate, not StorageNode/DAS by itself (R3.1).
- DAS MorkDB is code-real but delete-incomplete (R3.2, C4-E).

3. Winning seam selection

C4-MEMO-3. Hard label: READONLY-ATOMSPACE-BRIDGE.

C4-MEMO-4. Rationale:
- It respects Decko as source of truth (R1.A1-A2).
- It uses the mature Decko MCP/Rails/Postgres API surface for extraction (C4-C).
- It avoids MORK/DAS delete blockers during Phase 3 (C4-E).
- It avoids atomspace-bridge write stubs (C4-G).
- It can still exercise AtomSpace/DAS/MORK semantic-query value downstream.

C4-MEMO-5. Explicit non-selection:
- DECKO-MCP->DAS: too early; DAS/MorkDB delete and MORK server pin are unresolved.
- DECKO-MCP->HYPERON-SPACE->DAS: adds MeTTa runtime complexity without solving Decko semantics.
- DIRECT-MORK-HTTP: MORK-native surface lacks Decko schema, CRUD, history, permissions, and delete semantics.
- DIRECT-MORK-FFI: potentially low latency, but product semantics and Ruby/Rails integration are custom work.
- HYBRID: correct as a long-term architecture option, not Phase 3's first proof.
- DEFER: too conservative; a read-only mirror is feasible and valuable enough to prototype.

4. Phase 3 prototype scope

C4-MEMO-6. Prototype name: Decko read-only AtomSpace mirror.

C4-MEMO-7. Input:
- Use Decko MCP or Rails runner/Postgres export to pull a controlled card subset.
- Include card name, id, type, content, updated_at, children, links, referers, nests/nested_in, history summary, tags, and file metadata placeholders.
- Start with 500-2,000 cards; include representative RichText, Markdown, RawData, Pointer, File/Image, and cluster-pilot cards.

C4-MEMO-8. Mapping:
- Card -> ConceptNode/CardNode equivalent.
- Card type -> InheritanceLink or typed predicate.
- Parent/child plus-name hierarchy -> EvaluationLink or explicit relation atoms.
- Links/nests/referers -> relation atoms with relation type.
- Content hashes and revision metadata -> explicit data atoms, not MORK CountSink.
- File cards -> metadata atoms only in Phase 3; binary payload stays in Decko storage.

C4-MEMO-9. Backend:
- Start with atomspace-bridge-style read/import mapping or a small custom exporter into a local AtomSpace/MeTTa representation.
- Run optional DAS query experiments after the mirror exists, not before.
- Do not use DAS/MORK as mutable store in Phase 3.

C4-MEMO-10. Evaluation:
- Measure export time, mirror-build time, single-card fetch, relation query, semantic query, and repeated-run consistency.
- Compare against native PostgreSQL/CQL baseline for the same queries.
- Require 5 semantic insights that are materially harder in PostgreSQL: e.g. cross-cluster lineage, stale-claim detection, relation-distance queries, source-to-wiki-card provenance paths, and publication/repo/code triple links.

C4-MEMO-11. Stop criteria:
- Any loss of card identity/name/type/content.
- Any file-card payload corruption.
- Any history/delete/restore mismatch if those are included in the subset.
- p90 query latency over 2s for simple semantic lookups unless the query demonstrably cannot be approximated in SQL.
