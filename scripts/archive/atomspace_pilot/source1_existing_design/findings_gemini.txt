## Executive Summary

G1. Source 1 calibrates the Phase 3 (Jan-Apr 2026) "Atomspace Integration
    Prototype" against ROADMAP.md and SPEC.md, with the goal of proving
    superiority over PostgreSQL before committing to Phase 4 ("Atomspace
    Backend Swap" Apr-Jul 2026, conditional). Phase 5 ("AI Gamemaster
    Foundation", 2026+) leverages PLN for symbolic game-rule reasoning
    (e.g., "Fire spells burn wood").

G2. The Phase 2 MCP API is FULLY OPERATIONAL as of Dec 2, 2025 with JWT
    authentication, RS256 signing, JWKS, and role-based access (user/gm/
    admin). The MCP middleware is therefore live infrastructure, not a
    proposed component.

G3. SPEC.md core hypothesis: Decko's structured card model provides better
    semantic relationships that offset the database operation latency
    penalty (target: <10x slower than static files). The naming convention
    `[Type]: [Name] (+Context)` is established and should be mirrored in
    AtomSpace ConceptNode names.


## Section A — Requirements (REQUIREMENT)

(Gemini's "Section A: Choice Points & Assumptions / ARCHITECTURAL-INTENT")

G4. REQUIREMENT: Targeted evolution path. PostgreSQL (Phase 1) -> Dual
    Backend Prototype (Phase 3) -> AtomSpace Primary (Phase 4).

G5. REQUIREMENT: Conditional commitment. AtomSpace swap is explicitly
    conditional on proving clear value over PostgreSQL (e.g., semantic
    reasoning) within the Phase 3 window (Jan-Apr 2026).

G6. REQUIREMENT: Designated middleware pattern. MCP Adapter (Python-based)
    that exposes a Decko-like Card API to the Rails frontend while
    managing AtomSpace/MORK atoms in the backend.
