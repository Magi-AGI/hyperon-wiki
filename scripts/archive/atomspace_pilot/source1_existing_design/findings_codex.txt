# Source 1 - Existing Wiki Backend Integration Design - Codex Findings

Pilot: AtomSpace Backend Integration Cluster Pilot
Source: in-repo design and operations docs
Model lane: Codex
Date: 2026-04-29
Write policy: no wiki writes; no design-doc edits during extraction

## Executive Summary

C1. The existing design is not "replace PostgreSQL with Atomspace"; it is a gated migration program. The canonical requirement is: keep Decko/Rails/PostgreSQL as the production path unless an Atomspace prototype proves clear value over PostgreSQL. Evidence: `docs/ATOMSPACE-INTEGRATION.md:12-15`, `docs/ROADMAP.md:224-229`, `docs/ROADMAP.md:358-370`.

C2. The acceptance bar is explicit and quantitative. Fetch-card latency target is `<500ms`, acceptable `<2s`, unacceptable `>5s`; Phase 3 requires `<2s` latency for 90% of queries, MCP overhead `<10%`, 0 export errors in 10 runs, 1000+ cards, and 5+ PLN/semantic insights not visible in PostgreSQL. Evidence: `docs/ATOMSPACE-INTEGRATION.md:639-645`, `docs/ATOMSPACE-INTEGRATION.md:788-803`, `docs/ROADMAP.md:342-370`.

C3. The design docs contain a real adapter sketch, but it is a sketch, not implementation evidence. Local code has `mod/mcp_api/` for a Decko MCP/card API, but grep found no implemented `DeckoAtomspaceAdapter`, `AtomspaceBackend`, `DualWriteAdapter`, or `CARD_BACKEND` switch outside docs. The term "MCP adapter" is overloaded and must be disambiguated in later sources.

C4. Source 1 should be used as the requirements anchor for Sources 2-4, not as proof that AtomSpace/MORK/DAS provide the required APIs. The docs assert or assume a Python Hyperon Atomspace API, PLN query support, MORK distributed backend behavior, and MCP-mediated card operations. Those are Source 2/3/4 verification targets.

## A. Requirements

C5. REQUIREMENT: The production wiki remains Decko/Rails/PostgreSQL during the prototype phase. The Atomspace path is a read-only or mirrored evaluation path until it meets the gate. Evidence: `docs/ATOMSPACE-INTEGRATION.md:59-83`, `docs/ROADMAP.md:232-258`.

C6. REQUIREMENT: PostgreSQL remains the fallback even after an Atomspace attempt. The migration plan explicitly includes dual-write, consistency checks, a backend flag, and rollback to PostgreSQL. Evidence: `docs/ATOMSPACE-INTEGRATION.md:703-780`, `docs/ROADMAP.md:400-430`, `docs/ROADMAP.md:525-531`.

C7. REQUIREMENT: Any integration must preserve the Decko card model: card identity, name, type, content, pointers, timestamps, hierarchical `+` names, nesting, and cardtype-specific rendering behavior. Evidence: `docs/ATOMSPACE-INTEGRATION.md:120-228`, `docs/DECKO-SECTION-PATTERN.md:47-79`, `docs/DECKO-CONTENT-UPLOAD-GUIDE.md:40`, `docs/DECKO-CONTENT-UPLOAD-GUIDE.md:198`.

C8. REQUIREMENT: RichText vs Markdown semantics are not cosmetic. Section/TOC inclusion cards require RichText and `{{+child|content}}` syntax; content cards may be Markdown. Any Atomspace mapping that treats content as plain strings only will miss operational wiki behavior. Evidence: `docs/DECKO-SECTION-PATTERN.md:47-79`, `docs/DECKO-CONTENT-UPLOAD-GUIDE.md:243-245`, `docs/DECKO-CONTENT-UPLOAD-GUIDE.md:370-374`.

C9. REQUIREMENT: The integration must preserve file-card/upload behavior or explicitly scope it out. Browser file uploads had Decko-specific cache/action-context bugs and console uploads required a server-side workaround; this is not captured by the simple ConceptNode/StateLink card mapping. Evidence: `docs/DECKO-FILE-UPLOAD-BUG.md:10-29`, `docs/DECKO-FILE-UPLOAD-BUG.md:101-104`.

C10. REQUIREMENT: Operational access today is Rails/Decko API access, not direct file editing. Cards live in the database; normal inspection uses Rails console, `script/card runner`, or the MCP/card API. Evidence: `docs/CLAUDE.md:7-18`, `docs/CLAUDE.md:22-35`, `docs/DECKO-DATABASE-ACCESS.md:87-119`.

C11. REQUIREMENT: The integration must demonstrate semantic value, not just alternate storage. The docs repeatedly ask whether PLN/semantic reasoning finds relationships PostgreSQL cannot. Evidence: `docs/ATOMSPACE-INTEGRATION.md:406-429`, `docs/ATOMSPACE-INTEGRATION.md:788-803`, `docs/ROADMAP.md:294-304`, `docs/ROADMAP.md:358-370`.

C12. REQUIREMENT: Prototype scale targets include 100/1000/10000-card tests, 1000+ cards as Phase 3 success, and 100+ concurrent users for Phase 4. Evidence: `docs/ATOMSPACE-INTEGRATION.md:834-852`, `docs/ROADMAP.md:342-349`, `docs/ROADMAP.md:510-517`.

## B. Assumptions To Verify In Sources 2-4

C13. ASSUMPTION: `from hyperon import Atomspace, MCP` and `MCP.Client(atomspace_url)` are real current APIs suitable for the adapter. The code block is a design sketch, not proof. Verify in Source 4. Evidence: `docs/ATOMSPACE-INTEGRATION.md:248-253`.

C14. ASSUMPTION: The Atomspace query API can implement `fetch_card`, `search_cards`, create/update/delete, and relationship inference with the shown query shapes. Verify actual query semantics, mutation semantics, and deletion behavior in Source 2. Evidence: `docs/ATOMSPACE-INTEGRATION.md:257-404`.

C15. ASSUMPTION: `pln_query` exists and can produce actionable card relationship inferences within the latency budget. Prior PLN pilot results make this a high-risk claim unless scoped carefully. Verify in Sources 2 and 4. Evidence: `docs/ATOMSPACE-INTEGRATION.md:406-429`, `docs/ATOMSPACE-INTEGRATION.md:639-645`.

C16. ASSUMPTION: Card names can be sanitized and desanitized without loss. The proposed `_sanitize` replaces spaces, colons, and plus signs with underscores, and the doc itself notes original names may need metadata. This is a correctness risk because `+` hierarchy is a first-class Decko convention. Verify in Source 2/integration synthesis. Evidence: `docs/ATOMSPACE-INTEGRATION.md:443-479`.

C17. ASSUMPTION: MORK is available as a distributed Atomspace backend with horizontal scaling, replication, partitioning, and coordination. Verify in Source 3; do not treat the Source 1 text as implementation evidence. Evidence: `docs/ATOMSPACE-INTEGRATION.md:580-630`, `docs/ROADMAP.md:486-509`.

C18. ASSUMPTION: Atomspace/MORK consistency semantics can support Decko writes. The docs explicitly list eventual-vs-strong consistency and conflict handling as open questions. Verify in Source 3/4. Evidence: `docs/ATOMSPACE-INTEGRATION.md:610-630`, `docs/ATOMSPACE-INTEGRATION.md:819-826`.

C19. ASSUMPTION: Existing MCP work can serve as the integration transport. Current repo code supports a Decko MCP/card API, while the Atomspace adapter is not present. Source 4 must decide whether to reuse MCP, build a new Atomspace MCP server, or bypass MCP. Evidence: `docs/MCP-DEPLOYMENT-STATUS.md:13-50`, `docs/MCP-PHASE-2-COMPLETE.md:5-12`.

C20. ASSUMPTION: `atomspace-pgres` or `StorageNode` may be a better Decko integration seam than the sketched Python MCP adapter. Source 1 docs do not analyze this; Phase 0 surfaced it as a candidate. Verify in Source 2.

C21. ASSUMPTION: DAS is relevant to the wiki backend integration. Source 1 docs emphasize Atomspace/MORK/MCP, not DAS. Source 4 must establish whether DAS is the consumer-facing API the wiki should call or a separate distributed deployment layer.

## D. Design Options

C29. DESIGN OPTION: Shadow/nightly export mirror. Decko/PostgreSQL remains primary; Atomspace receives exports for read-only evaluation. This is the safest Phase 3 path and should be the baseline option. Evidence: `docs/ATOMSPACE-INTEGRATION.md:59-83`, `docs/ROADMAP.md:232-258`.

C30. DESIGN OPTION: Atomspace primary via adapter. Atomspace becomes source of truth, PostgreSQL becomes backup/archive, and an adapter provides Decko-like card operations. This is Phase 4 only, conditional on Phase 3. Evidence: `docs/ATOMSPACE-INTEGRATION.md:85-114`, `docs/ATOMSPACE-INTEGRATION.md:540-575`.

C31. DESIGN OPTION: Dual-write migration. Writes go to both PostgreSQL and Atomspace, reads initially stay PostgreSQL, then switch to Atomspace with PostgreSQL shadow validation. Evidence: `docs/ATOMSPACE-INTEGRATION.md:703-752`, `docs/ROADMAP.md:400-430`.

C32. DESIGN OPTION: MCP bridge. The repo already has a Decko MCP API surface; a bridge could expose Atomspace-backed query tools without replacing Decko storage. This may satisfy semantic-query goals with less risk than a backend swap. Evidence: `docs/MCP-DEPLOYMENT-STATUS.md:13-50`, `docs/MCP-PHASE-2-COMPLETE.md:181-230`.

C33. DESIGN OPTION: Direct Atomspace/StorageNode integration. Not developed in Source 1 docs, but Phase 0 surfaced Atomspace storage backends (`atomspace-storage`, `atomspace-pgres`, `atomspace-rocks`) as likely Source 2 targets. This may be more appropriate than HTTP/MCP if low latency is critical.

C34. DESIGN OPTION: MORK distributed backend. The docs frame MORK as optional scaling infrastructure, not required for the first prototype. Source 3 must establish whether MORK is a real backend option or still experimental for this use case. Evidence: `docs/ATOMSPACE-INTEGRATION.md:580-630`, `docs/ROADMAP.md:486-509`.

C35. DESIGN OPTION: Keep PostgreSQL primary indefinitely and use Atomspace for offline/batch reasoning only. This is compatible with the docs' go/no-go criteria and may be the pragmatic outcome if semantic value exists but latency or consistency fails. Evidence: `docs/ROADMAP.md:358-370`, `docs/ATOMSPACE-INTEGRATION.md:807-815`.
