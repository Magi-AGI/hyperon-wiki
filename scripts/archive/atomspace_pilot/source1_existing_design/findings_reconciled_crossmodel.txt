EXECUTIVE SUMMARY (CONSENSUS)
================================================================================

R1.1 Source 1 locks the ACCEPTANCE GATE, not the ARCHITECTURE CHOICE. Both
     models converge on this framing. AtomSpace integration is conditional
     on proving measurable semantic value over PostgreSQL while meeting
     latency, reliability, rollback, and Decko-semantics requirements.
     Cf. Codex C1, C62, C63 ↔ Gemini G1, G5.

R1.2 Quantitative acceptance criteria are explicit and shared:
     - Fetch card latency:    target <500ms / acceptable <2s / unacceptable >5s
     - Search/create latency: target <1s    / acceptable <3s / unacceptable >10s
     - PLN inference:         target <2s    / acceptable <5s / unacceptable >15s
     - Phase 3 scale:         1000+ cards no degradation
     - Phase 4 scale:         10,000+ cards / 100+ concurrent users
     - Phase 3 success:       <2s p90 latency, MCP overhead <10%, 0 export
                              errors in 10 runs, 5+ PLN/semantic insights
                              not visible in PostgreSQL
     Cf. Codex C2, C12 ↔ Gemini G29-G32.

R1.3 The "MCP adapter" term is overloaded. There are TWO distinct components
     that share the label and must not be conflated in subsequent sources or
     wiki edits:
     (a) The deployed Decko MCP/card API. Operational since 2025-12-02 with
         JWT/RS256/JWKS/role-based access. Live infrastructure.
     (b) The proposed Python `DeckoAtomspaceAdapter` bridge. Sketch only;
         no implementation in the local code tree (Codex C3 grep).
     Cf. Codex C3, C19, C60 ↔ Gemini G2, G6, G21, G22.

R1.4 The integration design is a GATED MIGRATION program, not a backend
     swap. Phase 3 (Jan-Apr 2026) prototypes against PostgreSQL primary;
     Phase 4 (Apr-Jul 2026) is conditional swap with dual-write rollback;
     Phase 5 (2026+) AI Gamemaster reasoning. Both reviewers concur.
     Cf. Codex C5, C6, C29-C31, C36, C37 ↔ Gemini G4, G5.

SECTION D — DESIGN OPTIONS (CONSENSUS CATALOG)
================================================================================

The reviewers' option catalogs largely overlap. Reconciled list:

R1.D1 Shadow / nightly export mirror. Decko/PostgreSQL primary; Atomspace
      receives exports for read-only evaluation. SAFEST Phase 3 baseline.
      [C29]

R1.D2 Atomspace primary via adapter. Atomspace becomes source of truth;
      PostgreSQL becomes backup/archive. PHASE 4 ONLY, conditional on
      Phase 3 outcome. [C30, G4]

R1.D3 Dual-write migration. Writes go to both stores; reads transition
      from PostgreSQL to Atomspace with PostgreSQL shadow validation.
      [C31, G4]

R1.D4 MCP bridge. Repo already has a Decko MCP API surface; a bridge
      could expose Atomspace-backed query tools without replacing Decko
      storage. May satisfy semantic-query goals with less risk than a
      backend swap.
      [C32, G6, G21]

R1.D5 Direct Atomspace/StorageNode integration. atomspace-storage,
      atomspace-pgres, atomspace-rocks are likely Source 2 targets.
      May be more appropriate than HTTP/MCP if low latency is critical.
      [C33, G19]

R1.D6 MORK distributed backend. Framed as optional scaling infrastructure,
      not required for first prototype. Source 3 must establish whether
      MORK is real backend option or still experimental for this use.
      [C34]

R1.D7 PostgreSQL-primary indefinitely with Atomspace for offline/batch
      reasoning. Compatible with the gate criteria; pragmatic outcome
      if semantic value exists but latency or consistency fails.
      [C35]

R1.D8 lib_spaces.metta — MeTTa-native pluggable storage abstraction.
      Distinct from above; could host any of the above as backends.
      [G20]

R1.D9 (NEW SEAM, Gemini-only) `run-card-runner.sh` — production wrapper
      script at `/home/ubuntu/run-card-runner.sh` for direct Ruby-to-Card
      operations. Potential high-value seam for administrative/bulk
      integration testing.
      [G18] — V1-6 Gemini-only finding.

R1.D10 Ruby AtomspaceBackend wrapper. Prospective `lib/atomspace_backend.rb`
       wrapping the MCP adapter from the Rails side. Implementation
       location, not an alternative to the adapter itself.
       [G22]
