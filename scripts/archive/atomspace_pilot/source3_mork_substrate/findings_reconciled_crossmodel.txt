R3.F2 SEMANTIC ROLE: query/reduction primitive.
      - Accumulates per-sink `unique: PathMap<()>`.
      - Computes `val_count()` during finalize.
      - Writes derived count facts when guards match or variables

      - No atomic increment/decrement primitives.
      - No Decko action IDs, revision timestamps, or transactional

      Codex (C3-F3): "Decko card revision mapping should NOT target
      CountSink directly."
      Gemini: "Could potentially model card versioning IF counts are
      repurposed as timestamps or sequence numbers."
      RECONCILED: Codex's stricter framing wins on architecture: as
      currently designed, CountSink is an MM2 query/reduction
      primitive within an execution, not a durable revision log.
      Gemini's "if repurposed" framing is hypothetical and would
      require:
        (a) Treating MORK as the system of record for revisions
            (currently rejected by Source 1 R1.A2 dual-write/rollback

        (b) Implementing custom counter-as-timestamp semantics outside
            CountSink's current contract.
      VERDICT for cluster narrative: card_actions / card_revisions
      should be represented explicitly as data atoms; CountSink can
      query/count them later, but should NOT be the primary store.

      paths (Codex C3-G1):
      (a) Ruby FFI -> mork_ffi: minimal ABI, no Ruby wrapper, no
          process model, no Decko schema mapping.
      (b) Rails HTTP client -> MORK server branch: feasible if server
          branch becomes deployment target. Most conventional non-DAS
          direct route.

          but requires DAS infrastructure (Redis/Mongo/MORK split).

      gives fast S-expression / path algebra and kernel primitives
      but does NOT define:
      - Decko card identity
      - RichText / file handling
      - Permissions
      - Redirects
      - Database transaction boundaries
      Source 1 R1.A5-A7 requirements remain unsatisfied by any MORK
      path without an adapter layer.

R3.G3 Cluster narrative: MORK should be treated as a SUBSTRATE unless
      Source 4 selects the DAS route. The "lowest-friction" framing
      (Source 1 R1.B1) does not survive Source 3: there is no
      zero-friction path; only varying engineering effort across
      paths (a), (b), (c) above.
