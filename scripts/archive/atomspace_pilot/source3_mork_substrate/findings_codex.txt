SECTION C - FFI / SDK BRIDGE FEASIBILITY
-------------------------------------------------------------------------------

C3-C1. mork_ffi is the strongest non-DAS direct bridge surface.

Evidence:
- mork_ffi/Cargo.toml:11-13 builds a `cdylib` named `mork_ffi`.
- mork_ffi/src/lib.rs:10-13 creates a process-global Space in a OnceLock.
- mork_ffi/src/lib.rs:41-42 exports `rust_mork(command, input)`.
- mork_ffi/src/lib.rs:60-117 implements string-command operations:
  add-atoms, remove-atoms, mm2-exec, get-atoms.
- mork_ffi/src/lib.rs:118-139 implements match.
- mork_ffi/src/lib.rs:145-150 exports `rust_string_free`.
- mork_ffi/mork.c:4-5 declares the Rust FFI symbols.
- mork_ffi/mork.c:8-29 wraps them as a SWI-Prolog foreign predicate.
- mork_ffi/mork.c:33-36 registers `mork/3`.
- mork_ffi/morkspaces.pl:7-32 maps the Prolog/MeTTa space operations
  add-atom, remove-atom, match, get-atoms, and mm2-exec onto mork/3.

C3-C2. Ruby FFI feasibility: POSSIBLE, not product-ready.

Static assessment:
- The C ABI shape is simple enough for Ruby FFI: two exported C-callable
  functions, both taking raw C strings / pointers.
- There is no Ruby binding, no Rails packaging, no Decko transaction model,
  no connection pooling, and no per-card schema layer.
- The global singleton Space at mork_ffi/src/lib.rs:10-13 means a Rails bridge
  would need explicit process/thread/lifecycle design before production use.

C3-C3. mork-rust-sdk is skeletal.

Evidence:
- mork-rust-sdk/Cargo.toml:1-11 declares a package named `mork-rust-sdk`.
- mork-rust-sdk/Cargo.toml:14 has no dependencies.
- mork-rust-sdk/src/lib.rs:1-43 is placeholder text split one character per
  line: "source code for the sdk will be put here".

C3-C4. mork-ts-sdk is a real HTTP client SDK, not a StorageNode or Rails
backend integration.

Evidence:
- mork-ts-sdk/README.md:1-3 describes a TypeScript client SDK for the MORK
  HTTP API.
- mork-ts-sdk/package.json:1-14 exposes package `mork-ts-sdk`.
- mork-ts-sdk/src/client.ts:3-15 defines MorkApiClient and dispatches requests
  to a base URL.
- mork-ts-sdk/src/client.ts:27-35 chooses text/plain for upload/transform and
  JSON for other bodies.
- mork-ts-sdk/src/requests/UploadRequest.ts:43-52 posts upload data to
  `/upload/{pattern}/{namespace+template}`.
- mork-ts-sdk/src/requests/ExportRequest.ts:49-72 exports via `/export/...`.
- mork-ts-sdk/src/requests/ClearRequest.ts:29-36 clears via `/clear/...`.
- mork-ts-sdk/src/requests/Request.ts:8-16 sets a 20 second default timeout.

C3-C5. Best direct bridge choices, ranked for Decko:

1. DAS MorkDB path: code-real and already has AtomDB semantics, but it is DAS
   infrastructure, not Decko-native.
2. MORK server HTTP path: viable as an external service if the server branch is
   adopted, but still needs Decko mapping and transaction semantics.
3. mork_ffi direct Ruby FFI: technically possible but highest engineering risk.
4. mork-ts-sdk: useful for JS/Node clients; not a Rails persistence seam.
5. mork-rust-sdk: not usable at this snapshot.

SECTION F - COUNT SINK PATTERN
-------------------------------------------------------------------------------

C3-F1. CountSink is real in MORK kernel.

Evidence:
- MORK/kernel/src/sinks.rs:127-132 defines the Sink trait.
- MORK/kernel/src/sinks.rs:512-516 comments examples and defines CountSink.
- MORK/kernel/src/sinks.rs:516-520 stores unique matches in PathMap.
- MORK/kernel/src/sinks.rs:526-533 inserts matching paths into the unique map.
- MORK/kernel/src/sinks.rs:534-591 finalizes by reducing the unique set and
  writing derived count facts.
- MORK/kernel/src/sinks.rs:1194-1208 lists CountSink in ASink alongside Add,
  Remove, Hash, Sum, FloatReduction, Pure, Z3, etc.
- MORK/kernel/src/sinks.rs:1229-1231 dispatches arity-4 `(count ...)` forms to
  CountSink.
- MORK/kernel/src/sinks.rs:1287, 1313, 1339 wire CountSink into request, sink,
  and finalize dispatch.

C3-F2. CountSink is a query/reduction primitive, not a persistent card-history
counter.

Evidence:
- CountSink accumulates a per-sink `unique: PathMap<()>` and swaps it out in
  finalize (sinks.rs:516-520, 539-541).
- It computes `val_count()` during finalize (sinks.rs:537-552).
- It writes derived output facts when guards match or variables substitute
  (sinks.rs:553-585).

No inspected CountSink code provides atomic increment/decrement operations,
Decko action IDs, revision timestamps, or transactional history semantics.

C3-F3. Decko card revision mapping should not target CountSink directly.

CountSink can answer "how many matching expressions exist?" inside an MM2
execution. It is not a durable revision log. If Decko history is mirrored into
MORK, card_actions/card_revisions should be represented explicitly as data
atoms; CountSink can query/count them later.

SECTION G - RAILS TO MORK BRIDGE
-------------------------------------------------------------------------------

C3-G1. Direct Rails-to-MORK is feasible only as custom engineering.

Candidate routes:

- Ruby FFI -> mork_ffi: minimal ABI exists, but no Ruby wrapper, no process
  model, and no Decko schema mapping.
- Rails HTTP client -> MORK server branch: feasible if server branch becomes
  the deployment target. This is the most conventional non-DAS direct route.
- Rails -> DAS -> MorkDB: strongest existing AtomDB semantics, but requires
  adopting DAS infrastructure and its Redis/Mongo/MORK split.
- Decko -> classical AtomSpace -> StorageNode: remains separate from MORK
  proper per Source 2.

C3-G2. If the goal is wiki backend integration, MORK should be treated as a
substrate unless Source 4 selects the DAS route.

MORK main gives fast S-expression/path algebra and kernel primitives. It does
not define Decko card identity, RichText/file handling, permissions, redirects,
or database transaction boundaries. Those must be provided by Decko, DAS, or a
new adapter.
