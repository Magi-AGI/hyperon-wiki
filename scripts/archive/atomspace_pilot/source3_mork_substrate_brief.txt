§4. METHODOLOGY RULES IN FORCE
------------------------------
All five rules established by prior cluster pilots apply unchanged.
Source 3 adds two new rules.

R1 PICKAXE BEFORE "DEAD" (ECAN Source 3 V0-1).
   Particularly relevant for MORK: features may have moved between
   crates during the 8-crate workspace evolution. Pickaxe before
   classifying any feature as "absent" or "removed."

R2 CALLER-ANALYSIS MUST BE TIME-INDEXED (OpenPsi V4-1).
   "X is called by Y" claims must include commit/date snapshot.

R3 BIDIRECTIONAL FORK-DIVERGENCE (OpenPsi V2-1, V4-2; Source 2 V2-8).
   For MORK-atomspace-builder, MORK-rejuve-bio, MORK-leungmanhin,
   MORK-zariuq, MORK2 — "non-canonical / stale by HEAD age" is the
   default classification; full A..B and B..A graph required ONLY
   for narrative-critical fork claims.

R4 C++ AND SCHEME/MeTTa BOTH (ECAN Source 4 V0-1).
   For MORK, the language axes are:
   - Rust (canonical kernel)
   - C (FFI surface in mork_ffi)
   - Python (PeTTa wrapper + python/sandbox/mork/mork.py)
   - MeTTa (lib_spaces.metta, .metta examples in mork_ffi)
   - Prolog (morkspaces.pl in mork_ffi)
   - TypeScript (mork-ts-sdk)
   Either-language-alone-grep produces false-negatives. Cross-grep
   the relevant equivalents in EACH language for any
   feature-presence/absence claim.

R5 CODE COMMENTS / DOC CLAIMS ARE HYPOTHESES (OpenPsi Source 3 brief #8).
   Particularly relevant: docs/ATOMSPACE-INTEGRATION.md and the wiki
   MORK Full card (ID 4194) state numbers and statuses ("500M+ atoms",
   "7-crate workspace", "PLN integration under development") that
   reviewers must verify against the codebase, not trust.

R6 SCHEMA-COMPATIBILITY ≠ STORE-COMPATIBILITY (Source 2 R6).
   Extends to MORK: "uses TrieMap" or "uses PathMap" or "uses RocksDB"
   does NOT imply Decko schema compatibility.

R7 [NEW] MULTI-LANGUAGE WORKSPACE CRATE-COUNT VERIFICATION.
   When the wiki cites a specific number of crates / files / sub-
   components, reviewers must count at HEAD using `Cargo.toml`
   workspace `members` declaration (or equivalent for non-Rust
   workspaces). Off-by-one errors are common in long-form wiki
   prose.

R8 [NEW] PROLOG AS A FIRST-CLASS REFERENCE ORACLE FOR UNIFICATION.
   MORK's PR #49 (2026-04-19) explicitly tests unification semantics
   against Prolog. Therefore: when reasoning about the unification
   behavior covered by PR #49, Prolog reference behavior is a valid
   first-class oracle (not just an analogy). DO NOT generalize this
   to all MORK semantics — the Prolog-equivalence claim is scoped to
   the unification subset that PR #49 actually exercises. For other
   semantic surfaces (sinks, reductions, frontends, FFI), Prolog is
   not authoritative.
   Reviewers should consult morkspaces.pl in mork_ffi specifically
   for unification-behavior verification, not for general MORK
   reasoning.
