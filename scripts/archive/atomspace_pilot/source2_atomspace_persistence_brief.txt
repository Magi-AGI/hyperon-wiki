Source 2 audits whether the AtomSpace and AtomSpace-storage stack can
satisfy the Source 1 acceptance gate (R1.1-R1.4) and the cross-source
forwards (S2.Q1-S2.Q6). The audit is dual-purpose per user mandate:

  (a) WIKI CONTENT CORRECTNESS — verify every status claim in the
      About Hyperon+AtomSpace, AtomSpace Full, and any ECAN/MORK/DAS

      Wiki cards in scope: ID 4017 (concise), ID 4403 (deep dive).
      AtomSpace Full claims under particular scrutiny: "classical C++
      AtomSpace v5.0+ mature", "Space API operational", and any
      backend-readiness claim.

  (b) INTEGRATION USE CASE — answer: which seam, if any, in this
      Source 2 surface can serve as the Decko-AtomSpace bridge under
      the Source 1 latency budget (<500ms fetch / <2s acceptable) and
      Decko-semantics constraints (R1.A5-A7: lossless `+` names,
      RichText inclusion, file-card behavior).

CODEX SOURCE-2 CARRY-FORWARD CAUTIONS (locked into this brief):

  C2-CARRY-1: atomspace-pgres SCHEMA-COMPATIBILITY DOES NOT IMPLY
              DECKO-COMPATIBILITY.
              The key question is NOT "does atomspace-pgres run on
              PostgreSQL?" — it does. The key question is whether it can
              read or mirror Decko's existing data model (the cards table,
              card_actions, card_changes, etc.) or whether it is merely a
              SEPARATE AtomSpace persistence backend that happens to use
              PostgreSQL as its physical store. If the latter, the seam
              still requires a translation/sync layer between two distinct
              schemas — most of the friction the design assumed it
              eliminated remains.

  C2-CARRY-2: KEEP "MCP ADAPTER" SPLIT INTO TWO TERMS.
              Source 1 V1-9 established the disambiguation. Source 2 must
              not collapse them. If Source 2 surfaces an "MCP" reference,
              tag it explicitly as either:
              (a) DECKO-MCP — the deployed Decko card API (live since
                  2025-12-02; mod/mcp_api/, JWT/RS256/JWKS).
              (b) ATOMSPACE-MCP — the proposed Python bridge from
                  docs/ATOMSPACE-INTEGRATION.md (sketch only; no
                  implementation in the local code tree).

GEMINI SOURCE-2 FOCUS (acknowledged):
Gemini's stated focus is the StorageNode contract and the PostgreSQL

§2. KEY QUESTIONS (FROM SOURCE 1 FORWARDS + CODEX CARRY-FORWARDS)
-----------------------------------------------------------------

S2.Q1 [from Source 1] STORAGE NODE / API SURFACE INVENTORY.
      Catalogue the AtomSpace API operations the Source 1 adapter sketch
      assumes: query (pattern matching), mutation (add/delete), values
      and metadata (TruthValue/AttentionValue/CustomValue), link typing,
      and the StorageNode protocol. For each, name the canonical
      header/source file and the supported operations.
      Affected repos: atomspace, atomspace-storage.
      Verify: does the API actually support the adapter's assumed shape?

S2.Q2 [from Source 1] LOSSLESS DECKO REPRESENTATION.
      Determine whether Decko card identity and hierarchy can be
      represented losslessly in AtomSpace. Specifically:
      - `+` hierarchical names (e.g., `Hyperon AI Algorithms+ECAN+AI`)
      - Original-name metadata (canonical vs sanitized form)
      - RichText inclusion cards (`{{+child|content}}` syntax) — content
        cards may differ from inclusion cards in mapping
      - File cards (binary content + storage path)
      - Card history / revisions
      - Permissions
      Each of the above maps to either a verified ATOM TYPE or a verified
      VALUE TYPE in the current AtomSpace; if any mapping is lossy or
      unsupported, flag.

S2.Q3 [HIGH-PRIORITY, from Source 1 R1.B1, expanded by Codex C2-CARRY-1]
      atomspace-pgres COMPATIBILITY WITH DECKO POSTGRES.
      Three sub-questions, in order:
      (a) What schema does atomspace-pgres write to? Catalogue tables,
          columns, indices. Cite the schema-creation SQL (likely under
          `opencog/persist/sql/multi-driver/` or `opencog/atomspace/
          persist/`).
      (b) Can atomspace-pgres point at an existing Decko-Postgres
          database without conflicting with Decko's tables (cards,
          card_actions, card_changes, etc.)? Schema-namespace check.
      (c) Can atomspace-pgres READ from Decko-shaped tables, or does it
          require its own schema and a separate sync layer? This is the
          decisive distinction per Codex C2-CARRY-1.
      Verify with the README + CMakeLists + actual SQL files at the
      atomspace-pgres `6fe9763` snapshot.

      HARD NEGATIVE TEST (Codex Source-2 sign-off requirement):
      Reviewers MUST explicitly distinguish two outcomes and label
      atomspace-pgres as exactly one of them in Section C:
      [BACKING-STORE-ONLY] — atomspace-pgres uses PostgreSQL as its
          internal AtomSpace persistence layer. Its tables are an
          AtomSpace-shaped schema (atoms, links, values) with no
          relationship to Decko's card schema. Pointing it at a Decko
          database either conflicts with Decko tables or simply ignores
          them. NOT a schema-compatibility win — still requires a sync
          or translation layer.
      [DECKO-SCHEMA-COMPATIBLE] — atomspace-pgres can read or write
          Decko's existing `cards`, `card_actions`, `card_changes`
          (etc.) tables in their native shape, mapping rows to atoms
          without intermediate translation. THIS is the schema-
          compatibility win.
      Anything other than [DECKO-SCHEMA-COMPATIBLE] is a NO for the
      "lowest-friction seam" claim. Section C verdict must use one of
      these two literal labels.

S2.Q4 [from Source 1] CLASSICAL C++ ATOMSPACE vs HYPERON SPACE API.
      Do NOT collapse. Catalogue the two API surfaces side-by-side:
      - Classical: opencog/atomspace + opencog/atomspace-storage
        (StorageNode subclasses)
      - Hyperon: hyperon-experimental's lib_spaces.metta + Python
        Atomspace bindings
      Identify whether the Source 1 design sketch (`from hyperon
      import Atomspace, MCP`) lands in classical C++ or Hyperon Rust.
      If Hyperon Rust, verify the `MCP` import. The literal symbol
      `MCP` from a `hyperon` module is suspect — likely either
      apocryphal or pre-implementation.

S2.Q5 [from Source 1] NAME-SANITIZATION CORRECTNESS.
      The Source 1 adapter sketch's `_sanitize` replaces `+`, `:`,
      spaces with `_`. Verify whether AtomSpace `ConceptNode` names
      can carry these characters natively (or via escaping). If yes,
      sanitization is unnecessary and lossy by design; recommend
      direct-name representation. If no, document the necessary
      escaping protocol.

S2.Q6 [from Source 1, ECAN Source 4 V0-1 rule]
      C++ AND SCHEME/MeTTa DUAL-API AUDIT.
      For any "X is supported / X is not supported" claim about the
      AtomSpace API, grep BOTH layers:
      - C++: AtomSpace, getValue, setValue, StorageNode, BackingStore,
        TruthValue, AttentionValue
      - Scheme: cog-set-value!, cog-value, cog-stimulate, cog-av-sti
      - MeTTa: lib_spaces.metta, &storage, get-state, set-state
      Either layer alone produces false-negatives.

S2.Q7 [NEW per Phase 0 finding] atomspace-bridge / BridgeStorageNode.
      Inspect the README + opencog/ headers + examples/ at the
      atomspace-bridge `40b8af7` snapshot. Determine:
      (a) What does BridgeStorageNode bridge BETWEEN?
      (b) Was it migrated from atomspace-storage or another repo?
          (per the commit message, "Migrate the BridgeStorageNode
          to here" — pickaxe `git log --all -S 'BridgeStorageNode'`
          across atomspace, atomspace-storage to find the migration
          source.)
      (c) Is it a viable Decko<->Atomspace seam, or is it bridging
          two AtomSpaces (e.g., classical<->Hyperon)?

S2.Q8 [NEW] atomspace2 vs canonical atomspace.
      Apply the bidirectional-fork-divergence rule (V2-1, V4-2):
      - `git log atomspace..atomspace2 --shortstat` — what's added.
      - `git log atomspace2..atomspace --shortstat` — what's behind.
      Determine whether atomspace2 is a divergent tradition or a stale
      experimental branch. If divergent, identify the architectural
      delta. atomspace2 has a pyproject.toml — note any Python-side
      surface that could simplify the integration.

S2.Q9 [NEW] STORAGE-NODE ECOSYSTEM MAP.
      Produce a SINGLE-PAGE map of all StorageNode subclasses currently
      deployed across atomspace-storage + atomspace-pgres + atomspace-
      rocks + atomspace-cog + atomspace-bridge. Include:
      - Class name, source file, parent class
      - "What it persists" + "what it does not"
      - Tested-vs-untested status
      - Read-only vs read-write
      This map is the canonical artifact reviewers should produce.

      STATUS-LABEL REQUIREMENT (Codex Source-2 sign-off):
      Each entry on the ecosystem map must carry exactly ONE status
      label drawn from this fixed vocabulary:
        CANONICAL              — the reference / interface-defining class
                                 (e.g., StorageNode itself in
                                 atomspace-storage)
        BACKEND-IMPLEMENTATION — concrete persistence backend
                                 (e.g., PostgresStorageNode,
                                 RocksStorageNode)
        BRIDGE-ADAPTER         — bridges between two storage layers,
                                 between AtomSpaces, or between
                                 AtomSpace and an external system
                                 (e.g., BridgeStorageNode,
                                 ProxyStorageNode if extant)
        LEGACY-FORK            — present in a fork that is stale or
                                 superseded; preserved for
                                 archaeological reference only
        EXPERIMENTAL           — provisional, pre-stable API, or
                                 explicitly marked as such by the
                                 author
        UNKNOWN                — reviewer cannot classify with
                                 confidence at this snapshot; flag
                                 for follow-up
      The map (Section I) MUST be a usable integration map, not
      repo-by-repo prose. Tabular form preferred.

ADDITIONAL Source-2-specific rule:

   another PostgreSQL-using application's schema." Always verify schema
   intersection or schema disjointness before claiming low-friction
   integration.
