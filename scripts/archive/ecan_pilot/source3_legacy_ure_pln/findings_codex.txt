## Executive verdict

C3-01. The Source 3 smoking-gun question resolves as: `source_selection_mode::STI` is **dead at current URE HEAD**, but it was **historically implemented and then removed / unwired**. The brief's "dead enum" hypothesis is correct only for current HEAD, not for the full git-history audit.

C3-02. The historical implementation was narrow: forward-chainer source selection could weight candidate source atoms by `s->getSTI()` and then use URE tournament selection. It was not a full ECAN-PLN control loop, not an AttentionBank coupling, not ECAN agent execution, and not evidence that PLN inference was executable-coupled to the ECAN currency economy.

C3-03. Current URE has only residues of attention coupling:

- `source_selection_mode { TV_FITNESS, STI, UNIFORM }` remains in `opencog/ure/forwardchainer/ForwardChainer.h`, but current constructors do not accept a mode and current source selection ignores the enum.
- `URE:attention-allocation` has Scheme wrappers and configs, but `ure-utils.scm` explicitly says it "isn't in use presently", and current `UREConfig` has no corresponding C++ parameter.
- Repo-wide current grep found no active `AttentionBank`, `AttentionValue`, `AttentionalFocus`, or `cog-bind-af` use in URE source paths.

C3-04. Legacy C++ PLN (`pln-singnet`) is a URE rulebase/library plus Scheme wrappers, not an independent attention-controlled runtime. It depends on URE and exposes wrappers such as `pln-fc`, `pln-bc`, and `pln-set-attention-allocation`; the latter inherits URE's dead attention-allocation path. Example configs set `URE:attention-allocation` to `0` / false.

C3-05. The ECAN-PLN cluster narrative should be: a narrow STI source-selection hook once existed in URE and was later removed during source/rule-selection rewrites. The richer ECAN-as-inference-control architecture remained paper-level or design-level at the executable layer. Modern Hyperon did not inherit a working ECAN-PLN control loop; it inherited, at most, a historical narrow hook that had already disappeared from current URE.

## Q1: `source_selection_mode::STI` status

### Current HEAD

C3-06. Current URE still declares the enum:

- `opencog/ure/forwardchainer/ForwardChainer.h:40-43`
- Values: `TV_FITNESS`, `STI`, `UNIFORM`

C3-07. Current URE constructors no longer accept a `source_selection_mode` argument:

- `ForwardChainer.h:58-69`
- `ForwardChainer.h:75-82`

C3-08. Current source selection uses `SourceSet` weights. The only current source-weight formula is TV fitness:

- `opencog/ure/forwardchainer/SourceSet.cc:38-54`
- Formula: `complexity_factor * strength * confidence`
- TODO in source: "Support more fitness functions"

C3-09. Current `select_source` consumes `_sources.get_weights()` and samples a discrete distribution. I found no current switch over `source_selection_mode` in active source paths.

### Historical implementation

C3-10. Git history shows `source_selection_mode::STI` was actually wired in commit `0a0b09912` (`2016-03-09 Misgana Bayetta, "Pass source selection mode to the constructor"`).

Evidence:

- Historical path: `opencog/rule-engine/forwardchainer/ForwardChainer.h`
- Constructor accepted `source_selection_mode sm = source_selection_mode::UNIFORM`
- Historical path: `opencog/rule-engine/forwardchainer/ForwardChainer.cc`
- `switch (_ts_mode)` contained:

```cpp
case source_selection_mode::TV_FITNESS:
    for (const Handle& s : to_select_sources)
        tournament_elem[s] = urec.tv_fitness(s);
    hchosen = urec.tournament_select(tournament_elem);
    break;

case source_selection_mode::STI:
    for (const Handle& s : to_select_sources)
        tournament_elem[s] = s->getSTI();
    hchosen = urec.tournament_select(tournament_elem);
    break;

case source_selection_mode::UNIFORM:
    hchosen = rand_element(to_select_sources);
    break;
```

C3-11. Earlier commit `36fdff1fb` (`2016-02-23 Nil Geisweiller, "Add UNIFORM source selection"`) had the enum shape but defaulted / initialized around uniform behavior. The decisive proof of STI being callable is `0a0b09912`, where the constructor accepted the mode and the switch used `s->getSTI()`.

C3-12. Commit `0b744dbab` (`2018-10-23 Nil Geisweiller, "Re-work source and rule selection in the forward chainer"`) removed the active switch and removed `_ts_mode = sm`. In the patch, the old `STI` case was removed with a comment:

```cpp
An attentionbank is needed in order to get the STI...
```

That comment is important: it shows the rewrite authors knew STI source selection needed attention infrastructure, but the new `SourceSet` path did not implement it.

C3-13. Commit `5a5b7785d` (`2019-01-08 Nil Geisweiller, "Have forward and backward chainer support separate rule-base atomspace"`) removed the `source_selection_mode sm` constructor argument. The enum itself survived as dead residue.

### Q1 classification

C3-15. Not classification: "never implemented" is false.

C3-16. Not classification: "still wired indirectly" is false for current HEAD based on grep and current constructor/source-selection code.

## Q3: URE inference-control architecture inventory

C3-24. URE still contains the mature control-theory stack that Source 7's `chaining/` only sketches. The key classes are real code, not just docs.

### BetaDistribution

C3-25. `opencog/ure/BetaDistribution.h` defines a beta distribution over TruthValues.

C3-26. `opencog/ure/BetaDistribution.cc` maps TruthValue to positive count and total count, then builds alpha/beta parameters with priors.

C3-27. `BetaDistribution::operator()` draws samples via inverse incomplete beta (`boost::math::ibeta_inv(alpha(), beta(), rng.randdouble())`).

### ThompsonSampling

C3-28. `opencog/ure/ThompsonSampling.h` and `.cc` implement a distribution over candidate TruthValues. The code computes per-candidate probabilities using each candidate's beta PDF and the product of other candidates' beta CDFs.

C3-29. `ThompsonSampling::operator()` also supports direct sample-and-max selection by drawing from each candidate's beta distribution.

### ActionSelection

C3-30. `opencog/ure/ActionSelection.h` / `.cc` wrap Thompson sampling for Handle-to-TV action maps. It extracts the TV sequence, computes Thompson probabilities, and maps them back to action handles.

### MixtureModel

C3-31. `opencog/ure/MixtureModel.h` describes a Universal-Operator-Induction-style mixture model over control rules:

- Each model/control rule receives a prior based on complexity.
- Each model's likelihood is represented as a beta distribution.
- Weighted-average TV is computed from these priors and beta factors.

C3-32. `MixtureModel.cc` implements:

- `prior_estimate(model) * beta_factor(model)` weighting
- normalized weighted average of beta means/variances
- complexity prior `exp(-cpx_penalty * length)`

### BIT / AndBIT / BITNode

C3-33. `opencog/ure/backwardchainer/BIT.h` and `.cc` implement Backward Inference Tree machinery.

C3-34. `BITNode::operator()` estimates probability of usefulness from its fitness bounds and current fitness; exhausted nodes return zero.

C3-35. `AndBIT::expand` applies backward chaining expansions to a FocusSet; `AndBIT::select_leaf` samples leaves by computed weights.

### ControlPolicy

C3-36. `opencog/ure/backwardchainer/ControlPolicy.h` / `.cc` implement backward-chaining control-rule selection.

C3-37. If no active control rules are found, `ControlPolicy` falls back to default rule TVs. If active control rules exist, it uses `MixtureModel` to derive success TVs, then `ActionSelection` to choose rules.

C3-38. `ControlPolicy` has explicit support for expansion control rules and recognizes `EvaluationLink` / `ExecutionLink` structures such as `URE:BC:expand-and-BIT` and `URE:BC:preproof-of`.

### BackwardChainer

C3-39. `opencog/ure/backwardchainer/BackwardChainer.h` accepts optional `control_as` and `focus_set` parameters. The comment states focus-set support is not implemented for BC.

C3-40. `BackwardChainer.cc` runs an iterative loop of `expand_bit`, `fulfill_bit`, and `reduce_bit`; expansion selection uses weighted BIT probabilities.

### ForwardChainer / SourceSet / SourceRuleSet

C3-41. Current forward chaining uses:

- `SourceSet`: current source weights from TV fitness (`strength * confidence * complexity_factor`)
- `SourceRuleSet`: source/rule pairs weighted by second-order TVs
- Thompson sampling over source/rule pair TVs

C3-42. This is inference-control code, but not ECAN code. It is a probabilistic control stack over rule/source quality estimates, not attention-currency dynamics.

### Test status

C3-43. URE's active tests are much thinner than its architecture:

- `tests/ure/CMakeLists.txt` actively includes `BetaDistributionUTest`, `ActionSelectionUTest`, and `UtilsUTest`.
- Forward-chainer tests are present but `ForwardChainerUTest` is commented out in the subdirectory CMake.
- Backward-chainer tests are disabled; the CMake file contains a note that code bit-rotted and tests no longer work / URE is no longer supported.

C3-44. Therefore the URE control stack is historically substantial, but its current maintenance/test status is degraded. Do not cite current URE as a healthy maintained inference-control runtime without this caveat.

## Q5: Inference-control meta-learning examples in legacy PLN

C3-53. Source 3 finds an important design/example corpus not present in the URE clone itself: `pln-singnet/examples/pln/inference-control-meta-learning/`.

C3-54. The README explicitly says it is not recommended for new users and is a toy experiment. It describes the loop:

1. Run backward inferences.
2. Create a corpus of inference traces.
3. Run the pattern miner and other learning algorithms.
4. Build inference-control rules from those patterns.
5. Repeat while reusing learned rules to speed the backward chainer.

C3-55. The same README says "No ECAN is necessary." This is strong evidence that the mature URE inference-control direction was trace/control-rule meta-learning, not ECAN currency integration.

C3-56. `distributed-inference-control-learning.md` describes the conceptual distributed version:

- Run reasoning over a collection of problems.
- Record traces of every URE decision.
- Mine hidden patterns in traces.
- Turn patterns into control rules so URE becomes more efficient.

C3-57. `icl.scm` invokes:

```scheme
(pln-bc target #:trace-as trace-as #:control-as control-as)
```

This directly exercises URE's backward-chaining trace/control-atomspace path.

C3-58. `pln-rb.scm` sets backward-control parameters:

- `URE:maximum-iterations`
- `URE:complexity-penalty`
- `URE:BC:MM:compressiveness`
- `URE:BC:MM:complexity-penalty`

C3-59. This corpus is the executable/documentary counterpart to URE's `ActionSelection`, `ThompsonSampling`, `MixtureModel`, `ControlPolicy`, and BIT architecture.

C3-60. It is not evidence of ECAN-PLN integration. It is evidence that URE/PLN pursued a separate Bayesian/control-rule meta-learning route.
