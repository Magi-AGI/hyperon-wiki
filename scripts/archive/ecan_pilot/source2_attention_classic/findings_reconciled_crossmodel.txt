## Headline answers (cluster-pilot questions)

### Q1: Is metta-attention a direct continuation of Classic ECAN, or a structurally inspired reimplementation?

**A (consensus):** **Direct continuation / partial port of the Classic C++ pragmatic variant**, not a paper-level reimplementation. The MeTTa code's "engineered surrogates" map line-by-line onto Classic C++'s engineered surrogates: target-funds/buffer rent, fund-buffer central-bank wage adjustment, normalized-STI fuzzy conjunction Hebbian, deterministic exponential decay. **Both Codex and Gemini converge:** the MeTTa port did not lose the paper math — Classic C++ had already lost it, and the MeTTa port preserved the loss faithfully.

### Q2: Are the MeTTa "engineered surrogates" simplifications, renamed equivalents, partial ports, or regressions?

**A (consensus):** **Mostly faithful ports of pre-existing Classic C++ surrogates.** Specific exceptions where the MeTTa port introduced new regressions:
- **`LOCAL_FAR_LINK_RATIO` parameter shadowing** (Source 1 finding `HebbianCreationAgent.metta:17` `localToFarLinks 1000.0`): a **MeTTa-port regression**. Classic C++ correctly reads `LOCAL_FAR_LINK_RATIO = 10` from `AttentionParamQuery` at `HebbianCreationAgent.cc:71-77`; default is `10` per `default-param-values.scm:61-64`. The 100×-larger hardcoded value in MeTTa is a port-introduced bug.
- **`recentVal` infrastructure dormancy** (Source 1 finding `recentVal.metta:22-67` updates commented out): a **MeTTa-port regression**. Classic C++ keeps `_maxSTI` and `_minSTI` as live `opencog::recent_val` instances (`ImportanceIndex.h:58-60`, updated at `ImportanceIndex.cc:110-135`). MeTTa lost the smoothing infrastructure that Hebbian normalization depends on.
- **WA rent compounding bug**: **inherited from Classic C++**, not introduced by the MeTTa port. `WARentCollectionAgent.cc:68-70` mutates member `STIAtomRent`/`LTIAtomRent` in place every WA rent event (orchestrator-verified; see V0-1 below).
- **Stochastic-importance-diffusion misnaming**: **inherited from Classic C++** (`StochasticImportanceDiffusion.cc/.h` is also deterministic decay; no `tanh`/`sigmoid` in active C++).

### Q3: Was Classic ECAN ever coupled to PLN/URE in executable code, or only in papers?

**A (consensus):** **Minimal/legacy URE coupling, no executable PLN inference-control integration.** Both Codex and Gemini converge on a single URE-facing utility:
> `cog-bind-af` documented as "A special-purpose pattern matcher used by the URE" at `attention-bank.scm:272-276` (orchestrator-verified). The primitive is defined in `AttentionSCM.cc`. This is a **URE-facing AtomSpace utility at the AttentionBank boundary**, not evidence that ECAN agents drive PLN inference control.

Other near-hits (paper/comment level only, not executable coupling):
- `experiments/README.md:2` labels the experiment "ECAN/PLN experimentation" but stated goals are HebbianLink creation, STI propagation, forgetting (no PLN inference invocation).
- `ExperimentSetupModule.h:1-9` preserves the legacy header guard `PLNDYNAMICSEXPSETUPMODULE_H_` from when the file was named `PLNDynamicsExpSetupModule.h`, but the active includes (`:11-15`) reference only experiment, CogServer, Agent, AttentionModule headers — no PLN headers.
- Many `TruthValue` hits are AtomSpace TruthValue usage, not PLN runtime coupling. `HebbianUpdatingAgent.cc:92-95` uses `SimpleTruthValue::createTV(tc, 0.1)` and `merge(newtv)` — that's AtomSpace machinery, not a PLN rule engine.

**Cluster verdict:** the Hyperon "missing control layer" was **already absent in OpenCog Classic** at the executable-code level. ECAN was architecturally/paper-coupled to PLN and exposed AF-aware utilities URE could *use*, but it was never a PLN inference-control runtime. The cluster narrative simplifies accordingly: this is not a Hyperon-era regression — it is a continuation of the OpenCog Classic state.

### Q4: Cluster-tradition decision — sixth tradition vs Tradition 2 extension?

**A (Codex C2-82, orchestrator-confirmed):** **Tradition 2 extension** with a sub-branch note, not a sixth tradition. Specifically: "iCog-Labs-Dev / SingularityNET Ethiopia maintained a modern MeTTa/C++ attention line isolated from the trueagi-io PLN production line." Reasoning:

The "iCog-Labs-Dev modern MeTTa/C++ ECAN line" therefore extends Tradition 2 (Historical OpenCog / URE / ECAN control), with the note that **only the attention sub-system was actively ported** — the URE/PLN inference-control side of Tradition 2 was not carried forward to MeTTa by this cohort.

### V0-1 — WA rent compounding state (Codex C2-85 reconciliation request)

**Codex flag:** "Confirm with Claude/Gemini whether WA rent compounding should be treated as a bug or intentional elapsed-time scaling. My reading: bug-prone state mutation, because it changes the member base rent rather than using a local scaled variable."

**[Claude orchestrator, repo-verified]** Direct read of `WARentCollectionAgent.cc:60-83`:
```cpp
60: {
61:     for (const Handle& h : targetSet) {
62:         AttentionValue::sti_t sti = get_sti(h);
63:         AttentionValue::lti_t lti = get_lti(h);
64:
65:         float last_update_time = _sdac.elapsed_time(h);
66:         std::cout << "last_updates " <<last_update_time<<std::endl;
67:         std::cout << "STIAtomRent: " <<STIAtomRent<<std::endl;
68:         STIAtomRent = STIAtomRent * last_update_time;
69:         std::cout << "STIAtomRent after multiplication: " <<STIAtomRent<<std::endl;
70:         LTIAtomRent = LTIAtomRent * last_update_time;
71:
72:         double stiRent = calculate_STI_Rent();
73:         ...
```

**Resolution:** **Confirmed as a likely bug.** Two corroborating signals:
- **`std::cout` debug-print scaffolding remains** in production source (`:66-67, :69`), suggesting this code is in active debugging state, not finalized intent.
- **Per-iteration mutation of member fields** inside a `for` loop over `targetSet`. If `targetSet` ever has more than one element (it currently doesn't because `WAImportanceDiffusionAgent::diffusionSourceVector` returns one random non-AF atom), the second iteration would compound on the first iteration's mutation. Even with single-element targets, **subsequent runs of the agent would compound across runs** because member fields persist.
- The intent appears to be "scale rent by elapsed time for this WA rent event," which would be correctly expressed with local variables: `double scaled_rent = STIAtomRent * last_update_time;`. The in-place mutation form is bug-shaped.

**However**, since this is **in Classic C++**, the MeTTa port at `WARentCollectionAgent.metta:30-31` is faithfully porting the Classic bug, not introducing a new one. Source 1's "WA rent compounding bug" should be reclassified as an **inherited Classic C++ bug**, not a MeTTa-port-only artifact.
