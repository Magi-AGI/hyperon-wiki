## Why this source, why now

Source 3 reconciliation settled the cluster-narrative-deciding question: ECAN-PLN coupling **was** narrowly executable (URE forward-chainer STI source-selection, 2016-03-09 to 2018-10-23) but was deliberately unwired in the 2018 URE rewrite (commit `0b744dbab`, "An attentionbank is needed in order to get the STI..."). The current standalone URE and legacy PLN repos preserve only the post-removal state.

Codex flagged four open questions (C3-80 through C3-83) that the standalone repos cannot answer:

1. **Scheme-binding history**: was `source_selection_mode::STI` ever exposed through Scheme APIs to PLN users before its 2018 removal? The C++ constructor path is verified; the historical Scheme bindings are not yet reconstructed.
2. **Pre-split monorepo coupling**: did the monorepo era contain stronger AttentionBank-to-URE / AttentionBank-to-PLN coupling than the standalone repos preserve? Standalone repos have post-split state only.
3. **`cog-bind-af` historical consumers**: Source 2 confirmed `cog-bind-af` exists as a URE-facing pattern matcher in `attention-bank.scm:272-276`. Did anything actually consume it in the monorepo era?
4. **`pln-set-attention-allocation` evolution**: was there ever a working version that did more than wrap URE's dead hook?

**Source 4's job is to settle these four questions definitively, then close the ECAN cluster pilot extraction phase.**

Gemini and Codex both pre-approved Source 4 = pre-split monorepo archaeology in their Source 3 review responses.

## Comparison targets — five primary deliverables

### Q1 (highest priority): Was `cog-bind-af` ever consumed by anything?

`cog-bind-af` is the URE-facing pattern-matcher utility documented at `opencog/attentionbank/scm/attention-bank.scm:272-276` (Source 2 finding). It's defined in `AttentionSCM.cc`. The Source 3 standalone URE pass found no consumer in URE source. **Source 4 must verify whether anything in the pre-split monorepo era ever called `cog-bind-af`.**

**Methodology:**

1. `git log --all -S 'cog-bind-af'` in `opencog-singnet/` — find all commits that added or removed `cog-bind-af` references. Report SHAs + dates + paths.
2. At commit `b31c7e3`, run `git grep cog-bind-af b31c7e3 -- '*.scm' '*.py' '*.cc' '*.h'` — find every active reference at the snapshot.
3. Cross-reference with `opencog/pln/` and `opencog/openpsi/` at `b31c7e3`. Did either subsystem invoke `cog-bind-af`?
4. If active consumers exist: cite them. If only definition + documentation hits: confirm dead utility.

**Outcome interpretation:**
- **Active consumers exist**: ECAN-PLN coupling existed via this Scheme bridge, separate from the URE C++ STI hook. Significant cluster-narrative addition.
- **No active consumers, ever**: utility was always paper-aspirational. Cluster narrative locks in.

### Q2: Did `opencog/pln/` (in-monorepo legacy C++ PLN) reference AttentionBank?

This is **distinct from** the standalone `pln-singnet/` audited in Source 3 (HEAD `a0983ada8`, 2021-10-28). The in-monorepo `opencog/pln/` at `b31c7e3` (2019-09-04) is an **earlier state** of the same codebase, before the split.

**Methodology:**

1. `git ls-tree -r b31c7e3 -- opencog/pln/` — inventory in-monorepo PLN subdirs and files. Compare to standalone `pln-singnet/` HEAD `a0983ada8` to see what was added/removed in the post-split period.
2. `git grep -n -E 'AttentionBank|AttentionalFocus|AttentionValue|attentionvalue|cog-bind-af|cog-av-sti|getSTI|set_sti|URE:attention-allocation' b31c7e3 -- opencog/pln/` — find every coupling reference at the snapshot.
3. Pickaxe history: `git log --all -S 'AttentionBank' -- opencog/pln/` and `git log --all -S 'getSTI' -- opencog/pln/` — find any pre-`b31c7e3` commits that wired AttentionBank into PLN and were later removed.
4. Compare against `pln-singnet/` standalone. If `opencog/pln/` at `b31c7e3` had AttentionBank references that `pln-singnet/` `a0983ada8` does not, cite the diff.

**Outcome interpretation:**
- **Active references with non-trivial logic**: legacy C++ PLN did directly read AttentionBank — significant additional ECAN-PLN coupling beyond the URE STI hook. Standing correction.
- **Wrapper/config-only references** (e.g., `pln-set-attention-allocation` setting URE's flag): consistent with Source 3 finding that PLN delegates to URE; no new coupling.
- **No references**: confirms PLN was URE-mediated and ECAN-decoupled even in the monorepo era.
