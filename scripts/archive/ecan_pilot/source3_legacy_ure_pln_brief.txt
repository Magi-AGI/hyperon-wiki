## Why this source, why now

Sources 1 + 2 reconciliation answered: **the PLN-cluster "missing inference-control layer" was already absent at the executable-code level in OpenCog Classic ECAN.** ECAN exposed `cog-bind-af` as a URE-facing utility but did not drive PLN inference control.

Codex and Gemini both recommended Source 3 = `opencog/opencog` monorepo to verify whether ECAN-PLN executable coupling **ever** existed in the pre-split codebase, or whether the 2009 paper's control-layer aspirations were "purely theoretical."

**Orchestrator finding during scoping:** the local `opencog/opencog` monorepo at HEAD `ae68bda` (2025-08-26, Linas VepЕЎtas, "Fix broken header includes") has been **fully trimmed** вҖ” only `nlp/`, `ghost/`, `eva/`, `openpsi/`, `cython/`, `neighbors/` subdirectories remain. The historical `attention/`, `pln/`, `ure/` subdirectories were split out into separate repos years ago and are no longer present in the monorepo at HEAD. The de-facto pre-split source is therefore distributed across three legacy standalone repos:

- `opencog/attention` вҖ” Source 2 (already audited)
- `opencog/ure` вҖ” Source 3a (this brief)
- `opencog/pln` вҖ” Source 3b (this brief, via local `pln-singnet/` fork)

**Smoking-gun preview from orchestrator scoping:** `opencog/ure/opencog/ure/forwardchainer/ForwardChainer.h:40-43` declares an enum `source_selection_mode { TV_FITNESS, STI, UNIFORM }`. The `STI` value is the architectural hook for ECAN coupling вҖ” URE forward chaining could have selected inference sources weighted by their STI (Short-Term Importance, the ECAN attention currency). However, the actual weight calculation at `SourceSet.cc:38-54` (`calculate_weight`) **only implements the `complexity_factor * strength * confidence` (TV_FITNESS) variant**; the implementation comment says "for now only one fitness function is hard coded // TODO: 1. Support more fitness functions". The `STI` enum value is **declared but never wired up**.

This is the cluster-narrative-deciding finding. Source 3's job is to verify rigorously:

1. Is `source_selection_mode::STI` actually dead enum, or did it ever have a working implementation that was later removed?
2. Are there other ECAN-coupling architectural hooks in URE/PLN that were left dead?
3. Does any URE/PLN code import AttentionBank, AttentionalFocus, or AttentionValue headers?
4. Did URE/PLN ever drive ECAN agents (write back STI)? Or was the data flow always one-way (ECAN вҶҗ AtomSpace, URE вҶҗ AtomSpace, no URE вҶ” ECAN direct path)?
5. What was the canonical PLN/URE inference-control architecture (MixtureModel, ThompsonSampling, BetaDistribution, ActionSelection вҖ” all visible at `opencog/ure/opencog/ure/`), and how does it relate to the PLN cluster Source 7 finding that ~5/16 features survived in modern MeTTa `chaining/`?

The answer to #1 alone, if it confirms dead enum, **closes** the cluster-narrative open item: "the lost glue never existed; the 2009 control-layer aspiration was architected but never wired up; both Classic and Hyperon are continuations of an unrealized intent."

## Comparison targets

### Q1 (highest priority): Was `source_selection_mode::STI` ever implemented?

**Orchestrator scoping evidence (verbatim from `opencog/ure/opencog/ure/forwardchainer/SourceSet.cc:38-54`):**
```cpp
double calculate_weight(const Handle& bdy, double cpx_fctr)
{
    // Calculate weight, for now only one fitness function is hard
    // coded
    //
    // complexity_factor * strength * confidence
    //
    // The minimum value is 1e-16 to not ignore completely the source
    // when the it is a default TV.
    //
    // TODO:
    // 1. Support more fitness functions
    // 2. Explicitely turn the fitness into a probability of success
    TruthValuePtr tv = TruthValueCast(bdy->getValue(truth_key()));
    double fitness = tv->get_mean() * tv->get_confidence();
    return std::max(1e-16, cpx_fctr * fitness);
}
```

**The `STI` enum value at `ForwardChainer.h:42` is never read by `calculate_weight`.** The function only returns `cpx_fctr * tv->get_mean() * tv->get_confidence()`.

**Agents to verify rigorously:**
- Run `git log -p --all -- opencog/ure/forwardchainer/SourceSet.cc opencog/ure/forwardchainer/ForwardChainer.cc opencog/ure/forwardchainer/ForwardChainer.h` and look for any historical commit that wired up STI weighting and was later removed. (If yes: cite SHA + author + date.)
- Repo-wide `rg "source_selection_mode|::STI\b|sti.*weight|STI.*weight|attentionvalue|AttentionBank|AttentionalFocus"` across active C++ source. Report all hits.
- Check `UREConfig.cc/.h` вҖ” is there a config option for source-selection mode? Is it parameterized at the URE config level?
- Check `examples/` and `tests/` вҖ” does any test or example use STI-weighted source selection? If yes: how does it set that mode?

**Outcome interpretation:**
- **If `STI` is confirmed dead enum** (declared, no implementation path, no historical implementation in git log): the cluster-narrative finding is **"the lost glue never existed in executable code."** This closes the missing-control-layer question definitively. The 2009 paper's ECAN-PLN coupling was **architected but never realized**.
- **If `STI` was once implemented and later removed:** find the commit, attribute it, and report. This becomes a **"the lost glue existed and was deliberately removed"** finding вҖ” different cluster narrative.
- **If `STI` is wired up via a path the orchestrator missed** (e.g., in a base class, via a config callback): cite the path. This is a **"the lost glue is alive but rarely used"** finding.
