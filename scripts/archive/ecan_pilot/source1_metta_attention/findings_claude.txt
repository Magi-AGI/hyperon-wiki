================================================================================
CLUSTER HEADLINE QUESTION (answered up front; defended in body)
================================================================================

**Q: Has the ECAN attention-currency mechanism (rent / wages / Hebbian update /
diffusion stochastic from the 2009 ECAN paper) actually been rebuilt in MeTTa,
or is `metta-attention/` a structural scaffold that mirrors OpenCog Classic
ECAN at the directory level without realizing the equations?**

**A (Claude single-pass):** It is **a working currency-economy runtime that
implements about half of the 2009 paper's mechanics, with three of the four
canonical equations replaced by simpler engineered surrogates and one (the
stochastic-diffusion sigmoid) absent entirely.** The repo is *not* empty
scaffolding — there is real arithmetic, real STI/LTI fund accounting, real
agent loops, real co-located tests, and a working "shifting and drifting"
experiment. But it is also *not* a faithful port of the 2009 paper:
- **Rent**: implemented as a fund-difference proportional formula, NOT the
  paper's `Rent * max(0, log2(si / recentMaxSTI))^2`.
- **Wages**: implemented as the paper's clamped-fund-difference formula,
  closely faithful.
- **Hebbian update**: implemented as a normalized-STI conjunction PLUS a
  PLN-book revision merge, NOT the paper's
  `c_ij' = ConjDecay * (si*sj/recentMaxSTI^2) + (1 - si*sj/recentMaxSTI^2) * c_ij`.
- **Stochastic diffusion (focus-boundary sigmoid)**: ABSENT; the
  AF-vs-WA split substitutes a categorical "are you in the AF list?"
  threshold for the paper's smooth `0.5*(1 + tanh(shape*(si - FocusBoundary)))`
  probability.

Compared to the PLN-cluster pilot's URE/chaining finding (where the inference
control architecture was simplified down to toy stubs in `chaining`),
metta-attention is **substantially more complete than chaining was relative to
URE**. ECAN in MeTTa is at "real but engineered" level, not "stub" level. But
the shape of the loss is the same: paper formulas were replaced with simpler
fund-bookkeeping that has the right *sign* but not the right *math*.

The "missing control layer" gap from the PLN pilot is **NOT closed** by this
source — there is zero PLN integration code anywhere in the import graph
(see ECAN-PLN integration audit below). metta-attention is a self-contained
attention runtime that nobody currently wires to PLN.

[CLAIM 5] The runner is named `test-superpose` rather than e.g.
`run-attention-cycle`. This naming, combined with its presence directly under

is a historical artifact, not a "test-only harness". Both `experiments/experiment.metta:76`

**Path:** `attention/ForgettingAgent/ForgettingAgent.metta`

**Implements:** the ForgettingAgent of 2009 ECAN — removes atoms with low

**Code-implementation status:** IMPLEMENTED but UNWIRED — see CLAIM 4
(commented out of `runner-Def.metta`). Internally, code is genuine:
- Filters atoms with `getLti $atom <= forgetThreshold` AND `getVlti == 0`

- Sorts atoms ascending by LTI then mean (`:59-74`).
- Removes atoms whose only incoming set is `ASYMMETRIC_HEBBIAN_LINK`

[CLAIM 8] The ForgettingAgent enforces the protect-Hebbian-only invariant by

restricted to `ASYMMETRIC_HEBBIAN_LINK` and `$niset` is the unrestricted

contains anything other than Hebbian links, it is preserved. This matches
the 2009 paper's "do not forget atoms with non-attentional incoming
structure" principle.

[CLAIM 9] LTI rather than STI is the forgetting key
(`ForgettingAgent.metta:33,41,86,88,103,105`), matching 2009 ECAN: STI
controls "what's important now", LTI controls "what's worth keeping".

`(0 0 0)` before removal, which (via `setAv`'s side effect on FUNDS_STI /
FUNDS_LTI in `getter-and-setter.metta:38-41,68-81`) returns the atom's
currency to the central bank. This is the **only fund-conservation
mechanism on the forgetting path**, and because the agent is unwired in the
runner, it never fires in the canonical cycle (claim 4).

### Module: HebbianUpdatingAgent.metta

**Path:** `attention/HebbianUpdatingAgent/HebbianUpdatingAgent.metta`
**LOC:** ~163
**Imports:** uses `getAttentionParam`, `getAfAtoms`, `getRandomAtomInAF`,
`getNormalisedZeroToOneSTI`, `getSti`, `getStv`, `setStv`, `bach-p`,
`getOutGoingAtoms`.
**Implements:** updates the strength/confidence of `ASYMMETRIC_HEBBIAN_LINK`
atoms based on the current STI of the link's endpoints. Maps to 2009
paper's "Hebbian update" / "target conjunction" concept.

**Code-implementation status:** PARTIAL — present, runs, computes
non-trivial values. But **the formula does not match the 2009 paper's
Hebbian conjunction equation `c_ij' = ConjDecay * (si*sj/recentMaxSTI^2) +
(1 - si*sj/recentMaxSTI^2) * c_ij`**. See claim 14.

**Code evidence:**
- `attention/HebbianUpdatingAgent/HebbianUpdatingAgent.metta:14-25`
  `HebbianUpdatingAgent-Run`: picks one random AF atom, calls
  `updateHebbianLinks` on it.
- `:43-65` `bachUpdateHeb`: pulls `$tcDecayRate = TC_DECAY_RATE = 0.1`
  (`AttentionParam.metta:24`); computes `$new-tc = targetConjunction(outgoing)`;
  combines with old TC as
  `$tc = ($tcDecayRate * $new-tc) + ((1 - $tcDecayRate) * $old-tc)`;
  wraps in `($tc 0.1)` then revises with old TV using PLN-book revision.
- `:77-93` `targetConjunction`: gets normalized STIs for source and target
  via `getNormalisedZeroToOneSTI` (which uses MAX_STI / MIN_STI from
  AttentionParam), then computes
  `$conj = (norm_i * norm_j) + (norm_j - norm_i) * abs(norm_j - norm_i)`,
  and rescales with `(conj + 1.0) / 2.0`.
- `:106-130` `mergeCalculation` PLN_BOOK_REVISION: count-based revision
  using `DEFAULT_K = 800`, with novelty discount `CVAL = 0.2`.

[CLAIM 13] The Hebbian update IS in the canonical update form
`new = decay * input + (1 - decay) * old` with `decay = TC_DECAY_RATE = 0.1`
(`HebbianUpdatingAgent.metta:53,56`). The exponential-moving-average
shape matches the 2009 paper's `ConjDecay`-driven update.

[CLAIM 14] The "target conjunction" used as `input` to that EMA is
**NOT** the paper's `si*sj/recentMaxSTI^2`. The implementation
(`HebbianUpdatingAgent.metta:88`) computes
`(norm_i * norm_j) + (norm_j - norm_i) * abs(norm_j - norm_i)`,
which is roughly `min(norm_i, norm_j)^2 + (norm_j - norm_i)*|norm_j - norm_i|`
or something similar — a custom asymmetric conjunction-like score — and
then rescales by `(x + 1.0)/2.0` (line 89) to keep it in [0,1]. The
normalization in `getNormalisedZeroToOneSTI` uses
`MAX_STI - MIN_STI` (importance-index.metta:220-235) where the params have
the perverse defaults `MIN_STI 100000, MAX_STI 0`
(AttentionParam.metta:4,6) — meaning the normalizer can be zero or
negative until those values are updated by the runtime. The `update`
function in `getter-and-setter.metta:476-493` does call
`updateAttentionParam MIN_STI / MAX_STI` based on actual atom STIs, so the
normalization becomes meaningful only AFTER atoms have been stimulated.

[CLAIM 15] The 2009 paper's `recentMaxSTI` (a sliding-window-decayed running
maximum used for both Hebbian update and rent) is **absent**. The code
substitutes the current `getMaxSTI()` (which is just the lifetime max so
far). See claim 2 — the recent-val infrastructure exists as commented-out
code only.

[CLAIM 16] After the EMA-driven `tc` is computed, it is re-merged with the
OLD truth value via `PLN_BOOK_REVISION` count-based merging
(`HebbianUpdatingAgent.metta:58,109-125`). This is a **double-update**: the
EMA already incorporates the old TC at decay rate, then the merge step
incorporates it again at count-weighted ratio. The 2009 paper does not
describe this; it appears to be a 2026-era engineering choice that biases
toward old values. This is a SUBSTANTIVE deviation from the paper.

[CLAIM 17] Diffusion in metta-attention is **deterministic-by-membership**

(see below) diffuses from a uniformly-random non-AF atom. There is **no

as specified in the 2009 paper**. The "focus boundary" is replaced by a hard
membership predicate `isAtomInAf` (`attentional-focus.metta:38-40`).

[CLAIM 18] The 95/5 split between non-Hebbian incident diffusion and
Hebbian-link diffusion (controlled by
`HEBBIAN_MAX_ALLOCATION_PERCENTAGE = 0.05`) is hardcoded in the formulas at
`AFImportanceDiffusionAgent.metta:33,47` and corresponds to one of the
paper's tunables. So this part is faithful to the paper structure even
though the dynamics around it are not.

### Module: ImportanceDiffusionBase.metta

**Path:** `attention/ImportanceDiffusionAgent/ImportanceDiffusionBase/ImportanceDiffusionBase.metta`
**LOC:** ~366
**Imports:** uses `getAttentionParam`, `getSti`, `getMean`, `getConfidence`,
`incidentAtoms`, `getOutGoingAtoms`, `concatTuple`, `setSti`,
`getTargetNeighborsAFI`, `diffusedValue`, `tradeList` (and is the source of
many of these for downstream callers).
**Implements:** the shared math machinery used by both AF and WA diffusion
agents: `diffuseAtom`, `probabilityVectorIncident`,
`probabilityVectorHebbianAjacent`, `combineIncidentAdjacentVectors`,
`calculateDiffusionAmount` (AF), `calculateDiffusionAmountWA` (WA),
`tradeList`, `tradeSti`, `filterTrade`.

**Code-implementation status:** IMPLEMENTED, central.

**Code evidence:**
- `ImportanceDiffusionBase.metta:166-168` AF diffusion amount:
  `MAX_SPREAD_PERCENTAGE * sti`.
- `ImportanceDiffusionBase.metta:174-189` WA diffusion amount:
  `sti - diffusedValue(atom, MAX_SPREAD_PERCENTAGE)` where `diffusedValue`
  is `sti * (1 - decayRate)^elapsedTime` from `stochastic-importance-diffusion.metta:108-119`.
  This is a **decay-based** diffusion-amount calculation: how much STI is
  "ready to diffuse" since last visit.
- `ImportanceDiffusionBase.metta:269-277` `calculateHebbianDiffusionPercentation`:
  multiplies link strength × confidence to get the Hebbian transfer weight.
- `ImportanceDiffusionBase.metta:301-319` `tradeSti`: deducts STI from
  source, adds to target — this is the **only fund-preserving STI transfer
  primitive** in the diffusion path.
- `ImportanceDiffusionBase.metta:341-365` `tradeList`: aggregates many
  pending trades and computes net delta per atom — note the comment hint
  that this aggregation is what allows the agent to be "right by accident"
  on conservation: trades are zero-sum at the math level, but only if
  `setSti` is called via `bach-p setStis` AFTER all trades are summed
  (which it is, in `AFImportanceDiffusionAgent.metta:93`).

[CLAIM 19] The diffusion path's STI conservation is mediated by `tradeList`
+ `bach-p setStis $trades` in two stages: (1) collect pending
`(source, target, value)` trades, (2) per atom, compute net delta as the
sum of incoming minus outgoing, (3) call `setSti` (which goes through
`setAv`, which has the FUNDS_STI bookkeeping side effect). For a balanced
zero-sum trade list, the net effect on FUNDS_STI is approximately zero —
"approximately" because every pass through `setSti` adjusts FUNDS_STI by
the per-atom delta, and floating-point accumulation can drift.

[CLAIM 20] The WA agent's `calculateDiffusionAmountWA` uses
`sti - diffusedValue(atom, MAX_SPREAD_PERCENTAGE)`
(`ImportanceDiffusionBase.metta:175-189`). This is the closest the repo
comes to the paper's stochastic mechanism: rather than a Bernoulli decision
of "diffuse or not", the agent computes "how much STI has accumulated since
last diffusion" as a deterministic decay-from-last-visit estimate. This is
a credible re-engineering of the paper's mechanism but is NOT the paper's
mechanism.

3. CONSERVATION-LAW AUDIT
================================================================================

[CLAIM 49] STI conservation invariant — claimed: `FUNDS_STI + Σ atom_sti =
STARTING_FUNDS_STI`. Mediated by:
  - `setAv` (`getter-and-setter.metta:16-48`) recomputing `FUNDS_STI`
    from `oldSti - newSti` delta.
  - `attentionValueChanged` (`getter-and-setter.metta:68-81`) writing the
    new `FUNDS_STI` back.

[CLAIM 50] LTI conservation invariant — same mechanism, parallel for LTI.

[CLAIM 51] **Conservation is NOT enforced — it is bookkeeped.** Nothing in
the code raises an error if total mass changes. The mechanism is "every
write to atom STI updates the bank by the negative delta", which is
arithmetically correct but vulnerable to:
  - Floating-point drift across many small updates.
  - Concurrent writes via `with_mutex` per agent (humutex, hcmutex,
    afmutex, afrmutex, warmutex, wamutex) — each mutex protects its own
    agent but they don't share a global STI-system mutex, so cross-agent
    races on `FUNDS_STI` writes via `setAv` are possible.

[CLAIM 52] **The `WARentCollectionAgent` parameter-mutation bug (claim 27)**
is a conservation-law violation in disguise: by overwriting
`StartingAtomStiRent` with a time-scaled local value, the next AF or WA
rent invocation pays a different rate than configured. The base STI rent
parameter is meant to be a constant; if it drifts, total system rent over
time deviates from the homeostatic-target schedule.

[CLAIM 53] **Forgetting refunds are correct in principle** (`refundAV` sets
AV to (0,0,0) which routes through `setAv` and refunds the full STI/LTI to
the bank, `ForgettingAgent.metta:210-215` + `getter-and-setter.metta:38-41`),
but **the agent is unwired in the runner** (claim 4). So while the code
exists for fund-recovery on forget, it never fires in canonical runs.

[CLAIM 54] AF eviction (`attentional-focus.metta:14-19`) does NOT touch
typeSpace AV, so it is fund-neutral by construction. ✓

[CLAIM 55] `tradeList` + `bach-p setStis` (`AFImportanceDiffusionAgent.metta:90-94`)
sums all pending diffusion trades and applies them via `setSti`. Because
each `setSti` routes through `setAv`, and the trades are zero-sum at the
math level, the bank should not net-change from a balanced diffusion pass.
But again, every transition goes through the same `setAv` that mutates
FUNDS_STI, so there is a per-trade fund-fluctuation. Over a balanced trade
list this should net to zero modulo float drift. **Not formally verified.**

8. STANDING-CORRECTIONS CANDIDATES
================================================================================

[CLAIM 76] **`localToFarLinks` parameter shadowing**
(claim 11, repeats here): `HebbianCreationAgent.metta:17` hardcodes
`(localToFarLinks) 1000.0` while `AttentionParam.metta:15` configures
`LOCAL_FAR_LINK_RATIO 10.0`. The two values disagree by 100×; the in-file
hardcoded value wins. Either the parameter is dead and should be removed
from AttentionParam, or the agent should read from the parameter.

[CLAIM 77] **`DIFFUSION_TOURNAMENT_SIZE` is dead** (claim 21): the param
is set in AttentionParam but never read by any module. The 2009 paper's
"diffusion tournament" mechanism (pick K candidates, run a tournament,
diffuse to the winner) is **not implemented**. The WA agent picks ONE
random atom; the AF agent uses top-K with `TOPK = 1.0` from AttentionParam,
so it picks one. Standing-correction: the tournament size param either
needs to be wired in or removed.

[CLAIM 78] **`RENT_TOURNAMENT_SIZE` is dead** (parallel to claim 77):
`AttentionParam.metta:23` sets `RENT_TOURNAMENT_SIZE 5.0` but no agent
reads it. The AF rent agent applies rent to ALL AF atoms; the WA rent
agent applies rent to ONE atom. No tournament.

[CLAIM 79] **`recentMaxSTI` / `recentMinSTI` are absent at runtime**
(claims 2 & 47): the `recentVal` infrastructure is commented out; the
2009 paper's sliding-window-decayed running max/min is not in the live
code. Standing-correction: either commit to the simplified lifetime
max/min (and remove the dead recentVal module) or wire it in.

[CLAIM 80] **The `StartingAtomStiRent` parameter is mutated as a side
effect by `WARentCollectionAgent`** (claim 27): `:30-31` writes back
into the param store on every call. This is a likely bug — the param is
nominally a constant rate. AF agent does not do this. Standing-correction
candidate of high importance: either intentional and undocumented, or a
real bug.

[CLAIM 81] **`merge-bin.metta` is dead infrastructure**
(claim 37): `&past-space` and `&recent-space` are bound but never
populated; `mergeBin` is defined but not called. Standing-correction
candidate.

[CLAIM 82] **`stochastic-importance-diffusion.metta` is misnamed**
(claim 36): the module computes a deterministic decay-since-last-visit
estimate, not a stochastic transition probability. Future readers will be
misled. Standing-correction candidate (rename to e.g.
`bin-elapsed-time.metta`).

[CLAIM 83] **`ForgettingAgent` is dead in the runner**
(claim 4): commented out of `runner-Def.metta:10` and excluded from the
test runner per `scripts/run-tests.py`. Either it doesn't work yet or the
canonical loop doesn't need it. Standing-correction candidate.

[CLAIM 84] **The `MAX_STI 0` and `MIN_STI 100000` defaults are inverted**:
`AttentionParam.metta:4-6` initializes `MIN_STI 100000` (very high) and
`MAX_STI 0` (very low). This is **deliberate** — the `update` function in
`getter-and-setter.metta:476-493` only updates these if a real STI value
crosses the (inverted) thresholds, so they self-correct on first
stimulation. But code that reads MAX_STI/MIN_STI before any stimulation
has occurred (e.g., normalization at startup) will hit divide-by-zero or
nonsense values. Defensive coding inside `getNormalisedZeroToOneSTI`
(`importance-index.metta:223-225`) handles `normaliser == 0` by returning
0, but the silent zero-result can mask logic bugs. Worth a wiki note.

[CLAIM 85] **`requirments.txt` is misspelled** (per README; `run.sh`
references it). Cosmetic.
