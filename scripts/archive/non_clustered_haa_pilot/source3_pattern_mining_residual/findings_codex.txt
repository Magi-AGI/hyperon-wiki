S3.Q8 - pattern-index paradigm
------------------------------

pattern-index is real and adjacent, but not a fourth Hyperon-era pattern-mining tradition.

Evidence:
- HEAD a72459eb11f3cccd7c3f93e427115d400e31a9d9.
- README.md:1 names it "Pattern Index - an index for fast lookup of patterns in the AtomSpace".
- README.md:7 says it is not a wrapper around PatternMatcher.
- README.md:28-35 says it has additional pattern-mining algorithms, follows the OpenCog wiki Pattern_miner methodology, and is not a wrapper around PatternMiner.
- README.md:39-44 frames disk-based index structure and huge SCM-file lookup without an AtomSpace in memory.
- README.md:54-55 lists C++ and Scheme query/mining examples.
- README.md:272-307 documents mining requests, including patternindex().minePatterns and pi-mine-patterns.
- PatternIndexAPI.h:55-61 describes creating/customizing/querying/mining indexes; PatternIndexAPI.h:182-208 exposes minePatterns; PatternIndexAPI.cc:507-525 implements minePatterns wrappers; TypeFrameIndex.cc:581-745 contains the mining routine.
- tests/pattern-index/PatternIndexAPIUTest.cxxtest:170-179 tests mining.

Fork state: singnet/pattern-index is 1 ahead / 3 behind opencog/pattern-index in local remotes; the origin-only commit is a singnet merge PR. It is stale legacy OpenCog C++/Scheme infrastructure.

Verdict: [PATTERN-INDEX-SEPARATE-PARADIGM-ADJACENT]. It should be mentioned in HAA Pattern Mining or Reference+GitHub Repositories as adjacent legacy AtomSpace indexing/mining infrastructure, but it should not be collapsed into the symbolic hyperon-miner tradition or the neural GNN tradition. A future curated card such as Knowledge Representations+Pattern Index would be reasonable if the user wants durable documentation.
