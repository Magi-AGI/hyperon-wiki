Overall verdict
---------------

MetaMo is [PAPER-LEANING-HYBRID] with an [IMPLEMENTATION-BACKED-CORE] in MetaMo-Python. The two papers define the formal framework; MetaMo-Python implements a compact reference skeleton for X = G x M, Psi, D, F = D o Psi, lax-law checking as bounded numerical distortion, homeostatic damping, and incremental blending. It does not implement proof-grade category-theory laws, enriched category machinery, tubular topology, AtomSpace/MeTTa embedding, or production Hyperon integration. MetaMo-Prototype is a separate [HEURISTIC-PROTOTYPE], and hyperon-openpsi is the [OPENPSI-PREDECESSOR-SUBSTRATE], not the primary MetaMo implementation at HEAD.

S2.Q1 - pseudo-bimonad formalism: paper vs implementation
---------------------------------------------------------

Paper side: verified. Paper-1 defines X = G x M, the appraisal comonad Psi, decision monad D, lax distributive law, and composite F = D o Psi in section 2; the archived JSON stores the full paper text at AGI_25_METAMO_One.json:7. Paper-2 repeats the operational equations for Hyperon/OpenPsi/MAGUS in sections 2-3; full text is at MetaMo_Two.json:7.

Implementation side: partially verified in MetaMo-Python. README.md:3 identifies it as a Python implementation of MetaMo; README.md:18-20 names Appraisal Comonad, Decision Monad, and Pseudo-Bimonad F = D o Psi; README.md:84-88 documents G and M vector dimensions. core/state.py:8-14 defines MotivationalState with G and M; core/config.py:1-7 fixes NUM_GOALS=8 and NUM_MODULATORS=6. category/functors.py:7-29 defines AppraisalComonad, category/functors.py:32-55 defines DecisionMonad, and category/functors.py:57 defines TranslationFunctor. category/bimonad.py:12-23 defines MetaMoPseudoBimonad and executes F = D o Psi; category/bimonad.py:42-61 implements a lax-law check as distance bounded by LAX_DISTRIBUTIVE_DELTA; category/bimonad.py:62 implements parallel_merge.

Limit: this is operational reference code, not a formal verification of monad/comonad laws. The code checks numeric distortion, not categorical coherence. Verdict: [IMPLEMENTATION-BACKED-CORE] plus [FORMAL-LAWS-PAPER-ONLY].

S2.Q3 - MetaMo-Python vs MetaMo-Prototype vs hyperon-openpsi
------------------------------------------------------------

MetaMo-Python is the primary MetaMo reference implementation. Its README names both AGI-25 papers (README.md:9-10), lists the theory components (README.md:14-20), and maps code directories to openpsi/appraisal.py and magus/decision.py (README.md:54-58). applications/research_assistant.py:4-11 imports TranslationFunctor, OpenPsiAppraisal, MagusDecision, MetaMoPseudoBimonad, blend_states, and stability damping; applications/research_assistant.py:42-58 constructs the bimonad, assistant, and translator; applications/research_assistant.py:76-107 runs parallel merges, final action selection, damping, and blending. It also has an LLM layer using google-genai (requirements.txt:1-3; llm/client.py:2-22; llm/conversation.py:7-31).

MetaMo-Prototype is a separate heuristic engine, not a duplicate of MetaMo-Python. README.md:1 is just a title. engine.py:18-178 implements a dictionary-state step pipeline, with goals/modulators read at engine.py:19-21, appraisal at engine.py:34, scoring at engine.py:100, routing at engine.py:145, action selection at engine.py:169, and return at engine.py:178. post_update is engine.py:225-297. engine_state.py:67-89 defines persistent goals, anti-goals, and modulators; engine_scoring.py:208-446 contains action-scoring heuristics. It has LangGraph/LLM plumbing (graph.py:10-12, 43-45, 112, 122, 286, 297-353; parser.py:15-16, 21, 230-237). There are no bimonad/comonad/MAGUS/OpenPsi formal abstractions in this repo; a targeted search for those terms found only runner.py prompt text and README title-level MetaMo references.

hyperon-openpsi is the operational OpenPsi predecessor/substrate. A targeted search at HEAD for "MetaMo", "metamo", "MAGUS", "magus", "individuation", and "transcendence" returned zero matches. The repo does implement OpenPsi-style modulators: main/modulator/modulator.metta:6-12 defines arousal, positive-valence, selection-threshold, securing-threshold, and resolution-level; main/modulator/modulator.metta:15-22 defines fetch-modulator/fetch-modulator-val; main/mind-agents/modulator-updater/modulator-updaters.metta:33-38 updates arousal/securing/selection; main/mind-agents/feeling-updater/feeling-updaters.metta:145-153 reads arousal, securing, selection, and resolution. Recent git history is OpenPsi-side work, e.g. commit 01936f1 on 2026-02-24 aligns formulas with the OpenPsi paper, not MetaMo.

Conclusion for wiki contradiction: +content card 7334 should not present hyperon-openpsi as THE MetaMo implementation. Coherent framing is: MetaMo-Python = primary compact MetaMo reference; MetaMo-Prototype = separate heuristic prototype; hyperon-openpsi = predecessor OpenPsi codebase whose appraisal/modulator machinery paper-2 abstracts as Psi.

S2.Q4 - MetaMo extends OpenPsi boundary
---------------------------------------

MetaMo extends/subsumes OpenPsi at the formal-design layer; it does not supersede the closed OpenPsi cluster pilot's implementation finding. Paper-1 says MetaMo sits one level higher than OpenPsi and that OpenPsi's procedures/parameters can be slotted into MetaMo (AGI_25_METAMO_One.json:7, section 1.1). Paper-2 explicitly maps OpenPsi to the comonadic appraisal layer and MAGUS to the decision layer (MetaMo_Two.json:7, sections 2-3).

The code supports the boundary distinction. MetaMo-Python contains OpenPsiAppraisal as a Python class (openpsi/appraisal.py:17) that maps novelty/risk/conduciveness into six-vector modulator updates (openpsi/appraisal.py:39-75). hyperon-openpsi at 3b356c5 has OpenPsi modulators/mind-agent style code but no MetaMo/MAGUS naming. Therefore the OpenPsi pilot V4-1 remains valid: MetaMo reframes the OpenPsi operational machinery as a Psi comonad; it is not evidence that hyperon-openpsi became a strict literal implementation of the MetaMo formalism.

S2.Q6 - five design principles
------------------------------

All five principles are present in paper-1 (AGI_25_METAMO_One.json:7, sections 3-5). Code backing in MetaMo-Python:

1. Modular Appraisal-Decision Interface: AppraisalComonad and DecisionMonad are separate ABCs (category/functors.py:7-55); MetaMoPseudoBimonad composes them in category/bimonad.py:21-39.
2. Reciprocal Motivational State Simulation: TranslationFunctor exists at category/functors.py:57; applications/research_assistant.py:58 creates an identity translator as a minimal demo.
3. Motivational Compositionality: parallel_merge is implemented at category/bimonad.py:62-81 and exercised in applications/research_assistant.py:76 and :93.
4. Homeostatic Motivation Stability: config constants appear in core/config.py:16-26; safe-region and contraction checks are dynamics/stability.py:18-81; homeostatic damping is dynamics/stability.py:84-88.
5. Incremental Objective Embodiment: blend-factor and blend-state logic are dynamics/coherence.py:10-35; the research assistant applies blend_states at applications/research_assistant.py:106-107.

Limit: MetaMo-Python has no checked-in test files by `rg --files -g "*test*" -g "*tests*" -g "*spec*" -g "!.git/**"`; AGENTS.md:23 also warns there is no formal test suite or package metadata. The five principles are implemented as demo/reference logic, not validated system behavior.

S2.Q11 - MetaMo <-> MAGUS boundary
----------------------------------

Paper-2 explicitly identifies MAGUS as the decision side of the composition: OpenPsi is the appraisal comonad, MAGUS is the hierarchical decision/goal system, and their composition forms F = D o Psi (MetaMo_Two.json:7, sections 2-3 and conclusion). The same paper attributes MAGUS to Mikeda 2024 in reference [8] (MetaMo_Two.json:7). MetaMo-Python has magus/decision.py, but this is a compact class with scoring and goal updates, not the Magi Archive's MAGUS framework.

Verdict: [MAGUS-BOUNDARY-EXTERNAL-COMPONENT]. Wiki text may say MetaMo uses/instantiates MAGUS as D, but should not imply MAGUS was created by Goertzel/Lian in the MetaMo papers or that MetaMo-Python implements the full MAGUS system.
