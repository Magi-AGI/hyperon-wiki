S3.Q1. METTATRON EVALUATOR STRATEGY
----------------------------------------------------------------

VERDICT: [METTATRON-LAZY-NONDETERMINISTIC-CARTESIAN]

Evidence (src/backend/eval/mod.rs):

  - The public entry point is `pub fn eval(value, env)` at
    mod.rs:169-172, which delegates to `eval_trampoline` at
    mod.rs:177-390.

  - `eval_trampoline` is an **iterative-with-explicit-stack**
    evaluator: it uses `WorkItem` enum (mod.rs:45-55) +
    `Continuation` enum (mod.rs:59-88) on a `work_stack: Vec<
    WorkItem>` (mod.rs:179) instead of recursive function calls.
    The doc-comment at mod.rs:174-176 explicitly frames this as
    avoiding stack overflow via "heap-allocated work items".

  - Bounded-depth guard: `const MAX_EVAL_DEPTH: usize = 1000`
    at mod.rs:93. Check at mod.rs:425-444 returns an Error if
    exceeded.

  - Cartesian-product semantics is the explicit name and
    implementation:

      * `const MAX_CARTESIAN_RESULTS: usize = 10000` at mod.rs:97.
      * `fn cartesian_product(results: &[Vec<MettaValue>]) ->
        Result<Vec<Vec<MettaValue>>, MettaValue>` at mod.rs:821-
        890. Computes ALL combinations of sub-expression result
        sets; returns an Error if total_size > MAX_CARTESIAN_
        RESULTS (mod.rs:840-854).
      * Invoked at mod.rs:587-596 inside `process_collected_
        sexpr` between sub-expression evaluation and rule
        matching.
      * FAST PATH at mod.rs:827-837: if every sub-expression
        result list has length 1 (deterministic), bypass the
        product construction and concatenate directly. This
        keeps deterministic arithmetic and most builtins on the
        O(n) path.

  - Lazy: the README claim "Expressions evaluated only when
    needed" is supported by the trampoline design (work items
    are pushed, not eagerly executed) and by the SExpr-collection
    Continuation pattern (`Continuation::CollectSExpr` at mod.rs:
    63-74) which defers parent processing until all children
    yield. However, MeTTaTron does NOT implement lazy
    binding/closures in the call-by-need sense — once a work
    item is popped, the expression is evaluated fully. The
    "lazy" framing is closer to "iterative deferred evaluation"
    than to Haskell-style thunks.

Boundary against S1 hyperon-experimental:

  - S1 verdict at v0.2.10 was `[SMALL-STEP-INTERPRETER]` +
    `[NONDETERMINISTIC-BRANCHING]`. MeTTaTron diverges on the
    SECOND axis: it is explicitly Cartesian-product (mod.rs:586-
    596 generates the full combinatorial product) where S1's
    interpreter does per-branch evaluation without materialising
    the full product up front. The semantic effect is similar
    in the common case but the runtime cost shape differs —
    MeTTaTron will fail-fast at 10,000 combinations whereas S1
    runs branches lazily.

  - On the small-step axis, MeTTaTron's `eval_step` (mod.rs:421-
    468) takes a single step and returns either `Done(result)`
    or `EvalSExpr { items, env, depth }`, then control returns
    to the trampoline. So MeTTaTron is also small-step in
    spirit, but the small-step boundary is implemented via a
    heap-stack rather than via the Rust call stack.

Boundary against S2 PeTTa / metta-morph:

  - C2-1 PeTTa "Prolog translation with smart dispatch and
    backtracking" — DIFFERENT paradigm.
  - C2-4 metta-morph Scheme-side `bind!` reset semantics — N/A
    (MeTTaTron does not implement `bind!` at HEAD; see Q-12).

S3.Q2. METTATRON COMPILATION VS INTERPRETATION
----------------------------------------------------------------

VERDICT: [TREE-WALK-INTERPRETER]

Justification: the name "MeTTa-Compiler" is misleading at HEAD
`efee4be`. The `compile` function is a parser+pre-processor, not
a code generator.

Evidence (src/backend/compile.rs):

  - `pub fn compile(src: &str) -> Result<MettaState, SyntaxError>`
    at compile.rs:18-59. Three-step pipeline:
      1. `TreeSitterMettaParser::new()` (compile.rs:26-31) +
         `parser.parse(src)` (compile.rs:33-41) → `Vec<SExpr>`.
      2. `sexprs.iter().map(MettaValue::try_from).collect()`
         (compile.rs:43-56) → `Vec<MettaValue>`.
      3. `MettaState::new_compiled(metta_values)` (compile.rs:58)
         → returns a state with `source` (parsed expressions) and
         empty `environment`.

  - At no point is there code-generation, JIT, AOT, or IR
    transformation beyond SExpr→MettaValue normalization. The
    doc-comment at compile.rs:1-8 confirms: "compile function
    parses MeTTa source code and produces a PathMap structure
    containing [parsed_sexprs, fact_db]".

  - Interpretation happens via the tree-walking trampoline in
    `eval` (mod.rs:169) which traverses the MettaValue tree —
    not compiled bytecode.

  - The `compile` test corpus (compile.rs:60-381) is entirely
    parser/AST-shape tests (test_compile_empty_input,
    test_compile_simple, test_compile_multiple_expressions,
    test_compile_operators, test_compile_gt, ...) — no
    code-generation tests.

Caveat — feature branches:

  - `origin/feature/wam-backend` (last commit 2026-03-15)
    exists, suggesting a Warren-Abstract-Machine compilation
    path is in development. The `origin/dylon/metta-compiler-
    optimizations` branch also suggests compilation work. **At
    HEAD `efee4be` on main, no WAM or compilation backend is
    integrated.**

  - `Cargo.toml:152-153` registers a `[[bench]] set_operations`
    entry and a default feature `interning` — neither
    constitutes compilation. The `[[bench]] expression_
    parallelism` entry is an interpreter-throughput bench.

Boundary against S1 / S2:

  - S1 hyperon-experimental: `[SMALL-STEP-INTERPRETER]` — same
    family as MeTTaTron at HEAD; S1 has no compiler either.
  - S2 PeTTa: `[HYBRID-COMPILED-INTERPRETED]` — DIFFERENT
    paradigm (PeTTa translates to Prolog clauses + uses
    Prolog runtime fallback). MeTTaTron does NOT translate to
    any backend language.
  - S2 metta-morph: `[HYBRID-MACRO-AND-RUNTIME]` — DIFFERENT
    paradigm (Chicken-Scheme macro layer + runtime bridge).
    MeTTaTron does NOT use macros for translation.

S3.Q3. METTATRON TYPE SYSTEM
----------------------------------------------------------------

VERDICT: [TYPES-DYNAMIC-WILDCARD-COMPATIBLE]

Evidence (src/backend/eval/types.rs):

  - Three runtime ops registered as special forms at eval/
    mod.rs:523-525:
      * `:` → `eval_type_assertion` at types.rs:47-75
      * `get-type` → `eval_get_type` at types.rs:79-86
      * `check-type` → `eval_check_type` at types.rs:90-101

  - `infer_type(expr, env)` at types.rs:105-197 dispatches on
    MettaValue variant:
      * Ground types return `Atom("Bool"/Number/String/Nil/
        Type/Error")` (types.rs:108-118).
      * Atoms starting with `$`, `&`, or `'` are wrapped as
        `MettaValue::Type(Arc::new(Atom(name)))` (types.rs:121-
        126) — type variables.
      * Non-variable atoms look up the environment's typing-
        table (`env.get_type(name)`, types.rs:129) and return
        `Atom("Undefined")` on miss (types.rs:138).
      * SExprs match arithmetic operators to `Number` (types.rs:
        153-155), comparison operators to `Bool` (types.rs:156-
        158), `->` to `Type` (types.rs:159-162), and otherwise
        try to extract a return type from an arrow-type entry
        in the environment (types.rs:166-175).

  - Wildcard handling: MeTTaTron uses the bare token "Undefined"
    (types.rs:138, 182) rather than S1's `%Undefined%`
    convention. The tree-sitter grammar at tree_sitter_parser.rs:
    272 recognises a separate `special_type_symbol` kind for
    `%Undefined%`/`%Irreducible%`/etc. which round-trips as a
    plain Atom. So the wire-format is compatible (`%Undefined%`
    parses) but the runtime emits "Undefined" not "%Undefined%".

  - Arrow types supported via `->` constructor (types.rs:159-
    162); arrow-type return-extraction at types.rs:166-175.

  - "Did you mean?" suggestion table at types.rs:7-26 maps
    common variants ("bool"→"Bool", "int"→"Long", ...).

Static vs dynamic: types.rs runs at evaluation time — the
TYPE-COMPATIBLE-WITH-S1 framing holds because dispatch is
identical pattern-on-MettaValue. There is no compile-time type
check pass.

Boundary against S1:

  - S1 v0.2.10 verdict was `[DYNAMIC-CHECKED]` with `%Undefined%`
    wildcard semantics. MeTTaTron's types are also dynamic; the
    wildcard token name differs ("Undefined" vs "%Undefined%")
    but the grammar accepts both. So
    [TYPES-DYNAMIC-WILDCARD-COMPATIBLE] is correct — wire
    compatible with S1, with a minor runtime-token convention
    drift.

S3.Q10. ASYNC + THREADING MODEL
----------------------------------------------------------------

VERDICT: [ASYNC-COARSE-GRAINED]

Per V2-PROTOCOL-1 specificity, [ASYNC-COARSE-GRAINED] is the
correct tag. The parallelism granularity at HEAD is batch-of-
top-level-eval-expressions, NOT per-expression or per-rule.

Evidence:

  - Async-feature gating:
      * Cargo.toml:112-115:
          [features]
          default = ["interning", "async"]
          async = ["tokio"]
          interning = []
      * `tokio` is OPTIONAL behind the `async` feature flag.
        Default builds include it; one CAN opt out and the
        sync path remains functional.

  - Core eval (eval/mod.rs:169-390) is SYNCHRONOUS. No async
    fn, no .await calls inside the trampoline. The work_stack
    runs on a single thread.

  - Parallelism lives at the RHOLANG INTEGRATION layer, NOT
    the core evaluator:
      * `pub async fn run_state_async` at rholang_integration.
        rs:267-365. Gated by `#[cfg(feature = "async")]` at
        rholang_integration.rs:266.
      * Batching strategy (rholang_integration.rs:313-347):
        - Walk top-level expressions in order.
        - Accumulate non-(rule-def, ground-fact) expressions
          into `current_batch`.
        - When a rule-def or ground-fact is encountered, FLUSH
          the current batch via `evaluate_batch_parallel`
          (which runs all batched evals concurrently), THEN
          process the rule-def/ground-fact sequentially.
        - Repeat until source is exhausted.
      * `evaluate_batch_parallel` at rholang_integration.rs:
        370-419 uses `tokio::task::spawn_blocking` (NOT plain
        `tokio::spawn`) so each batched-eval runs on the
        Tokio blocking pool — appropriate for the CPU-bound
        synchronous trampoline.
      * Results are collected via `.await` then sorted by
        original index (rholang_integration.rs:410) to
        preserve output order.

  - Granularity:
      * NOT per-rule: rules in `match` / general dispatch are
        evaluated sequentially within a single `eval` call.
      * NOT per-pattern: pattern matching is intra-thread.
      * NOT per-sub-expression: sub-expressions of a single
        SExpr are evaluated sequentially in the trampoline.
      * COARSE-GRAINED at the BATCH-of-top-level-eval-
        expressions level only.

  - Determinism / order:
      * Rule defs and ground facts are FORCED SEQUENTIAL (per
        rholang_integration.rs:323-345) because they mutate
        the environment.
      * Within a batch, evaluations are independent (env is
        cloned at rholang_integration.rs:385) so order does
        not affect results. Outputs are re-sorted to original
        index for stable display.

  - Configurability: per the README claim P-4 ("True
    parallelization with configurable threading via Tokio"),
    threading IS configurable — but only via Tokio's
    `Runtime::Builder` at the embedding-application level
    (i.e., the Rholang runtime configures the blocking-pool
    size). MeTTaTron itself does NOT expose a per-process
    config knob for parallelism level; it inherits whatever
    Tokio runtime the caller sets up. The
    `[dev-dependencies]` tokio block at Cargo.toml:123-129
    requests `["rt-multi-thread", "process", "time", "sync",
    "macros"]` for tests; the runtime-dep at Cargo.toml:73
    requests only `["rt", "sync"]`, suggesting MeTTaTron
    doesn't bring its own multi-thread runtime by default.

  - benches/expression_parallelism.rs (registered at
    Cargo.toml:51) is the dedicated parallelism benchmark;
    benches/e2e/e2e_throughput.rs + e2e_throughput_
    configurable.rs (Cargo.toml:141-149) cover throughput.

Verdict justification:

  - [ASYNC-PER-EXPRESSION-PARALLEL] — REJECTED. Within a
    single expression's evaluation, the trampoline is single-
    threaded.
  - [ASYNC-PER-RULE-PARALLEL] — REJECTED. Rules within a
    match are evaluated sequentially.
  - [ASYNC-PER-PATTERN-PARALLEL] — REJECTED. Pattern matching
    is intra-thread.
  - [ASYNC-COARSE-GRAINED] — CORRECT. Parallelism granularity
    is batch-of-top-level-eval-expressions between rule-def
    barriers.

README claim P-4 ("True parallelization with configurable
threading via Tokio") is TRUE in a narrow sense: the
parallelism IS real (tokio::task::spawn_blocking on multiple
expressions concurrently) and IS configurable (via the
embedding runtime). The "True parallelization" wording may
mislead readers into expecting per-rule or per-expression
granularity, which is NOT what's implemented. Recommend a wiki
note that the parallelism is batch-level if a Full card is
created.
