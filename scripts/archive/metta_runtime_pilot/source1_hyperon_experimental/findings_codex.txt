Q1. Interpreter semantics
-------------------------

Verdict: [RUST-MINIMAL-INTERPRETER] [NONDETERMINISTIC-PLAN-SEMANTICS]
[QUERY-DRIVEN-REWRITE-EVALUATOR]

The active interpreter is not a thin wrapper around another runtime. The core
evaluation loop is Rust-native and plan/stack based:

- `lib/src/metta/interpreter.rs:181` stores `max_stack_depth`; `:254`
  defines `interpret_init`; `:269` defines `interpret_step`; `:285`
  defines `interpret`.
- `lib/src/metta/interpreter.rs:374` dispatches `interpret_stack`; `:392`
  enforces stack depth.
- `lib/src/metta/interpreter.rs:504` defines `eval_impl`; `:559` defines
  `eval_result`; `:604` defines `query`.
- `lib/src/metta/interpreter.rs:687`, `:746`, `:809`, and `:893` implement
  `chain`, `collapse-bind`, `unify`, and `superpose-bind` behavior.
- `lib/src/metta/interpreter.rs:922` handles native grounded symbol calls.

The evaluation model is explicitly nondeterministic. `docs/minimal-metta.md`
describes an interpretation plan that may branch and says the interpreter does
not select one branch. `python/tests/scripts/b4_nondeterm.metta` asserts that
multiple equality definitions and `superpose` produce multiple results.

Important boundary: this is not a deterministic "first match wins" rewrite
engine. If both a specific equation and a catchall equation match, both can
contribute reductions unless the program makes them mutually exclusive.

Verdict: [DYNAMIC-CHECKED-TYPES] [DEPENDENT-TYPE-EXPRESSIVE-BUT-NOT-PROVER]
[UNDEFINED-TYPE-WILDCARD]

dynamic checker with permissive undefined/meta-type behavior, not a static

Q11. v0.2.x quirks re-check
---------------------------

Verdict: [V0210-QUIRKS-REFINED]

Three remembered quirks were checked against HEAD v0.2.10:

1. `bind!` + `new-space` is not a reset primitive.

Evidence:

- `lib/src/metta/runner/stdlib/module.rs:246-279` implements `BindOp` by
  registering a token in the tokenizer.
- `lib/src/metta/runner/stdlib/space.rs:13-30` implements `new-space` by
  creating a new `GroundingSpace`.
- No code path in `BindOp` clears or garbage-collects atoms in an already
  bound space. The safe operational guidance remains: do not treat
  `!(bind! &name (new-space))` as a reliable reset of an existing live space.

2. `cond` is not a builtin at HEAD.

Evidence:

- `rg -n "\bcond\b|\(cond\b|cond\(" lib/src/metta python/tests docs`
  found no stdlib/builtin `cond` implementation. Hits were local variable
  names in docs/examples, not a callable form.
- `git log -S "(cond"` returned no source history for a `cond` form.
- `stdlib.metta` provides `if` at `:511-513` and `case` at `:1224-1233`.

So the current wording should be updated from "cond is broken" to "no builtin
cond exists at v0.2.10; use `if`, `case`, `match`, or user-defined equations."

3. Catchall equations alongside specific equations still produce duplicate or
   multiple reductions when both match.

Evidence:

- `docs/minimal-metta.md` describes nondeterministic branching semantics.
- `lib/src/metta/interpreter.rs:604` queries all matching equality forms.
- `python/tests/scripts/b4_nondeterm.metta` explicitly tests multi-result
  behavior.

This is semantic, not a stale bug report.

Overall Source 1 verdict
------------------------

[REFERENCE-RUST-METTA-RUNTIME] [HYBRID-RUST-PLUS-METTA-STDLIB]
[DYNAMIC-CHECKED-TYPES] [MODULE-SYSTEM-ACTIVE-EVOLVING]
[IN-MEMORY-ATOMTRIE-INDEX] [PYTHON-PARITY-BROAD-BUT-OBSERVER-GAP]
[C-API-BROAD-LOWLEVEL] [SANDBOX-PROTOTYPES-NOT-PACKAGED]

hyperon-experimental at v0.2.10 is the right baseline for the MeTTa runtime
cluster. It is not just a parser or a Python wrapper: it contains the Rust
interpreter, dynamic type system, stdlib, module loader, Space model, C API,
Python API, builtin modules, tests, and sandbox precedents.

The cluster narrative should not overstate it as a finished production
runtime. The module system remains evolving, type checking is dynamic and
permissive around `%Undefined%`, sandbox integrations are prototype-level, and
the Python observer binding gap remains live.
