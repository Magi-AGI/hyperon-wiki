Q2. SPACEEVENT / SPACEOBSERVER
--------------------------------------------------------------------------------

VERDICT: [OBSERVER-PATTERN-CONFIRMED] [SYNCHRONOUS-IN-PROCESS]
[SINGLE-THREADED-LIMITATION].

Evidence:
- `SpaceEvent` has Add/Remove/Replace variants at `hyperon-space/src/lib.rs:20-29`.
- `SpaceObserver::notify` is at :63-66.
- `SpaceCommon` stores observers in `RefCell<Vec<Weak<RefCell<dyn SpaceObserver>>>>`
  at :96-100; registration is :106-109; notification iterates synchronously and
  calls `observer.borrow_mut().notify(event)` at :113-125.
- Observer lists are intentionally not cloned with `SpaceCommon` at :128-135.
- The implementation uses `Rc`/`RefCell`, not `Arc`/`Mutex`, in core `DynSpace`
  (`hyperon-space/src/lib.rs:281-297`). Multi-threaded sharing is not supported
  by this core wrapper.
- Rust tests exercise event collection in `lib/src/space/grounding/mod.rs:254-281`
  and subsequent add/remove/replace tests. C tests exercise observer registration
  and event-based atom counts in `c/tests/check_space.c:127-207`.

Observer execution is in the writer's stack frame. A slow or panicking observer
will directly affect the writer unless Phase 4 adds an async/event-bus boundary.

Q3. PERSISTENCE / DURABILITY
--------------------------------------------------------------------------------

VERDICT: [NO-NATIVE-RUST-PERSISTENCE-TRAIT] [EXTENSIBLE-VIA-C-API]
[PYTHON-CUSTOM-SPACE-WRITABLE].

Evidence:
- Native `GroundingSpace` is in-memory: it stores `AtomIndex` plus `SpaceCommon`
  (`lib/src/space/grounding/mod.rs:14-21`) and constructs empty/from-vec spaces
  at :23-40. No persistence/durability hook appears in `Space`/`SpaceMut`.
- The C API allows custom storage backends: `space_new` wraps a caller-supplied
  `space_api_t` and payload at `c/src/space.rs:64-72`; the function table includes
  query/subst/add/remove/replace/count/iterate/free hooks at :508-607.
- `CSpace` forwards `SpaceMut` calls into the caller's function table at
  `c/src/space.rs:783-797`.
- Python custom spaces use that C hook: `PY_SPACE_NO_SUBST_API` at
  `python/hyperonpy.cpp:387-397`, `space_new_custom` at :938-940, and Python
  `AbstractSpace` defines add/remove/replace at `python/hyperon/base.py:26-42`.
- The sandbox SQL space is evidence that a Python custom space can persist to
  Postgres (`python/sandbox/sql_space/sql_space.py:131-181`), but it is sandbox
  code, query-string based, and not a native durable Hyperon Space trait.

Q4. CONCURRENCY
--------------------------------------------------------------------------------

VERDICT: [INTER-PROCESS-SYNC-REQUIRED] [NO-BUILT-IN-MULTI-PROCESS-MIRROR].

Evidence:
- Core `DynSpace` is `Rc<RefCell<dyn SpaceMut>>` (`hyperon-space/src/lib.rs:281-297`).
- The C API explicitly warns that `space_get_payload` is likely to change for
  multi-threading and that payload must not be modified while other operations
  access the space (`c/src/space.rs:110-119`).
- DAS code uses `Arc<Mutex<Properties>>` for DAS parameters
  (`lib/src/metta/runner/builtin_mods/das.rs:1-6,64`), but that does not make
  ordinary `GroundingSpace`/`DynSpace` thread-safe.

For Phase 4, multiple Decko/Rails worker processes cannot share an in-memory
`GroundingSpace`. A central mirror service or explicit shared event bus is
required.

--------------------------------------------------------------------------------
Q5. SPACE TAXONOMY
--------------------------------------------------------------------------------

VERDICT: [LAYER-2-WRITE-CAPABLE-MIRROR] [NOT-DURABLE-BACKEND].

Taxonomy:
- `Space` is read/query/visit (`hyperon-space/src/lib.rs:151-216`).
- `SpaceMut` adds add/remove/replace (`hyperon-space/src/lib.rs:218-279`).
- `DynSpace` is type-erased shared in-process wrapper (`hyperon-space/src/lib.rs:281-338`).
- `GroundingSpace` is the canonical in-memory implementation
  (`lib/src/space/grounding/mod.rs:14-21,181-211`).
- `ModuleSpace` delegates mutations to its main `DynSpace`, not deps
  (`lib/src/space/module.rs:7-10,82-91`).
- `CSpace` is the custom backend adapter (`c/src/space.rs:688-808`).

This confirms the AtomSpace cluster pilot's "Layer 2" classification while
refining it: Layer 2 is not durable, but it is write-capable and observer-capable.

Q7. ATOM IDENTITY UNDER MUTATION
--------------------------------------------------------------------------------

VERDICT: [STRUCTURAL-IDENTITY] [REPLACE-IS-REMOVE-THEN-ADD]
[EVENT-CARRIES-OLD-AND-NEW].

Evidence:
- `GroundingSpace::replace` removes `from`, inserts `to`, and emits
  `SpaceEvent::Replace(from.clone(), to)` only on success
  (`lib/src/space/grounding/mod.rs:119-126`).
- `AtomIndex::remove` queries structurally and removes a matching trie key
  (`hyperon-space/src/index/mod.rs:206-211`).
- No UUID/stable durable atom id is exposed in `SpaceMut`.

This is acceptable for a mirror if Decko identity is encoded in the Atom content
or metadata convention. It is not a built-in L1 stable-id mechanism.

Q8. PYTHON BINDINGS
--------------------------------------------------------------------------------

VERDICT: [WRITE-API-EXPOSED] [OBSERVER-API-MISSING-IN-PYTHON]
[PYTHON-CUSTOM-SPACES-AUTO-NOTIFY-C-OBSERVERS].

Evidence:
- Python exports native space writes at `python/hyperonpy.cpp:946-948`.
- `SpaceRef.add_atom/remove_atom/replace_atom` call those bindings at
  `python/hyperon/base.py:205-221`.
- Python custom spaces are mutable through `AbstractSpace.add/remove/replace`
  (`python/hyperon/base.py:26-42`) and `SpaceRef(space_obj)` creates a C-backed
  custom space at :174-183.
- Python custom-space mutations emit C observer events automatically:
  `py_space_add` notifies at `python/hyperonpy.cpp:422-435`; remove at :437-454;
  replace at :456-476.
- But `python/hyperonpy.cpp` exposes no `space_register_observer` binding, and
  `python/hyperon/base.py` has no observer registration wrapper. I also checked
  `origin/add_python_oper_ext`, `origin/pyrust`, and `origin/artur-das-v102`;
  observer support remains C/Rust-only there.

Phase 4 implication: a Python mirror service can write atoms today, but Python
cannot subscribe to SpaceEvent without adding bindings or using a Rust/C event
forwarder.

Q12. DECKO-CALLBACK INTEGRATION TARGET
--------------------------------------------------------------------------------

VERDICT: [OPTION-D-CONFIRMED-WITH-EVENT-FORWARDER-GAP].

Recommended Source 2-informed wiring:
1. Decko post-commit callback emits a durable event (Redis/Sidekiq or equivalent).
2. A single mirror service consumes Decko events and calls Hyperon writes
   (`SpaceRef.add_atom/replace_atom/remove_atom` if Python; direct Rust/C if not).
3. The mirror service owns the in-memory `DynSpace` to avoid multi-process
   `Rc<RefCell>` sharing problems.
4. L1 agent writes go through the same service boundary, not directly through
   multiple Rails workers.
5. Event fanout to agents/Decko write-back needs a C/Rust `SpaceObserver` bridge
   or new Python observer bindings.
6. Decko identity/version/revision semantics must be encoded at Source 5; Space
   identity is structural and in-memory.

Option D is confirmed as the primary architecture. Option C (extending
atomspace-bridge) is not required by Source 2. The main unresolved implementation
gap is observer exposure / event-forwarding, not write support.

--------------------------------------------------------------------------------
BOTTOM LINE
--------------------------------------------------------------------------------

Source 2 confirms that Hyperon Space is a viable write-capable in-memory mirror
substrate for Phase 4 Option D. It has real Rust/C/Python write calls and a real
Rust/C observer model. It does not provide durable persistence, stable atom IDs,
multi-process sharing, or Python observer subscription. Therefore Phase 4 should
continue with Option D, but SPEC_DRAFT must include a mirror-service boundary,
an event-forwarder/Python-observer-binding task, and Source 5 Decko identity /
revision semantics before any L1 write-back loop is considered complete.
