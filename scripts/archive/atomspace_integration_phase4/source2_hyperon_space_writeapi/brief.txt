1. CONTEXT
--------------------------------------------------------------------------------

Phase 4 (B)+(L1) co-canonical AtomSpace+Postgres write-through with agent-
writable atoms. Source 1 (atomspace-bridge) closed with consensus: bridge is
[READ-ONLY-BY-DESIGN] [BULK-IMPORT-THEN-READ], NOT viable as Phase 4 write
substrate. Recommendation: Option D (Decko-callback-driven AtomSpace mirror).

For Option D to work, the AtomSpace mirror must:
  - Accept atom writes from Decko-side change-event consumers (post-commit).
  - Accept atom writes from agents (the L1 self-improvement loop).
  - Notify observers (agents subscribed to changes; Decko callbacks for
    write-back to durable Postgres).
  - Provide read-side query/match for agent reasoning.
  - Handle reasonable concurrency without serializing the entire mirror.

The cluster-pilot Source 2 R2.B4 (closed 2026-04-29) classified
hyperon-experimental's Hyperon Space as Layer 2: "GroundingSpace / SpaceMut /
DynSpace; lib_spaces.metta — MeTTa-facing demos; not primary Decko backend."
Source 2 here verifies and EXTENDS that classification with a write-API
audit specifically scoped to Phase 4 needs.

5. RESEARCH QUESTIONS
--------------------------------------------------------------------------------

Q1. SPACE WRITE API (CORE GATING QUESTION).
    Audit hyperon-space/src/lib.rs and lib/src/space/grounding/mod.rs for
    the actual public write API. Answer:
      (a) What write methods exist on the public Space surface? Cite each
          method's signature (file:line).
      (b) Are write methods on a single Space type, or split across types
          (Space trait vs SpaceMut trait vs GroundingSpace concrete vs
          DynSpace dyn-dispatch)?
      (c) Are there abstract Space traits (read-only) and SpaceMut sub-
          traits (mutable)? If so, which Spaces implement which?
      (d) For Phase 4 Option D (Decko callbacks → mirror), which write
          method is the right entry point?

Q2. SPACEEVENT + SPACEOBSERVER PATTERN.
    Per orchestrator pre-read, hyperon-space/src/lib.rs declares SpaceEvent
    {Add, Remove, Replace} + SpaceObserver trait with notify(). Verify and
    characterize:
      (a) Is observer registration explicit (register_observer call) or
          implicit (every observer auto-notified)?
      (b) Are observers per-Space or global?
      (c) What happens if an observer panics or returns an error from
          notify()?
      (d) Does notify() run synchronously (in the writer's stack frame)
          or queued/async?
      (e) Is the observer set thread-safe?
      (f) For Phase 4: can a Decko-side process register an observer that
          forwards SpaceEvents back to Decko's change-data-capture?

Q3. PERSISTENCE / DURABILITY MODEL.
    Hyperon-experimental's Space is in-memory. Phase 4 (B) keeps Postgres as
    durability. Question: how does Hyperon Space integrate with persistent
    backends?
      (a) Is there a PersistentSpace trait or hook? If yes, what's its API?
      (b) The `das` feature + new-das! op constructs a DAS-backed DynSpace
          (per cluster pilot). Audit the path from DynSpace → DAS calls.
          Where is the boundary?
      (c) Does Hyperon Space expose snapshot / dump / load primitives that
          Phase 4 could use for crash recovery + Decko-mirror reconciliation?

Q4. CONCURRENCY MODEL.
    Audit how Hyperon Space handles multi-threaded readers/writers:
      (a) RefCell-based (single-threaded interior mutability)? Per the
          pre-read at lib.rs:8 (use std::cell::RefCell), this looks like
          single-threaded.
      (b) Or Arc<Mutex>/Arc<RwLock> wrappers for thread-safe access?
      (c) Or is Send/Sync handled at a higher layer (DynSpace wrapper)?
      (d) For Phase 4 Option D: Decko's Rails workers are multi-process;
          inter-process communication to Hyperon Space requires either
          (i) a Hyperon Space process with IPC/HTTP boundary, or (ii) a
          per-worker Hyperon Space synced via the observer pattern.
          Which is the natural fit given the API?

Q5. DYNSPACE / SPACEMUT / GROUNDINGSPACE TAXONOMY.
    Cluster pilot R2.B4 named three: GroundingSpace, SpaceMut, DynSpace.
    Verify the taxonomy at HEAD:
      (a) GroundingSpace — concrete impl. Where? hyperon-space crate or
          lib/src/space/grounding/?
      (b) SpaceMut — trait or struct? Difference from base Space?
      (c) DynSpace — dyn-dispatch wrapper for Box<dyn Space>?
      (d) Are there other Space variants? (CompositeSpace, ProxySpace,
          remote Space wrappers?)
      (e) Phase 4 entry point: which type does a Decko-callback consumer
          most naturally instantiate?

Q6. PERFORMANCE / SCALE.
    The hyperon-space/benches/ directory is the source of truth. Audit:
      (a) What benchmarks exist? Read/write/query/match?
      (b) What atom counts are exercised?
      (c) What's the throughput ballpark? (Cite bench numbers if in repo;
          per Source 1 V2-Q10/14 lesson, do NOT extrapolate quantitative
          ceilings without benchmark evidence.)
      (d) Indexing strategy? hyperon-space/src/index/ subdirectory.
          Is there a single-key index, or pattern-matched indexing?
      (e) Memory footprint per atom (compare to MORK's ~64 bytes/atom
          per cluster pilot Source 3).

Q7. ATOM IDENTITY UNDER MUTATION.
    Source 1 Q11 found bridge has content-addressed atom identity, no PK
    flagging. Hyperon Space is also content-addressed (atoms are
    structurally hashed). Question:
      (a) Is `replace(old, new)` a true in-place identity-preserving
          mutation, or does it create a new atom and remove the old? If
          the latter, does the SpaceEvent::Replace observer get the OLD
          atom or just an Add+Remove pair?
      (b) Does Hyperon Space support primary-key-style stable identity
          atoms (e.g., a NodeId(N) + value-relation pattern)?
      (c) For Phase 4 (L1) round-trip: an agent reads Card 7432, modifies
          its content, writes back. What's the canonical mutation pattern
          in Hyperon Space terms?

Q8. PYTHON BINDINGS COVERAGE.
    Decko is Ruby; Phase 4 callbacks fire from Rails. Cross-language
    integration options:
      (a) Decko Rails worker → HTTP → Python service → hyperon Python →
          Hyperon Space.
      (b) Decko Rails worker → spawn child Python process per event.
      (c) Decko Rails worker → message queue → Python consumer → Hyperon
          Space.
    Audit python/hyperon/ to determine:
      (a) Is the Space write API exposed to Python? Does Python's DynSpace
          (or equivalent) support add/remove/replace?
      (b) Is the SpaceObserver pattern accessible from Python? If yes,
          can a Python-defined observer receive events from a Rust-side
          Hyperon Space?
      (c) What's the Python concurrency model? Are GIL/threading concerns
          relevant?
      (d) Is there an existing Python HTTP/RPC server pattern (e.g., FastAPI
          + hyperon)? (Cross-ref: Magi Archive `Notes+MAGUS Research+Event
          Driven Agent Architecture Analysis - Hyperon Experimental` ID 2454
          may have prior work.)

Q9. DAS MODULE INTEGRATION (recently-landed).
    Commits 97af6eb8 + 2f3cccff added das.metta + documentation. Audit:
      (a) What's the canonical DAS-from-MeTTa API (new-das! and surrounds)?
      (b) Does the das module itself expose write hooks back into the
          parent Hyperon Space?
      (c) Is the DAS feature default-on (per cluster pilot R2.B4) at HEAD?
          Verify via Cargo.toml.
      (d) For Phase 4: if we use Hyperon Space as the mirror but want
          eventual DAS deployment for distributed access, does the API
          generalize cleanly?

Q10. PR #852 EVENTAGENT (unmerged).
    The pr-852 branch is mentioned in the Magi Archive (ID 2447, "Hyperon
    EventAgent PR #852 Analysis - MAGUS Implementation"). Audit:
      (a) git log pr-852 ^main: what's added on the branch?
      (b) Does it add an event-driven agent loop pattern Phase 4 could
          model on?
      (c) Is the branch active or stale (last commit date)?
      (d) Cross-ref the Magi Archive analysis card for prior thinking.

Q11. CLUSTER-PILOT FINDING VERIFICATION (R2.B4 lib_spaces.metta drift).
    R2.B4 cited "lib_spaces.metta" as a key file. Glob at HEAD returns no
    matches. Resolve:
      (a) git log --all --oneline -S 'lib_spaces' to find historical mentions.
      (b) git log --all --diff-filter=D --name-only -- '*lib_spaces*' to
          find deletions.
      (c) Determine successor: is it das.metta? Another builtin_mods file?
          A renamed location?
      (d) If genuinely missing, this is a CF5.6 inheritance from the
          cluster pilot — flag as V-N-X for the reconciliation. Update the
          R2.B4 source-of-truth note in this Source 2's reconciliation.

Q12. DECKO-CALLBACK INTEGRATION TARGET.
    Synthesizing Q1–Q11: design the canonical Phase 4 Option D wiring:
      (a) Decko Rails after_save callback fires.
      (b) Rails worker enqueues a change event (bug/permission state /
          before-after card content).
      (c) Mirror-side process (in-process Python or out-of-process IPC?)
          consumes the event.
      (d) Mirror-side translates the event to Hyperon Space write calls
          (add/replace/remove).
      (e) Agent observers receive SpaceEvent notifications.
      (f) Agent writes back to Hyperon Space via add/replace/remove.
      (g) (How?) Agent writes propagate back to Decko via a Decko-mediated
          API that creates legitimate card_actions / card_changes (Source 6
          territory; Source 2 just identifies the seam).

    For each step, state:
      - Is it possible with Hyperon Space at HEAD?
      - Is it possible with the API on a feature branch (pr-852,
        artur-das-v102)?
      - Does it require new code in hyperon-experimental? If yes, what?
      - Does it require new code on the Decko side? (Source 5 territory.)
