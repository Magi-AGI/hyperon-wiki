[VERIFIED] Core observer infrastructure at hyperon-space/src/lib.rs:

  pub enum SpaceEvent {                                    // line 22

  }
  impl SpaceCommon {
      pub fn register_observer<T: SpaceObserver + 'static>(// line 106

      pub fn notify_all_observers(                         // line 113

[PRE-RELAY-VERIFIED] Gemini's findings_gemini.txt cites SpaceEvent at
line 18 (correct: 22) and SpaceObserver at line 56 (correct: 63) and
SpaceCommon at line 94 (correct: 98). Drifts are ~4-5 lines, consistent
with the broader CF5.6 pattern. SUBSTANCE correct.

(a) Registration mode: EXPLICIT.
    Caller invokes `space.common().register_observer(MyObserver{...})`
    (per the doc-comment example at hyperon-space/src/lib.rs:36-62).
    Auto-deregistration when SpaceObserverRef + clones drop (Weak refs;
    cleaned up at line 122-123 in notify_all_observers).

(b) Per-Space scope: VERIFIED.
    Observers are stored in SpaceCommon, which is owned per-Space (e.g.,
    GroundingSpace.common at lib/src/space/grounding/mod.rs:19).
    Clone of SpaceCommon (line 128-136) explicitly does NOT clone observers:
    "We don't want to clone observers when a space is cloned, as that leads
    to a situation where an observer can't know which space an event
    pertains to" (lib.rs:131-132). So observers attach to one Space-instance
    and never auto-propagate to clones.

(c) Panic handling: NONE explicitly designed.
    notify_all_observers iterates and calls observer.borrow_mut().notify()
    in sequence (line 116-117). A panic in notify() would unwind the
    writer's stack frame, unwinding the original add/remove/replace call
    that triggered notification.
    (No catch_unwind, no error-return convention — observers must be
    non-panicking for production reliability.)

(d) Synchronous, in-stack-frame: VERIFIED.
    notify_all_observers is called directly from add/remove/replace
    inside GroundingSpace (lib/src/space/grounding/mod.rs:73, 96, 123).
    The writer's call to space.add(atom) returns only AFTER all observers
    have processed the event.

    Phase 4 implication: a Decko-side observer registered against a
    central mirror Space MUST be fast (not block on network/db round-trip)
    or the agent's write call gets stalled. Long-running Decko propagation
    work should be queued from inside notify, not done synchronously.

(e) Thread-safety: NOT THREAD-SAFE.
    Uses Rc<RefCell<...>> throughout (not Arc<Mutex/RwLock<...>>).
    SpaceCommon.observers is RefCell<Vec<Weak<RefCell<dyn SpaceObserver>>>>;
    DynSpace is Rc<RefCell<dyn SpaceMut>>.
    Send + Sync are NOT implemented on these types.

    Phase 4 implication: cannot share a single Space across OS threads
    naively. A Decko deployment with multiple Rails workers requires
    either:
    (i) a single-threaded "Space service" process with IPC boundary
        (gRPC/HTTP/Unix socket) accepting writes from Rails workers, OR
    (ii) one Space-per-worker, synchronized via a shared event bus
         (Redis/RabbitMQ/Kafka) where all workers consume the same
         change-event stream.

(f) Decko-side observer feasibility: ARCHITECTURALLY YES, with the
    caveat in (e). The Rust-side trait + the C-side function-pointer
    shim allow a Decko-callback consumer to register an observer.
    BUT: Python bindings do NOT expose register_observer (see Q8). For
    Phase 4, the Decko-side observer needs to live in Rust or in a
    custom-extension layer that bridges the C API to Python.

VERDICT: [OBSERVER-PATTERN-FIRST-CLASS] [SINGLE-THREADED-RUNTIME]
[OBSERVER-EXEC-SYNCHRONOUS]. The mechanism Phase 4 Option D needs is
in the Rust core and the C API, but is NOT YET in the Python wrapper
(Q8 elaborates).

Q4. CONCURRENCY MODEL
================================================================================

[VERIFIED] Single-threaded by design.
  hyperon-space/src/lib.rs:7-8:
    use std::rc::{Rc, Weak};
    use std::cell::{RefCell, Ref, RefMut};

  No use of std::sync, parking_lot, Arc, Mutex, RwLock anywhere in
  hyperon-space/src/lib.rs.

  All Space-mutation paths flow through:
    DynSpace.0: Rc<RefCell<dyn SpaceMut>>             (lib.rs:282)
    SpaceCommon.observers: RefCell<Vec<...>>          (lib.rs:99)
    GroundingSpace.common: SpaceCommon                (grounding/mod.rs:19)
    GroundingSpace.index: AtomIndex<D>                (grounding/mod.rs:18)

[INFERRED] Multi-process Decko deployment options:

  Option α: Central Space Service.
    - Single-threaded Hyperon Space process (or per-CPU-core processes
      with shard routing) listens on IPC (Unix socket / gRPC / HTTP).
    - Rails workers → IPC → Hyperon Space process → SpaceMut::add/etc.
    - Observers run inside the Space service.
    - Pros: single source of truth; observer notifications local.
    - Cons: latency overhead; single-threaded throughput cap; SPOF.

  Option β: Per-Worker Mirror + Event Bus.
    - Each Rails worker has a local Hyperon Space instance.
    - Decko-side after_save fires on whichever worker handled the request;
      that worker writes to its local Space AND publishes to an event
      bus (Redis pub/sub / Kafka / RabbitMQ).
    - All workers' Spaces consume the same event stream and apply.
    - Observers run in each worker's Space.
    - Pros: horizontally scalable; no SPOF.
    - Cons: eventual consistency between workers; agent reads may differ
            momentarily across workers; more complex sync logic.

  Option γ: Out-of-Process Hyperon Space + Per-Worker Cache.
    - Central Space service holds authoritative state.
    - Each worker caches a read-only mirror, refreshed via observer
      events from the central service.
    - Writes go to central service.
    - Pros: better than α for read-heavy workloads.
    - Cons: cache invalidation complexity.

  RECOMMENDED FOR PHASE 4: Option α (Central Space Service) initially,
  for simplicity and observer-locality. Migrate to β or γ if scale
  requires. The DAS module (Q9) provides a built-in path to distributed
  semantics if Phase 5+ requires it.

VERDICT: [INTER-PROCESS-SYNC-REQUIRED]. The single-threaded core is not
a Phase 4 blocker — it just dictates the deployment topology.

Q7. ATOM IDENTITY UNDER MUTATION
================================================================================

[VERIFIED] replace is implemented as remove-then-insert at
lib/src/space/grounding/mod.rs:119-125:

  pub fn replace(&mut self, from: &Atom, to: Atom) -> bool {
      let is_replaced = self.index.remove(from);
      if is_replaced {
          self.index.insert(to.clone());
          self.common.notify_all_observers(
              &SpaceEvent::Replace(from.clone(), to));
      }
      is_replaced
  }

[VERIFIED] Atomicity within single-thread:
  The remove-then-insert sequence runs in one stack frame; no race
  window for single-threaded callers. (Multi-threaded callers can't
  share a Space anyway per Q4.)

[VERIFIED] SpaceEvent::Replace carries BOTH old and new atoms:
  hyperon-space/src/lib.rs:28: Replace(Atom, Atom)
  Observers receive both, allowing them to track the transition.

[INFERRED] Atom identity is content-addressed (structural):
  Atoms are compared via PartialEq derived on the Atom type (in hyperon-
  atom). There is no UUID or stable handle. Two atoms with identical
  structure are equal; mutations create new atoms (per the Replace
  semantics) rather than modifying handles in-place.

[INFERRED] Phase 4 (L1) round-trip implication:
  When an agent writes a modified atom representing card 7432:
    - If using replace(old_atom, new_atom): the SpaceEvent::Replace
      observer can recognize that the OLD atom corresponded to card 7432
      and the NEW atom is its update.
    - If using add(new_atom) without remove(old_atom): observer sees
      Add event but cannot easily disambiguate "this is card 7432's
      update" vs "this is a wholly new card".

  For Phase 4 (L1) reliability, agents should standardize on
  replace(...) for updates (vs naïve add). This becomes a Decko-side
  protocol: when the mirror service receives a Decko after_update event
  for card 7432, it should call space.replace(old_atom_7432, new_atom_7432).

VERDICT: [STRUCTURAL-IDENTITY] [REPLACE-IS-REMOVE-THEN-ADD]
[REPLACE-EVENT-CARRIES-OLD-AND-NEW]. Phase 4 (L1) round-trip is
SUPPORTED via replace+observer, BUT requires a Decko-side protocol
discipline for using replace consistently.

Architecture for Phase 4 Option D (refined from Source 2 findings):

  ┌─────────────────────────────────────────────────────────────────────┐
  │                    Phase 4 (B)+(L1) Wiring                          │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  Decko Rails worker (Ruby)                                          │
  │  └── after_save / after_update / after_destroy callback fires      │
  │      └── Build change-event {action_type, card_id, before, after}   │
  │          └── Enqueue to Redis/Sidekiq (or message bus)             │
  │                                                                     │
  │  Mirror Service (Rust process or Python process)                    │
  │  └── Consume change-event from queue                                │
  │      └── Translate to atom representation                           │
  │          ├── If action_type=create: space.add(atom)                 │
  │          ├── If action_type=update: space.replace(old, new)         │
  │          └── If action_type=delete: space.remove(atom)              │
  │              └── SpaceEvent fires synchronously                     │
  │                                                                     │
  │  Observer chain (registered against the mirror Space)              │
  │  ├── L1 agent observer: forwards to BasicEventBus → EventAgent      │
  │  └── Decko-mediated write-back observer: handles agent-driven       │
  │      atom edits, translates to Decko Rails REST API calls (Source 6 │
  │      design) creating proper card_actions / card_changes records.   │
  │                                                                     │
  │  L1 Agent loop (in Python or Rust)                                  │
  │  ├── Subscribes to BasicEventBus for change notifications          │
  │  ├── Performs reasoning over current Space state                    │
  │  └── Writes updated atoms via space.add / space.replace / etc.      │
  │      └── Triggers observer chain again (Decko-mediated write-back)  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘

  S1. Decko after_save → Redis enqueue.
      Feasible at HEAD: YES (Decko Rails has standard Sidekiq integration).

  S3. Mirror Service → space.add/replace/remove.
