Q2. SPACEEVENT + SPACEOBSERVER PATTERN
--------------------------------------------------------------------------------

[CONSENSUS] [OBSERVER-PATTERN-FIRST-CLASS] [SYNCHRONOUS-IN-PROCESS]
[SINGLE-THREADED-LIMITATION]. `SpaceEvent` enum {Add, Remove, Replace} at
hyperon-space/src/lib.rs:22-29; `SpaceObserver::notify(&mut self, event)` at
:63-66; `SpaceCommon` at :97-100 with `observers: RefCell<Vec<Weak<RefCell<
dyn SpaceObserver>>>>`; `register_observer<T>` at :106-109 returning a
`SpaceObserverRef<T>`; `notify_all_observers` iterates and calls
`observer.borrow_mut().notify(event)` synchronously at :113-125.

Properties (consensus across reviewers):
  (a) Explicit registration via `space.common().register_observer(...)`.
      Auto-deregistration when SpaceObserverRef + clones drop (Weak refs).
  (b) Per-Space scope. SpaceCommon is owned per-Space; observers are NOT
      global.
  (c) Synchronous, no error path. Trait method returns `()`. A panicking
      observer panics the writer.
  (d) Synchronous, in writer's stack frame. The original add/remove/replace
      call returns only after all observers have processed.
  (e) NOT thread-safe. Rc/RefCell only.
  (f) Decko-side observer FEASIBLE in Rust/C, BLOCKED in Python until a
      `space_register_observer` binding is added (Q8).

[CF5.6 V-Gemini-Q2] Gemini cited:
  - SpaceEvent at lib.rs:18 — actual is :22-29.
  - SpaceObserver at :56  — actual is :63-66.
  - SpaceCommon  at :94   — actual is :97-100.
  Adjudication: -4 to -6 line drift. Substance correct. Adopt Claude/Codex
  lines for SPEC_DRAFT.

[NEW-CLAUDE Q2] `SpaceCommon::clone` at lib.rs:128-135 deliberately drops
observers (the cloned SpaceCommon has `observers: RefCell::new(vec![])`).
The in-source comment explicitly states the design rationale ("an observer
can't know which space an event pertains to"). Combined with
`GroundingSpace`'s `#[derive(Clone)]` at mod.rs:16, every clone of a space
has zero observers. Phase 4 implication: mirror-fanout cannot rely on
observer-cloning; each replica must register observers explicitly. Codex
mentioned the Clone behavior in passing at :128-135 but didn't tease out
the fanout consequence.

[NEW-CLAUDE Q4-Q2-bridge] C API doc-comment at c/src/space.rs:117-119
states explicitly:
  "This API is likely to change in the future to implement multi-threading
   as the space's payload object must not be modified while other operations
   can access the space"
Maintainer-acknowledged: the API is single-threaded BY DESIGN at HEAD; a
multi-threaded variant is planned but not yet present. Phase 4 should not
assume thread-safety lands within Phase 4's timeline.

Q4. CONCURRENCY MODEL
--------------------------------------------------------------------------------

[CONSENSUS] [SINGLE-PROCESS-IN-MEMORY] [INTER-PROCESS-SYNC-REQUIRED].
The core uses `Rc<RefCell<...>>` throughout. `Send`/`Sync` are not
implemented. Multi-Decko-worker deployments cannot share a `GroundingSpace`
in memory; deployment topology must use either:
  α. Central Space service with IPC boundary.
  β. Per-worker Space synchronized via shared event bus.
  γ. Out-of-process authoritative Space + per-worker read-only cache.

[CONSENSUS-RECOMMENDATION] Option α (central service) for Phase 4 baseline,
for simplicity and observer-locality. β/γ are scaling alternatives if
α becomes a bottleneck.

[CONSENSUS-CODEX] DAS uses `Arc<Mutex<Properties>>` at
`lib/src/metta/runner/builtin_mods/das.rs:64`, but only for DAS-specific
parameter passing — does NOT make `GroundingSpace`/`DynSpace` thread-safe.

[NEW-CLAUDE Q4] (Reproduced from Q2 adjudication): the maintainer's own

design is intentional and that thread-safety is a future change, not present

Q7. ATOM IDENTITY UNDER MUTATION
--------------------------------------------------------------------------------

[CONSENSUS] [STRUCTURAL-IDENTITY] [REPLACE-IS-REMOVE-THEN-ADD-AT-INDEX-LEVEL]
[REPLACE-EVENT-CARRIES-OLD-AND-NEW]. `GroundingSpace::replace` at
mod.rs:119-126 is implemented as `index.remove(from)` + `index.insert(to)`,
but the observer signal is a single `SpaceEvent::Replace(from, to)` — so
observers CAN distinguish a Rust/Python `.replace()` call from a separate
remove-then-add pair, IF the call comes from the Rust/Python `replace`
surface.

[CONSENSUS] Atom identity is structural (atoms are equality-by-tree-content).
No primary-key / stable-handle mechanism in `SpaceMut`. For Phase 4 (L1)
round-trip, agents writing through Rust/Python should standardize on
`replace(old, new)` for updates — Decko-side after_update events fire
mirror.replace_atom; agent-side reasoning chooses replace for in-place
updates.

[NEW-CLAUDE Q7] (Cross-references Q1-NEW-CLAUDE asymmetry.) At the MeTTa
surface, no `replace-atom!` op exists; MeTTa-writing L1 agents are forced
to remove-then-add and the Replace observer signal is lost. SPEC_DRAFT
should include the small grounded-op patch to add `replace-atom!`.

Q8. PYTHON BINDINGS COVERAGE
--------------------------------------------------------------------------------

[CONSENSUS] [WRITE-API-EXPOSED] [OBSERVER-API-MISSING-IN-PYTHON]
[C-OBSERVER-API-EXISTS-AND-IS-BINDABLE].

Write surface in Python:
  - `SpaceRef.add_atom` (base.py:205-209), `remove_atom` (:211-215),
    `replace_atom` (:217-221) call hyperonpy's `space_add/space_remove/
    space_replace`.
  - `AbstractSpace` Python base class with virtual `add`/`remove`/`replace`
    (base.py:26-42) for custom-space subclasses.
  - `SpaceRef(space_obj)` constructor (base.py:174-183) wraps a non-CSpace
    Python object via `hp.space_new_custom`.
  - `PY_SPACE_NO_SUBST_API` at `python/hyperonpy.cpp:387-397` is the C
    function table for Python custom spaces. `py_space_add/remove/replace`
    at :422-435 / :437-454 / :456-476 auto-emit `space_event_t` to the
    C observer machinery via `space_params_notify_all_observers` after
    the Python callback returns. So a Python custom space's mutations DO
    fire SpaceEvents that a C-side observer can receive — but only into
    observers registered through the C API.

Observer surface in Python:
  - Verified-negatively (pickaxe in `python/`): `register_observer` and
    `space_register_observer` return ZERO matches.
  - `python/hyperon/base.py` does not expose any observer-registration
    method on SpaceRef or GroundingSpaceRef.
  - `python/hyperonpy.cpp` does not bind `space_register_observer`.
  - `PY_SPACE_NO_SUBST_API` has 9 function-pointer slots
    (query/[subst-NULL]/add/remove/replace/atom_count/iter/free) and NO
    observer slot. Even Python custom Spaces cannot register their own
    observers via this API at HEAD.
  - Cross-branch verification (Codex): `origin/add_python_oper_ext`,
    `origin/pyrust`, `origin/artur-das-v102` — none add the binding.

[CONSENSUS] The C API at `c/src/space.rs:464` already exposes
`space_register_observer(space, observer_api, observer_payload) ->
space_observer_t` with full `space_observer_api_t` plumbing (declared at
:328-342). The C side handles the heavy lifting; the gap is purely the
pybind11 layer in hyperonpy.cpp.

[CONSENSUS-FOUR-UNBLOCK-PATHS] (Claude original; Codex agrees with
implication; Gemini implies via "bridge code required"):
  Path I:  Extend space_api_t with observer slot; update Rust + Python
           wrappers. Touches upstream; ~1-2 weeks.
  Path II: Rust-side bridge service holding the central GroundingSpace,
           exposing gRPC/HTTP for write+observer; Decko Rails workers and
           agents talk to it. Doesn't touch upstream; ~2-3 weeks.
  Path III:PyO3-driven extension crate that imports hyperon-space directly
           and exposes observer registration to Python. Doesn't touch
           upstream; ~2-3 weeks.
  Path IV: PR upstream to add `space_register_observer` binding to
           `python/hyperonpy.cpp`. Variable timeline (review-cycle-bound);
           if accepted, becomes baseline.

[CONSENSUS-RECOMMENDATION] Path II for Phase 4 schedule (Rust-only,
fastest). Path IV in parallel for upstream alignment. Phase 4 SPEC_DRAFT
should commit to Path II as the implementation strategy and Path IV as
the upstream-contribution strategy.

convention). Either grep alone is insufficient; the dual-token grep was
what definitively confirmed the gap.
