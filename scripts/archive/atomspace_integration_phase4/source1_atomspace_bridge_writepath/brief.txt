1. CONTEXT (read before doing the research pass)
--------------------------------------------------------------------------------

Phase 4 of the Hyperon Wiki AtomSpace integration program is the (B)+(L1)
co-canonical AtomSpace+Postgres write-through with agent-writable atoms.
Decko/Rails/PostgreSQL stays the durability + permissions + history layer;
AtomSpace serves as a queryable, inspectable, AND writable mirror; both stores
are authoritative; agents close self-improvement loops by writing atoms back
into the mirror.

(B) and (L1) are distinct write directions and Source 1 must audit both:
- (B) Decko->AtomSpace propagation: a card edit in Postgres surfaces as an
  atom mutation in the mirror. This is the direction R2.B4 already flagged
  as "write stubs empty" — Source 1 verifies and quantifies the gap.
- (L1) Agent->AtomSpace->Decko round-trip: an agent writes an atom in the
  mirror; that mutation must propagate back to Decko's card_actions /
  card_changes revision tables so Decko's history remains the loop's
  durable memory. This direction depends on atom-identity stability
  (Q11), conflict-resolution semantics (Q12), and per-cardtype Decko
  semantics preservation (parked at Source 5). Source 1 audits whether
  the bridge code surface admits this direction at all.

This is distinct from:
- Phase 3 (READONLY-ATOMSPACE-BRIDGE), which is locked in by the AtomSpace
  cluster pilot (closed 2026-04-29) and is a strict read-only mirror.
- Phase 5+ ((A) Postgres replacement) — explicitly deferred.
- L2 (ECAN attention dynamics) and L3 (PLN inference loops) — explicitly
  deferred.

Foundational planning: Magi Archive cards 11446 ("Atomspace Layer - Wiki
Integration Plan", 2026-03-11) + 11448 ("Phase 1 Implementation Draft",
2026-03-11). Phase numbering reconciliation per (Y) decision (2026-05-01):
Phase 3 = read-only mirror (cluster-pilot lock-in), Phase 4 = write-through
co-canonical, Phase 5+ = primary-store migration, Phase 6+ = distributed.

5. RESEARCH QUESTIONS (answer per agent, then reconcile)
--------------------------------------------------------------------------------

Q1. WRITE-PATH PRESENCE/ABSENCE.
    Audit BridgeStorage.cc for the StorageNode write-side virtual methods:
    storeAtom, storeValue, updateAtom, removeAtom (or whatever the current
    atomspace-storage parent class names them). For each method:
      a. Implemented (with body)?
      b. Stub (throws "not implemented" or returns no-op)?
      c. Inherited default?
    Cite line numbers. This is the ground-truth answer to "is bridge
    write-capable today?"
    Cross-ref: removeAtom semantics intersect the MorkDB link-delete
    blocker flagged by the AtomSpace cluster pilot (closed 2026-04-29).
    If bridge would ever delete atoms representing trashed/restored
    Decko cards, that blocker applies to bridge's Phase 4 viability for
    deletion paths even when bridge writes against Postgres directly
    (the blocker is at the Atomese semantics layer, not the storage
    backend layer).

Q2. LLAPI WRITE PRIMITIVES.
    Audit llapi.h for the abstract write methods (insert, update, delete,
    or whatever the abstraction names them). Then check ll-pg-cxx.cc for
    whether the Postgres backend implements them. Three outcomes:
      a. Abstract methods exist + Postgres impl exists → bridge has a real
         write path that StorageNode just isn't wiring through.
      b. Abstract methods exist + Postgres impl is stubs → architectural
         intent for writes, no execution.
      c. Abstract methods absent → reads only by design.
    This determines whether Phase 4 write-through can REUSE bridge or has
    to add a new code path.

Q3. SCHEMA-MAPPING BIDIRECTIONALITY.
    SQLReader.cc converts SQL rows → atoms. Question: is there a sibling
    SQL writer (file present? referenced? generated at runtime?). If yes,
    audit its mapping rules and how they relate to the reader. If no,
    note explicitly — that's a sharp signal.

Q4. POSTGRES-BACKEND TRANSACTION/CONCURRENCY MODEL.
    Audit ll-pg-cxx.cc for: transaction begin/commit/rollback usage,
    isolation level, prepared-statement use, connection-pool semantics
    (single connection vs pool), error handling on conflict. Does
    ll-pg-cxx assume sole writer, or does it tolerate concurrent writers?
    Decko's Rails app has its own connection pool and transaction
    boundaries; Phase 4 needs both writers to coexist on the same DB
    without deadlocks or lost updates.

Q5. ATOM-TYPE MODEL & SCHEMA DISCOVERY.
    Read bridge-types/atom_types.script and the SQLReader to understand
    how generic SQL tables map to atom types. Bridge's premise (from the
    README/notes) is "any SQL table → AtomSpace as nodes/edges". Decko
    has very specific tables (cards, card_actions, card_changes,
    card_references, etc.) — the question is whether bridge's mapping is
    flexible enough that Decko's schema falls out naturally, or whether
    Phase 4 needs custom-mapping shims.

Q6. EDGELINK vs EVALUATIONLINK MIGRATION.
    The HEAD~12 rename "Change EvaluationLink to EdgeLink" matters: any
    Phase 4 design that consumes bridge output must use EdgeLink. Confirm
    EdgeLink is the canonical link type in bridge HEAD. Also confirm what
    the cardinality / arity / properties of EdgeLink are vs the older
    EvaluationLink form. Note: EdgeLink is also referenced in
    atomspace-linas Design Notes (Source 3); cross-check.

Q7. RELATIONSHIP TO atomspace-storage AND atomspace-pgres.
    BridgeStorageNode inherits from StorageNode (defined in
    atomspace-storage). atomspace-pgres is a separate sibling — a
    different StorageNode for atomspace-pgres's rigid Atom/Value
    schema. Verify:
      a. Is atomspace-bridge's parent class the SAME StorageNode atomspace-
         pgres uses, or a different one?
      b. Are write semantics defined at the StorageNode level (so any
         subclass can implement them) or at the subclass level?
      c. Does atomspace-bridge depend on atomspace-pgres at all (header,
         linkage), or is it independent?
    Answer determines whether write-through implementation could reuse
    atomspace-pgres's writer code (since pgres can write its own tables)
    bridged through bridge's schema-mapping logic.

Q8. DECKO-ADAPTABILITY ASSESSMENT.
    Given Q1–Q7, assess three architectural options for Phase 4:
      Option A. Bridge reads Decko's tables directly with a custom mapping
                file; writes flow Decko → Postgres → bridge re-reads
                (read-only path; this is essentially Phase 3).
      Option B. Decko-side exporter produces an atomese stream / file;
                bridge ingests via existing read-path. No write coupling.
      Option C. Add a write-path to bridge that writes back into Decko's
                Postgres tables, respecting card_actions / card_changes
                revision semantics. This is the Phase 4 (B)+(L1) target.
      Option D. Bypass bridge entirely; Phase 4 is a Decko-side
                Rails-callback-driven AtomSpace mirror that emits
                atoms directly to a Hyperon Space (Source 2 territory).
    For each option, note feasibility, blockers, and which other Source's
    findings change the answer.

Q9. LINAS'S STATED DIRECTION.
    Read opencog/persist/bridge/notes/* (if present) and the README for
    Linas's stated future intent. Cross-reference with atomspace-linas
    Design Notes A–D (Source 3) for any mentions of bridge in the larger
    AtomSpace direction. Question: did Linas intend bridge as a
    read-only importer permanently, or as a stepping stone toward
    bidirectional sync? His "Add mention of sensory perception-action"
    commit suggests he sees bridge as part of a larger I/O pattern, not
    a final state.

Q10. CONCURRENCY-CEILING + LATENCY HEADROOM.
    Decko writes (single-card edits) are infrequent (~tens-to-hundreds per
    minute peak across all users). Bridge reads can be staged. Question:
    if Phase 4 writes-through synchronously on every Decko edit, does
    bridge's design tolerate that write rate, or is bridge built for
    bulk-import-then-read patterns? Cite evidence (commits, comments,
    code structure).

Q11. ATOM-IDENTITY STABILITY UNDER ROUND-TRIP. (L1 prerequisite)
    When BridgeStorage materializes a Decko row (e.g., card 1234) as an
    atom on read, then is asked to re-materialize the same row later (or
    receives a write referencing card 1234), does it produce a STABLE
    atom identifier (URI, handle, or content-addressed name) that the
    AtomSpace and any agent code can rely on across sessions? Probe:
      a. SQLReader.cc atom-naming/keying logic — is the atom name a
         function of (table, primary-key) or generated freshly per read?
      b. BridgeStorage write-path (if any present) — does an incoming
         atom-write recognize an existing (table, primary-key) atom as
         an UPDATE, or always as an INSERT/CREATE?
      c. Re-open semantics: after `cog-bridge-close` + `cog-bridge-open`,
         do the same Decko rows surface as the same atoms?
    File:line evidence. If no current support, characterize the minimum
    extension (new naming convention, ID column convention, content-hash).
    This is foundational to (L1): without identity stability, agent
    "memory" of specific cards fragments and the loop cannot close.

Q12. CONFLICT-RESOLUTION SEMANTICS UNDER CO-CANONICAL WRITES.
    Audit ll-pg-cxx.cc and BridgeStorage.cc for any sync-token,
    version-counter, optimistic-lock, MVCC interaction, transaction-id,
    or sequence-number mechanism that could let bridge detect a Decko-
    side write occurring concurrently with an agent-side atom-write
    against the same logical row. If none present, characterize the
    assumed concurrency model:
      a. Sole-bridge-process (only bridge writes Postgres; Decko offline)?
      b. Sole-Decko-writer (bridge reads only; agent writes go elsewhere)?
      c. Single-DB-writer-of-the-moment (caller-coordinated locking)?
      d. Last-writer-wins with no detection (silent corruption risk)?
    The (B) co-canonical promise is meaningless if bridge silently loses
    or overwrites Decko-side writes. This Q determines whether Phase 4
    needs a sync layer above bridge.

Q13. SCHEMA-EVOLUTION TOLERANCE.
    Decko's schema is not fixed: cardtypes can be added; modules can add
    columns; new metadata fields appear over a release cycle. Audit how
    bridge reacts to a new column or new table appearing in the source
    DB after a `cog-bridge-open`:
      a. Auto-rediscover on next query?
      b. Manual reload-everything required?
      c. Stale until process restart?
      d. Hard-coded schema, additions silently ignored?
    Cite the relevant code path (SQLReader, llapi, or BridgeStorage).
    Phase 4b durability concern: if bridge ossifies on first-open
    schema, every Decko module update breaks the mirror until manual
    intervention.

Q14. AGENT-WRITE-RATE CEILING. (L1 complement to Q10)
    Q10 audits whether bridge tolerates Decko's write rate (tens-to-
    hundreds/min). Agent-driven AtomSpace writes (L1) can be 10^2 to
    10^4 per second during reasoning loops. Audit:
      a. Lock granularity in BridgeStorage / ll-pg-cxx (per-atom?
         per-table? global?).
      b. Transaction overhead per atom-write (commit-per-atom vs batch).
      c. llapi batching / prepared-statement reuse.
      d. Connection-pool semantics (single connection serializes all
         writes, vs pool that can parallelize).
    Code-structure evidence, not benchmarks. Output: a qualitative ceiling
    estimate (orders of magnitude) and the bottleneck identification. If
    bridge's design implies a sub-Decko-rate ceiling, Phase 4 must route
    L1 agent-writes through a different substrate (cf. Source 4 MORK
    write-path comparison).
