CRITICAL HARDENING: silent no-ops vs throwing-stubs. A throwing stub raises an exception

  - No exception
  - No log message
  - No DB write
  - Apparent success
This is more dangerous than throwing-stubs because failure is not observable from the
caller's frame. Any Phase 4 extension MUST add at least an explicit throw in the
unimplemented-write paths to surface accidental usage.

  $ grep -rn 'INSERT|UPDATE\s|DELETE\s|UPSERT|ON CONFLICT' opencog/ tests/ examples/ lib/

Brief Q2 outcomes:
  (a) Abstract methods exist + Postgres impl exists → NO (no write-specific abstract

  (b) Abstract methods exist + Postgres impl is stubs → NO

Q4. POSTGRES-BACKEND TRANSACTION/CONCURRENCY MODEL
--------------------------------------------------------------------------------

[NOT FOUND via grep] Zero transaction control in the entire codebase:
  $ grep -rn 'BEGIN|COMMIT|ROLLBACK|isolation|SERIALIZABLE|REPEATABLE READ|FOR UPDATE|FOR SHARE'
    opencog/ tests/ examples/ lib/
  → No matches.

Every PQexec is auto-commit (Postgres default). Each query is implicitly its own
transaction.

[NOT FOUND via grep] Zero prepared statements:
  $ grep -rn 'PQexecParams|PQprepare|PQexecPrepared'
  → No matches.

Bridge uses PQexec (ll-pg-cxx.cc:98) with raw SQL strings constructed by string
concatenation. The escape_single_quotes utility at llapi.h:124-134 is the only
defense against SQL injection; sanitization is opt-in per call site.

[VERIFIED] Connection-pool model:
  llapi.cc:5-10  comment: "This class is thread-enabled but not thread-safe. Two
                          threads should not try to use one instance of this class
                          at the same time. Each thread should construct it's own
                          instance of this class. This class uses no globals."
  ll-pg-cxx.cc:5-10  same comment, repeated for the Postgres driver.

  BridgeStorage.cc:43           #define POOL_SIZE 3
  BridgeStorage.cc:91-92        if (0 == _initial_conn_pool_size)
                                    enlarge_conn_pool(POOL_SIZE, _name.c_str());
  BridgeStorage.cc:73-83        enlarge_conn_pool: pushes new LLPGConnection
                                instances onto conn_pool (concurrent_stack).
  BridgeStorage.h:54            concurrent_stack<LLConnection*> conn_pool;

  SQLResponse.h:82-85  comment: "If the pool is empty, this will block, waiting for
                                a connection to be returned to the pool. Thus, the
                                size of the pool regulates how many outstanding
                                SQL requests can be pending in parallel."

  SQLResponse.h:61-76  Response constructor / destructor: grabs a connection on
                       first exec(), releases it back to the pool on destruction
                       (RAII pattern).

[INFERRED] Conflict-detection model: NONE.
  No version-counter, no MVCC interaction, no row locks, no UPSERT (ON CONFLICT)
  clauses anywhere. Each query is a standalone auto-commit transaction. Writes (if
  ever wired) would be subject to standard Postgres MVCC defaults; bridge would not
  detect concurrent Decko-side writes against the same row.

VERDICT: [NO-TRANSACTIONS] [NO-PREPARED-STATEMENTS] [NO-CONFLICT-DETECTION].
Bridge assumes single-writer-of-the-moment semantics implicitly; the connection
pool serializes at the libpq layer per-connection but does not coordinate across
connections. With Decko's Rails connection pool also writing to the same DB,
Phase 4 would face race conditions undetectable by either side.

[INFERRED] Write-semantics location:
  Per brief Q7, the question is whether write semantics are defined at the
  StorageNode-parent level (so any subclass could implement them) or at the
  subclass level (atomspace-pgres-only). The presence of `void storeAtom(...)` etc.
  declarations in BridgeStorage.h:103-110 demonstrates these methods exist as
  *virtual interface methods* on the parent StorageNode — that's why bridge
  declares-and-overrides them. So:
    - StorageNode (parent, in atomspace-storage) defines the write-method INTERFACE
      contract.
    - Each subclass (atomspace-bridge, atomspace-pgres, atomspace-rocks, atomspace-
      cog, etc.) must implement them OR provide empty no-op overrides.
    - Bridge chose empty no-ops (Q1).

[INFERRED] Reuse opportunity for Phase 4 (B):
  In principle, a Phase 4 (B) extension could:
    (a) Add a write-path to bridge that reuses atomspace-pgres's writer — NO,
        because bridge does not link atomspace-pgres and atomspace-pgres writes
        to its own rigid Atom/Value tables (per R2.B4: "atomspace-pgres is
        BACKING-STORE-ONLY and not Decko-compatible"), not arbitrary SQL tables.
    (b) Build a new write-path inside bridge that constructs INSERT/UPDATE/DELETE
        SQL from incoming Atom writes, keyed off the SignatureLink schema — YES
        in principle, but requires significant new code (the schema-to-SQL
        write-mapping is the missing piece).

VERDICT: Bridge inherits StorageNode interface from atomspace-storage; is INDEPENDENT
of atomspace-pgres at the linker level. Phase 4 (B) cannot piggyback on atomspace-
pgres's writer; would need from-scratch write-mapping logic in bridge or in a new
adapter.

Q11. ATOM-IDENTITY STABILITY UNDER ROUND-TRIP (L1 prerequisite)
--------------------------------------------------------------------------------

[VERIFIED] AtomSpace identity model (used by bridge):
  - add_node(TYPE, name)      → returns existing handle if (TYPE, name) already
                                exists; else creates new. (Standard AtomSpace
                                content-addressing.)
  - add_link(TYPE, [outgoing])→ returns existing handle if same outgoing seq
                                already linked; else creates new.

[VERIFIED] Bridge's atom-naming for Decko-equivalent rows:
  - Predicate: SQLReader.cc:90  add_node(PREDICATE_NODE, tablename) — STABLE per
                                table name (deterministic).
  - Variables (column names): SQLResponse.h:157  add_node(VARIABLE_NODE, colvalue)
                                — STABLE per column name.
  - Row data (per row, per column): SQLResponse.h:236
                                add_node(tnp->get_kind(), colvalue)
                                — STABLE per (column-type, column-value-string).
  - Row Edge: SQLResponse.h:221-222
                                Handle row = as->add_link(LIST_LINK, elts);
                                as->add_link(EDGE_LINK, pred, row);
                                — STABLE per (table predicate, row's column-value
                                atom sequence).

[VERIFIED via re-derivation] Re-open semantics:
  After cog-bridge-close + cog-bridge-open + re-load, the same row will re-emit
  the same atoms because:
    (a) information_schema.columns query is deterministic (same schema → same
        SignatureLink content);
    (b) row data on a non-changing row produces same column-value strings → same
        atoms via add_node;
    (c) AtomSpace add_link/add_node deduplicates by content, so the EdgeLink for
        a row hashes/compares equal to the prior session's EdgeLink (assuming
        prior atoms were retained).

[CRITICAL FOR (L1)] Atom identity is content-addressed by ROW-DATA + PRIMARY-KEY,
NOT by primary-key alone:
  - Bridge does NOT distinguish primary-key columns from data columns in the
    SignatureLink. SQLResponse.h:140-152 (tabledesc_cb) treats every column as a
    TypedVariable with no primary-key flag.
  - Result: if a Decko card 1234's content changes from "hello" to "world", bridge
    would emit a NEW EdgeLink representing the new row state, ALONGSIDE the old
    EdgeLink (which still represents "hello"). The AtomSpace gets BOTH; bridge has
    no GC for stale EdgeLinks.
  - Re-running fetch-incoming-set on the same Predicate (table) does not delete
    stale EdgeLinks — it just re-emits current rows, with the AtomSpace dedup'ing
    re-emitted atoms but not detecting absence (deletion).

[CRITICAL FOR (L1)] Agent-write update-vs-insert recognition: NONE.
  - storeAtom is empty no-op (Q1), so any agent atom-write is silently dropped.
  - Even if storeAtom were extended, recognizing "this incoming EdgeLink represents
    an UPDATE to an existing row, not a NEW row" requires:
       (1) identifying which outgoing atoms in the EdgeLink correspond to primary-
           key columns (currently not flagged in SignatureLink);
       (2) querying the AtomSpace for an existing EdgeLink with the same primary-
           key but possibly different other-column values;
       (3) issuing UPDATE SQL rather than INSERT SQL based on (2).
  - None of (1), (2), (3) exist in bridge code.

VERDICT for Phase 4 (L1):
  - Atom-identity is [CONTENT-ADDRESSED-IDENTITY]: stable across re-reads of the
    SAME row state, but creates a NEW atom for any changed row state (without
    GC of old).
  - L1 round-trip is NOT supported: no primary-key flagging, no update-vs-insert
    recognition, no GC for stale atoms.
  - Minimum extension for L1: (a) annotate SignatureLink with primary-key column
    indices (e.g., a new TypedVariable annotation or sibling tag link); (b) extend
    storeAtom to look up existing EdgeLink by primary-key, distinguish UPDATE from
    INSERT, and emit corresponding SQL; (c) extend removeAtom (and trigger from
    fetchIncomingSet refresh logic) to GC stale EdgeLinks.

  This is multi-week-to-multi-month work, NOT a small extension.

Q12. CONFLICT-RESOLUTION SEMANTICS UNDER CO-CANONICAL WRITES
--------------------------------------------------------------------------------

[NOT FOUND via grep] Zero version-counter / sync-token / optimistic-lock /
sequence-number / MVCC-interaction code. Already documented under Q4.

[INFERRED] Concurrency model assumption (per brief Q12 options):
  (a) Sole-bridge-process — currently moot because bridge has no write code;
      future extension would be in this category by default unless explicit sync
      is added.
  (b) Sole-Decko-writer — what bridge currently is (reads only, Decko writes
      Postgres).
  (c) Single-DB-writer-of-the-moment with caller-coordinated locking — this is
      what Phase 4 (B) would pragmatically need, but bridge does NOT facilitate
      this (no advisory-lock primitives, no row-version checks).
  (d) Last-writer-wins with no detection (silent corruption risk) — what would
      happen if bridge writes were naïvely added without sync.

  Currently, since bridge does not write, the model is effectively (b) Sole-Decko-
  writer. A naïve Phase 4 extension would put us in (d) without explicit work.

[VERIFIED via Q4] No transaction control (no BEGIN/COMMIT/ROLLBACK), no isolation-
level setting, no FOR UPDATE / FOR SHARE row locks, no UPSERT (ON CONFLICT). Bridge
operates in Postgres default auto-commit + READ COMMITTED isolation, which is the
weakest of the standard isolation levels.

[INFERRED] (B) co-canonical promise viability:
  The (B) co-canonical promise — both stores authoritative — is meaningless if
  bridge silently overwrites Decko-side writes that happened during bridge's
  read-modify-write window. With no version-counter and no optimistic-lock,
  bridge cannot detect such races. The Phase 4 (B) architecture MUST add a sync
  layer above bridge — either:
    - Postgres advisory locks (pg_advisory_lock) acquired by bridge before
      writing a row;
    - A version-counter column on each Decko-mirrored table (Decko's
      `card_actions.id` could serve this role for cards, but not for ad-hoc
      atoms);
    - A change-data-capture mechanism (Postgres LISTEN/NOTIFY, logical replication,
      or Decko-side callback hooks) that tells bridge to invalidate its mirror
      view when Decko writes happen.

VERDICT: Bridge is structurally ill-equipped for co-canonical writes. Outcome (d)
without remediation. Phase 4 (B) requires either a new sync layer above bridge or
a different write substrate.

Q13. SCHEMA-EVOLUTION TOLERANCE
--------------------------------------------------------------------------------

[VERIFIED] Schema is queried per-call:
  load_one_table (SQLReader.cc:56-95) executes information_schema.columns query
  on every invocation. Each call FRESHLY reads the current schema.

[VERIFIED] But SignatureLink uniqueness is enforced:
  get_row_desc (SQLReader.cc:143-157):
    line 150  HandleSeq sigs = tablename->getIncomingSetByType(SIGNATURE_LINK);
    line 151  if (1 != sigs.size())
    line 152-154  throw RuntimeException(TRACE_INFO,
                  "Cannot find signature for %s\n",
                  tablename->to_short_string().c_str());

  This throws if the SignatureLink count is anything other than 1.

[INFERRED] Schema-evolution failure mode:
  Suppose at t=0, table `cards` has columns [id, name, content, type_id].
    load_one_table emits SignatureLink_A wrapping VariableList_A with 4
    TypedVariables.
  At t=1, a Decko module migration adds column `cards.metadata`.
    load_one_table re-runs, emits NEW VariableList_B with 5 TypedVariables.
    Since AtomSpace add_link is content-addressed and VariableList_B differs from
    VariableList_A (different content), a NEW SignatureLink_B is created.
    AtomSpace now has both SignatureLink_A and SignatureLink_B incoming to the
    same PredicateNode "cards".
  At t=2, any subsequent fetch-incoming-set or load_rows invocation calls
    get_row_desc("cards"), which finds 2 SignatureLinks (sigs.size() == 2) and
    THROWS "Cannot find signature for cards".

  Workaround: cog-close + cog-open (full reset). New process re-discovers schema
  from scratch and emits only one (current) SignatureLink, which then satisfies
  the uniqueness invariant.

  Issue: the workaround DESTROYS in-flight AtomSpace state. Phase 4's L1 loop
  cannot rely on the AtomSpace mirror surviving a Decko module migration.

VERDICT: [SCHEMA-EVOLUTION-BREAKS-RUNTIME]. Bridge tolerates schema STATIC; bridge
BREAKS PERMANENTLY on schema evolution mid-session. Phase 4b durability concern
confirmed: every Decko module update would force a bridge restart.

  Minimum extension: relax get_row_desc to pick the most-recent SignatureLink
  (by some timestamp metadata) OR delete stale SignatureLinks during schema
  re-discovery OR enforce a singleton-SignatureLink-per-table invariant via
  explicit removal before re-add. None of these exist.

Q14. AGENT-WRITE-RATE CEILING (L1 complement to Q10)
--------------------------------------------------------------------------------

[VERIFIED] Lock granularity: LLConnection is per-thread, not thread-safe (llapi.cc:
5-10 + ll-pg-cxx.cc:5-10 comments). The pool (concurrent_stack<LLConnection*>)
serializes connection-acquisition; once a thread holds a connection, that
connection is single-threaded for that thread.

[VERIFIED] Transaction overhead: each PQexec is auto-commit. Per-atom-write would
mean per-atom-PQexec round-trip = network/socket latency + Postgres parse + plan +
execute + WAL write + commit. On a local DB, ~1-3ms per atom; on a remote DB,
~5-50ms.

[VERIFIED] Batching: NONE. No prepared statements, no PQexecParams, no copy-mode,
no multi-row INSERT statement construction. (Q4 grep confirmed.)

[VERIFIED] Connection-pool semantics: 3 connections shared across all threads
(BridgeStorage.cc:43 + SQLResponse.h:82-85).

[INFERRED] Agent-write-rate ceiling (qualitative):
  - Best case (local DB, 1ms per write): 3 connections × 1000 writes/sec/connection
    = 3000 atom-writes/sec.
  - Realistic case (local DB, 3ms per write, no batching): ~1000 atom-writes/sec.
  - Remote DB or contention: ~100-500 atom-writes/sec.

  Agent reasoning loops can produce 10²-10⁴ atom-writes/sec (per brief Q14).
  Bridge as-is is at the LOW END of acceptable for low-end agent loops; at
  HIGH-END agent loops (10⁴/sec), bridge would saturate the connection pool and
  serialize the loop.

  For comparison: the brief mentions Source 4 covers MORK + atomspace-rocks
  alternatives. atomspace-rocks (RocksDB-backed) typically supports 10⁴-10⁵
  atom-ops/sec because it is in-process and doesn't pay socket/SQL overhead.

[INFERRED] Bottleneck identification:
  1. Connection-pool size (POOL_SIZE=3 hard-coded) — can be lifted by editing
     BridgeStorage.cc:43 and recompiling, or by exposing it as a constructor
     parameter.
  2. Lack of prepared statements — every PQexec parses + plans the SQL string from
     scratch, ~30-100% per-query overhead.
  3. Lack of batching — no multi-row INSERT, no COPY mode.
  4. Auto-commit — every write fsyncs WAL; group commit (one fsync per batch)
     would 5-10× improve throughput.

VERDICT: Bridge is below threshold for high-rate L1 agent loops. Phase 4 must
either:
  (a) extend bridge with prepared statements + batching + larger pool +
      transaction control, OR
  (b) route L1 agent-writes through a different substrate (cf. Source 4 MORK
      write-path comparison; if MORK supports atom-rate writes natively, that's
      the better L1 target than extending bridge).

Q8. DECKO-ADAPTABILITY SYNTHESIS (Options A/B/C/D)
--------------------------------------------------------------------------------

Option A. Bridge reads Decko's tables directly with a custom mapping file; writes
flow Decko → Postgres → bridge re-reads (read-only path; this is essentially
Phase 3).

  Feasibility (Phase 4 (B) target): N/A — Option A is Phase 3.
  As a Phase 3 fallback: VIABLE. Bridge already does this for FlyBase; Decko's
  schema fits the public-only filter; type-mapping is mostly OK except for
  timestamps (timestamps would need a Type extension to be addable to the
  AtomSpace, but lack of timestamps doesn't break Phase 3 read-only — those
  fields just don't surface in the mirror).

Option B. Decko-side exporter produces an atomese stream / file; bridge ingests
via existing read-path. No write coupling.

  Feasibility (Phase 4 (B) target): N/A — Option B is also read-only.
  Useful if we want to bypass bridge's information_schema.columns introspection
  (e.g., for schema-evolution control or for Decko-cardtype-aware serialization).
  Useful as an architectural fallback if Option A's public-schema-only filter
  becomes problematic.

Option C. Add a write-path to bridge that writes back into Decko's Postgres
tables, respecting card_actions / card_changes revision semantics. This is the
Phase 4 (B)+(L1) target.

  Feasibility: HIGH ENGINEERING COST. Specifically requires:
    - Atom-write→SQL mapping logic (currently zero) — extend SQLReader sibling
      "SQLWriter" for INSERT/UPDATE/DELETE generation.
    - Primary-key flagging in SignatureLink (Q11) — bridge's TypedVariable
      doesn't distinguish PK columns; ALL columns are equal.
    - Transaction control (Q4) — BEGIN/COMMIT around multi-table writes for
      Decko card-edit atomicity (card + card_action + card_change must commit
      together).
    - Update-vs-insert recognition (Q11) — agent edits to existing rows must
      become UPDATE, not duplicate INSERT.
    - Timestamp/jsonb type-mapping (Q5) — Decko's created_at/updated_at currently
      ignored.
    - Schema-evolution tolerance (Q13) — current bridge breaks on Decko module
      migrations; Phase 4 needs a singleton-SignatureLink invariant.
    - Conflict-detection (Q12) — version-counter or pg_advisory_lock layer.
    - Connection-pool sizing + prepared statements + batching (Q14) — for L1
      agent-write rates.
    - Test suite (Q0 setup; tests/ is empty) — no regression baseline.

  Estimated cost: multi-month full-time dev. Bridge-as-extension is essentially
  building a new persistence layer on top of bridge's read scaffolding. The read
  scaffolding's IP is non-trivial (generic schema discovery + EdgeLink mapping +
  pool management), but the write side is greenfield.

  Source dependencies:
    - Source 5 (Decko semantics inventory) defines what Decko-side state must
      reach the mirror — this constrains the Atom-write→SQL mapping above.
    - Source 7 (Magus plan reconciliation) — March 2026 plan may have committed
      to a particular mechanism; if so, Option C work plan must align.

Option D. Bypass bridge entirely; Phase 4 is a Decko-side Rails-callback-driven
AtomSpace mirror that emits atoms directly to a Hyperon Space (Source 2 territory).

  Feasibility: HIGH for (B) Decko→AtomSpace direction. Decko has Rails callbacks
  on every card edit (`set_callbacks`, `subset_card_actions` events) that can fire
  a write to an AtomSpace-side queue. The agent then consumes the queue and writes
  Hyperon Space atoms.

  For (L1) agent→Decko direction: requires a separate mechanism — Hyperon Space
  write-hooks (Source 2) that fire on agent-driven atom writes, then translate to
  Decko's Rails REST API or direct SQL. Either way, doesn't depend on bridge.

  Engineering cost: MEDIUM. Decko-side Ruby callbacks are well-documented; Rails
  REST API for card edits is well-tested; the atom-to-card-edit translation is
  net-new but bounded. Type model can be designed Decko-natively (timestamps
  preserved, jsonb preserved) rather than constrained by bridge's existing
  mapping.

  Source dependencies:
    - Source 2 (hyperon-experimental Hyperon Space write-API) defines what
      target abstraction we write to.
    - Source 5 (Decko semantics) defines what Decko-side hooks we use.
    - Source 6 (agent-facing MCP surface options) — option D could expose a
      higher-level MCP interface than bridge does; design TBD.

[RECOMMENDED OPTION] D, with C as fallback if Source 2 (Hyperon Space) findings
show writable-Hyperon-Space is also unsupported.

Reasoning:
  1. Architectural mismatch: bridge is bulk-import-then-read (Q9, Q10). Phase 4
     wants sustained low-latency write-through. Extending bridge means fighting
     its design grain.
  2. Type model ceiling: bridge can't represent Decko's full schema (timestamps,
     jsonb). Option C requires extending the type model anyway; Option D builds
     the type model fit-to-purpose from the start.
  3. Test coverage: bridge has zero test coverage (Q0). Option C builds writes on
     a tested-zero base. Option D can build with a fresh test suite from day 0.
  4. Identity model: bridge's content-addressed identity (Q11) doesn't model
     Decko card identity well. Option D can use Decko's card-id as the primary
     atom-identity scheme directly.
  5. Conflict-detection: bridge has none (Q12). Option C must add a sync layer
     ABOVE bridge anyway. Option D can use Decko's existing card_actions revision
     log as the conflict-detection mechanism natively.

  However, Option D has its own risks:
    - Depends on Source 2 finding that Hyperon Space supports the needed
      write-API.
    - Requires Decko Rails-side work, not just AtomSpace-side work.
    - Doesn't use bridge's already-existing schema-discovery code.

  Source-dependent re-evaluation: if Source 4 finds MORK is the right L1 substrate
  AND Source 2 finds Hyperon Space is read-only, Option D might evolve to
  D' = Decko-callback → MORK write-path → agents read MORK, write back via MORK.

Final recommendation: **Option D**, contingent on Source 2 + Source 4 + Source 5
findings. Option C is the fallback if those findings show D is infeasible.
Options A and B are read-only-only and don't satisfy Phase 4 (B)+(L1) by themselves.
