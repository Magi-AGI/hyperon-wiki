Q2. LLAPI WRITE PRIMITIVES

VERIFIED: llapi.h exposes only one abstract database operation:
LLConnection::exec(const char *, bool=false) at
opencog/persist/bridge/llapi.h:54. There are no typed insert/update/delete
primitives in the LLAPI contract.

VERIFIED: the Postgres backend implements that one generic exec call using
PQexec in opencog/persist/bridge/ll-pg-cxx.cc:91-99. It accepts command-ok,
empty-query, and tuple-result statuses at ll-pg-cxx.cc:100-103 and throws a
RuntimeException on failure at ll-pg-cxx.cc:112-130.

VERIFIED: there is no bridge-specific transaction, prepared-statement, insert,
update, delete, upsert, or row-diff API in llapi.h or ll-pg-cxx.h. The only
possible write substrate is arbitrary SQL text sent through exec.

VERIFIED: llapi.cc contains an INSERT only inside a disabled UNIT_TEST_EXAMPLE
block at opencog/persist/bridge/llapi.cc:175-205. That is not part of the
BridgeStorage write path.

Outcome: Q2 outcome C. Abstract write primitives are absent. The bridge can
execute arbitrary SQL strings at the driver layer, but no semantic write API
exists for mapping AtomSpace mutations back to SQL rows.

Q4. POSTGRES-BACKEND TRANSACTION/CONCURRENCY MODEL

VERIFIED: bridge hard-codes a pool size of 3 in
opencog/persist/bridge/BridgeStorage.cc:42-43, opens that pool on first open at
BridgeStorage.cc:91-92, and checks connectivity by popping/pushing one
connection at BridgeStorage.cc:117-128.

VERIFIED: Response obtains a connection from the pool and blocks if the pool is
empty. The comment explicitly says the pool size regulates the number of
outstanding parallel SQL requests at opencog/persist/bridge/SQLResponse.h:78-87.

VERIFIED: the Postgres connection wrapper is "thread-enabled but not
thread-safe"; two threads must not use one instance at the same time
(opencog/persist/bridge/ll-pg-cxx.cc:5-10). The pool gives each Response a
connection instance, so read concurrency is limited by the pool.

VERIFIED: bridge uses raw PQexec in ll-pg-cxx.cc:98. No PQprepare,
PQexecParams, PQexecPrepared, BEGIN, COMMIT, ROLLBACK, FOR UPDATE, lock, or
isolation-level use was found in opencog/persist/bridge.

VERIFIED: SQL is constructed by string concatenation:
- table description query: SQLReader.cc:62-65.
- table SELECT query: SQLReader.cc:165-177.
- WHERE query: SQLReader.cc:248-259.

INFERRED: bridge tolerates concurrent readers up to the pool limit and relies
on PostgreSQL's default per-statement snapshot behavior. It does not coordinate
with Rails transactions, Decko connection pools, or concurrent writers. It
contains no lost-update or deadlock avoidance logic because it performs no
writes.

Risk note: the current read path interpolates table/column/atom names into SQL.
For a controlled bridge over trusted metadata this may be acceptable; for
agent-controlled atoms it becomes a SQL-injection and quoting hazard before any
write path is even considered.

Q8. DECKO-ADAPTABILITY ASSESSMENT

Option A. Bridge reads Decko's tables directly; Decko writes Postgres; bridge
re-reads.

VERIFIED feasibility: technically feasible as a Phase 3-style read/import path
for public-schema tables, using load_tables and fetchIncomingSet. Evidence:
load_tables public schema at SQLReader.cc:97-124; row loading at
SQLReader.cc:188-209 and SQLResponse.h:212-224.

Blockers:
- public schema only and no schema qualification.
- no custom mapping implementation despite README desired custom templates at
  README.md:13-18.
- Decko semantics flatten into generic EdgeLink rows.
- no write-back and no stable Decko identity model.

Assessment: useful as an importer/reference, not a Phase 4 co-canonical write
solution.

Option B. Decko-side exporter produces atomese stream/file; bridge ingests via
existing read path; no write coupling.

INFERRED feasibility: a Decko-side exporter is feasible and keeps Decko as the
semantic authority. However, bridge itself does not ingest Atomese streams; it
reads SQL through Postgres. If the exporter writes an intermediate SQL schema,
bridge could read that schema. If it emits Atomese directly, bridge is mostly
irrelevant.

Blockers:
- still no agent-write round trip.
- semantics live in exporter, not bridge.
- Source 5 must define Decko cardtype/revision/permission mapping.

Assessment: reasonable incremental read-mirror path, but not sufficient for L1.

Option C. Add a bridge write path into Decko Postgres tables.

VERIFIED blockers:
- StorageNode write methods are empty no-ops (Q1).
- LLAPI lacks write primitives beyond raw exec (Q2).
- no SQLWriter or inverse mapper (Q3).
- no transactions/conflict/versioning (Q4/Q12).
- row identity is full tuple content, not Decko primary identity (Q11).
- bridge has no Decko history model.

INFERRED feasibility: possible only as a new subsystem, not a small extension.
It would have to become a Decko-aware service that creates card_actions and
card_changes correctly, respects permissions, handles trash/restore, and
coordinates with Rails transactions. At that point, the center of gravity is
Decko semantics, not generic bridge semantics.

Assessment: not recommended as Phase 4 baseline.

Option D. Bypass bridge for write-through; use Decko-side Rails callbacks/API
to maintain a Hyperon/AtomSpace mirror.

INFERRED feasibility: strongest fit for Phase 4 because Decko remains the
durability, permissions, and history owner. Decko can emit canonical atom
mutations after successful card transactions and require agent writes to enter
through a Decko-mediated API that creates normal card_actions/card_changes.

Blockers:
- depends on Source 2 for Hyperon Space write API maturity.
- depends on Source 5 for Decko semantic inventory.
- depends on Source 6 for agent-facing MCP/API surface.
- if Hyperon Space is not ready, the same Decko-side pattern can target a
  classical AtomSpace mirror first.

Recommendation: Option D for Phase 4 target; Option A as a read-only reference
and test fixture; Option B as a conservative interim exporter if Source 2 is
not ready. Do not bet Phase 4 on Option C unless the plan explicitly budgets a
new Decko-aware sync/write subsystem and treats bridge as a source of read-path
ideas only.

Q10. CONCURRENCY-CEILING + LATENCY HEADROOM

VERIFIED: no synchronous Decko->bridge write path exists, so current Decko edit
write-through capacity is 0 writes/sec.

VERIFIED: bridge's implemented path is read-oriented and can issue at most
pool-size-limited concurrent SQL requests. Pool size is 3 at
BridgeStorage.cc:42-43; Response blocks on pool exhaustion at SQLResponse.h:82-87.

VERIFIED: no write queue, batching, prepared statements, or transaction batching
exists in bridge. Barrier is empty at BridgeStorage.cc:149-151.

VERIFIED: bridge documentation emphasizes incremental/as-needed loading rather
than full bulk import at README.md:26-29, but examples also use bulk table
loads. basic-demo warns a genotype_id load can take 1 to 5 minutes and 7GB RAM
for almost 4M rows at examples/basic-demo.scm:183-193. It reports featureloc
bulk load at about 46K rows/sec and 250GB RSS at examples/basic-demo.scm:231-233.

INFERRED: Decko's human write rate, tens to hundreds per minute, is not the hard
technical rate problem for Postgres. The hard problem is correctness: making
Decko revision/permission/history writes atomically. A direct bridge write path
would be latency-sensitive but would still be dominated by Decko semantics and
transaction correctness, not raw row throughput.

Q11. ATOM-IDENTITY STABILITY UNDER ROUND-TRIP

VERIFIED: current row identity is content-shaped:
- Each SQL value becomes a ConceptNode or NumberNode by value at
  SQLResponse.h:235-237.
- Each row becomes ListLink(column value atoms), then EdgeLink(table predicate,
  row list) at SQLResponse.h:212-224.

VERIFIED: primary key metadata is not used as a row identity. README explicitly
punted on key modeling and stores naked primary/foreign-key values as atoms at
README.md:75-83 and README.md:123-124.

VERIFIED: select_where can query a column value and comments that if the entry
is a primary key only one row is returned at SQLReader.cc:236-240, but the code
does not know which column is primary. It simply builds WHERE column = value at
SQLReader.cc:248-259.

INFERRED: unchanged rows should rematerialize as the same Atomese expression
within AtomSpace's normal type/name/outgoing uniqueness rules. Across sessions,
the semantic expression can be reproduced, but no persistent handle/UUID is
stored by bridge.

INFERRED: this is not stable enough for L1 round-trip writes:
- updating any non-key column changes the ListLink and therefore changes the
  row EdgeLink identity.
- identical duplicate rows collapse into one atom.
- no incoming atom write can be classified as UPDATE vs INSERT because there is
  no table+primary-key identity convention.
- Decko card identity should be card id/name plus revision semantics, not full
  current row tuple identity.

Minimum extension required: introduce explicit stable identity atoms such as
DeckoCardNode/CardIdNode keyed by card id, or a canonical Edge/Value convention
keyed by (schema, table, primary-key), and store mutable fields/history as
separate values or relation atoms. Writes must resolve against that identity
before creating Decko revisions.

Q12. CONFLICT-RESOLUTION SEMANTICS UNDER CO-CANONICAL WRITES

VERIFIED: no bridge sync token, version counter, optimistic lock, xmin/MVCC
read, sequence check, updated_at check, FOR UPDATE, advisory lock, or transaction
protocol was found.

VERIFIED: bridge has no BEGIN/COMMIT/ROLLBACK usage; ll-pg-cxx.cc:91-99 sends a
single SQL string through PQexec. SQLReader constructs SELECTs only.

VERIFIED: current write methods are no-ops, so there is no conflict behavior to
evaluate at runtime.

Finding: current model is Q12(b), sole-Decko-writer with bridge as reader. If a
direct bridge write path were added without a new sync layer, the likely outcome
would be silent lost updates, constraint errors, or application-level corruption
because Decko and bridge would not share revision/permission/conflict semantics.

Phase 4 requirement: conflict detection and resolution must sit above bridge or
inside a Decko-aware writer. Decko's own revision tables and Rails transaction
boundaries must be the source of truth for agent writes.

Q13. SCHEMA-EVOLUTION TOLERANCE

VERIFIED: open() does not discover table schemas; examples explicitly call
cog-bridge-load-tables after cog-open at examples/basic-demo.scm:47-55.

VERIFIED: load_tables discovers public tables when called at SQLReader.cc:97-124.
load_one_table discovers columns for one table when called at SQLReader.cc:56-65.

VERIFIED: fetchIncomingSet(Predicate) loads table data using the already-known
signature at SQLReader.cc:366-383. It does not rediscover columns.

VERIFIED: get_row_desc requires exactly one Signature for a table and throws if
the count is not exactly one at SQLReader.cc:143-156.

INFERRED:
- New table after cog-bridge-open: not visible until cog-bridge-load-tables is
  manually called again.
- New column on existing table: ignored by existing signature until manual
  reload.
- Reload after column change can create a second, different Signature for the
  same PredicateNode. Because get_row_desc requires exactly one signature, this
  can break later loads instead of cleanly refreshing the schema.
- Same table name across schemas remains unresolved; README lists this as an
  open question at README.md:291-293.

Finding: schema evolution tolerance is manual and fragile. For Decko module
evolution, bridge needs explicit schema refresh/dedupe/versioning or a
Decko-side export contract with stable atom schema.

Q14. AGENT-WRITE-RATE CEILING

VERIFIED: current L1 agent write ceiling through bridge is 0 writes/sec because
storeAtom/removeAtom/storeValue/updateValue are empty no-ops (SQLReader.cc:395-409).

VERIFIED: bridge has no batching or write queue. It has a three-connection pool
(BridgeStorage.cc:42-43) and blocking per-query connection acquisition
(SQLResponse.h:82-87). It uses raw PQexec (ll-pg-cxx.cc:98) and no prepared
statement reuse.

VERIFIED comparison: atomspace-pgres has a real async write queue and watermarks
at atomspace-pgres/.../SQLAtomStorage.h:238-239 and SQLAtomStorage.cc:61-73.
Its comment assumes hardware can do about 1K atom stores/sec or better and sets
an 800/150 backlog at SQLAtomStorage.cc:69-73. Bridge has none of that machinery.

INFERRED qualitative ceiling:
- Current bridge: 0 agent writes/sec.
- Straightforward direct-SQL bridge extension: likely suitable for low human/
  Decko write rates only if correctness is solved; it is not shaped for 10^2 to
  10^4 reasoning-loop writes/sec.
- Decko-canonical writes into card_actions/card_changes should not be driven at
  raw reasoning-loop rates. L1 needs batching, throttling, staging, or a separate
  high-rate atom substrate with Decko commits reserved for durable semantic
  milestones.

Finding: if Source 4 identifies MORK/DAS/other substrates with better high-rate
atom writes, L1 agent-loop writes should route there or through a staging layer,
then commit selected Decko revisions through a Decko-mediated API. Bridge is not
the high-rate write substrate.
