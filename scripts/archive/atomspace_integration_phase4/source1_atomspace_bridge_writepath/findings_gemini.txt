Q2. LLAPI WRITE PRIMITIVES
--------------------------------------------------------------------------------

[VERIFIED] llapi.h (line 54) defines a generic virtual `exec(const char *, bool=false)` method. This allows raw SQL execution, which technically supports writes (INSERT/UPDATE/DELETE). However, there are no abstract write-specific methods (e.g., `insert_row`, `update_row`).

[VERIFIED] ll-pg-cxx.cc (line 101-103) handles `PGRES_COMMAND_OK`, which is the status for successful non-SELECT commands.

VERDICT: [READ-ONLY-BY-DESIGN] for high-level Atom operations, though the low-level API allows raw SQL writes.

Q8. DECKO-ADAPTABILITY ASSESSMENT
--------------------------------------------------------------------------------

[RECOMMENDED OPTION] **Option D** (Bypass bridge, use Decko-side callbacks).

Reasoning:
- Bridge is architecturally optimized for bulk loading and querying, not low-latency bidirectional sync.
- The lack of transaction support and conflict detection would require significant refactoring.
- The current type mapping drops essential Decko fields (timestamps).
- Option D allows for a more natural mapping of Decko's complex card semantics without being constrained by bridge's generic table-to-EdgeLink mapping.

VERDICT: [CONTENT-ADDRESSED-ONLY].
