Q1. WRITE-PATH PRESENCE/ABSENCE
  CONSENSUS [VERIFIED across all three]: All five StorageNode write methods
  (storeAtom, removeAtom, storeValue, updateValue, storeAtomSpace) are declared
  at BridgeStorage.h:103-110 and overridden as empty-body silent no-ops at
  SQLReader.cc:395-425. Read-side stubs (getAtom, getLink, loadValue, loadType,
  loadAtomSpace) are also empty no-op overrides. Only fetchIncomingSet
  (SQLReader.cc:366-387) and fetchIncomingByType (line 389-393, delegates) are
  wired. barrier() is also empty (BridgeStorage.cc:149-151) despite write-fence
  documentation comments at lines 144-148.

  Verdict label: [WRITE-PATH-ABSENT-SILENT-NO-OP].

  R2.B4 from AtomSpace cluster pilot 2026-04-29 ("write stubs empty") is
  VERIFIED and HARDENED: stubs are silent-no-op overrides (not throws), with
  pickaxe (V1-Q1 corrected scope; see Dissents) showing they have never been
  given bodies in repo history.

Q2. LLAPI WRITE PRIMITIVES
  CONSENSUS [VERIFIED across all three]: llapi.h declares only one virtual
  method on LLConnection: `virtual LLRecordSet *exec(const char *, bool=false)`
  (llapi.h:54). LLRecordSet has only read-side methods. ll-pg-cxx.cc:91-136
  implements exec via raw PQexec (line 98). PGRES_COMMAND_OK is in the accepted
  result set (line 101), so the API admits writes via raw SQL strings, but no
  caller constructs write SQL. The single INSERT at llapi.cc:204 is inside a
  doubly-disabled #ifdef and is dead example code.

  Verdict label: [READ-ONLY-BY-DESIGN] at the structured-API level; raw
  exec() is technically write-capable but unused.

Q3. SCHEMA-MAPPING BIDIRECTIONALITY
  CONSENSUS [VERIFIED across all three]: No SQLWriter.cc exists. SQLReader.cc
  contains exclusively read-side mapping; SQL-construction primitives generate
  only SELECT statements. SQLResponse.h callbacks emit add_link/add_node calls
  (in-memory AtomSpace insertion) only, never write-SQL.

  Verdict: bridge is structurally read-only, not architecturally ambivalent
  about writes.

Q4. POSTGRES-BACKEND TRANSACTION/CONCURRENCY MODEL
  CONSENSUS [VERIFIED via grep]: Zero BEGIN/COMMIT/ROLLBACK, zero PQexecParams/
  PQprepare/PQexecPrepared, zero isolation-level / SERIALIZABLE / REPEATABLE READ
  / FOR UPDATE / FOR SHARE / ON CONFLICT in entire codebase. Auto-commit only.
  String-concat SQL via PQexec, with escape_single_quotes (llapi.h:124-134) as
  the only injection defense. POOL_SIZE=3 (BridgeStorage.cc:43; corrected from
  Gemini citation — see V3-Q4). Pool is the bottleneck per SQLResponse.h:82-85.
  Per-thread connection model (llapi.cc:5-10 + ll-pg-cxx.cc:5-10).

  Verdict labels: [NO-TRANSACTIONS] [NO-PREPARED-STATEMENTS] [POOL-SIZE-3-HARDCODED].

Q5. ATOM-TYPE MODEL & SCHEMA DISCOVERY (Phase 4 BLOCKER)
  CONSENSUS [VERIFIED across all three]: Type-mapping at SQLResponse.h:153-204
  silently drops timestamp, date, jsonb, bpchar columns (sets tcol = nullptr).
  Column types text/varchar → ConceptNode; int4/int2/int8/float4/float8/bool →
  NumberNode. atom_types.script defines only FOREIGN_STORAGE_NODE +
  BRIDGE_STORAGE_NODE — no domain-specific types. Schema discovery uses
  information_schema.columns (portable) + pg_tables WHERE schemaname='public'
  (Postgres-specific, hard-coded at SQLReader.cc:110, code comment line 109
  self-acknowledges as "the correct hack-of-the-moment").

  Decko impact (Phase 4 blocker):
  - cards.created_at, cards.updated_at: timestamp → IGNORED. Decko's revision
    chronology cannot be reconstructed from the AtomSpace mirror without
    extending table_column_cb to handle timestamp.
  - card_changes.diff_obj or any jsonb columns: IGNORED.
  - card_actions.action_type: varchar → ConceptNode (OK).
  - All Decko PK/FK fields (bigint): NumberNode (OK).

  Verdict labels: [PUBLIC-SCHEMA-HARDCODED] [TYPE-MODEL-INCOMPLETE-FOR-DECKO].

Q6. EDGELINK vs EVALUATIONLINK MIGRATION
  CONSENSUS [VERIFIED via pickaxe]: Migration completed at commit 2bc00ef. At
  HEAD, only EDGE_LINK references exist in bridge sources (SQLReader.cc:287, 340;
  SQLResponse.h:222). The README example at README:64-73 + line 108-109 confirms
  EdgeLink is canonical. EdgeLink arity at bridge usage: outgoing[0]=Predicate
  (table name), outgoing[1]=List (column-value atoms in column-position order).

  Note V3-Q6 dissent (see Dissents): Gemini's atom_types.script:15 citation is
  a CF5.6 drift; correct location of EdgeLink definition is in the parent
  AtomSpace project, not bridge.

Q7. RELATIONSHIP TO atomspace-storage AND atomspace-pgres
  CONSENSUS [VERIFIED across all three]: BridgeStorage inherits from StorageNode
  (BridgeStorage.h:48; StorageNode declared in atomspace-storage). CMakeLists.txt
  declares dependencies on AtomSpace 5.0.3 (line 82), AtomSpaceStorage (line 91),
  and PGSQL 12 (line 101, REQUIRED commented out — sloppy CMake; see F2). NO
  dependency on atomspace-pgres anywhere. Bridge implements its own ll-pg-cxx
  driver from scratch.

  Codex contributed valuable additional finding (CF1-Q7): atomspace-pgres @
  6fe97630071455c8bd491e7ddc2ec97936a9979a HAS real writer code:
    SQLAtomStorage::storeAtom    SQLAtomStore.cc:68
    SQLAtomStorage::removeAtom   SQLAtomDelete.cc:94 (with explicit transaction
                                                       at lines 111-113)
    Value writes                  SQLValues.cc:302-308, 643
    Bulk store                    SQLBulk.cc:265
  This means atomspace-pgres COULD provide a write-path implementation
  reference, BUT it writes to atomspace-pgres's rigid Atom/Value tables
  (per R2.B4), not arbitrary SQL tables, so the implementation is not directly
  reusable for Decko's schema. atomspace-pgres remains BACKING-STORE-ONLY and
  Decko-incompatible.

  Verdict: bridge is INDEPENDENT of atomspace-pgres at the linker level. Phase 4
  cannot piggyback on atomspace-pgres's writer.

Q8. DECKO-ADAPTABILITY (OPTIONS A/B/C/D)
  CONSENSUS [recommendation]: All three reviewers recommend Option D (Decko-
  callback-driven AtomSpace mirror, bypassing bridge) as the primary Phase 4
  (B)+(L1) path, with Option C (extending bridge with full write semantics) as
  fallback only if Source 2 (Hyperon Space write-API) findings show Hyperon
  Space cannot provide usable writes.

  Reasoning convergence:
  1. Bridge is bulk-import-then-read; Phase 4 wants sustained low-latency
     write-through. Architectural mismatch.
  2. Type model can't represent Decko's full schema (timestamps, jsonb dropped).
  3. Zero test coverage; Option C builds writes on a tested-zero base.
  4. Atom-identity model is content-addressed, no PK flagging — Option C must
     extend SignatureLink + add update-vs-insert recognition + add stale-atom GC.
  5. No conflict-detection — Option C must add a sync layer above bridge.

  Source-dependent re-evaluation: if Source 2 finds Hyperon Space is also
  read-only-at-HEAD, Option C becomes the necessary fallback. If Source 4 finds
  MORK supports atom-rate writes natively, Option D might evolve to D' (Decko
  callback → MORK write-path).

Q10/Q14. CONCURRENCY-CEILING (combined)
  CONSENSUS [VERIFIED on architecture]: POOL_SIZE=3 hard-coded at
  BridgeStorage.cc:43; pool is the bottleneck (SQLResponse.h:82-85); per-thread
  connection model. No batching, no prepared statements, no transaction control.
  README scale evidence (basic-demo.scm:215-254) confirms bridge is built for
  bulk-import (250GB FlyBase featureloc, 46K rows/sec sustained read).

  DISSENT V2-Q10/14 (Codex methodological pushback, ACCEPTED): Claude's
  ~1000/sec atom-write ceiling estimate and Gemini's ~1k-2k writes/sec are
  inferences without benchmark evidence. Codex correctly notes "I would not
  lock 'about 1000/sec' as a finding without benchmark evidence." Claude
  retracts the specific number; reframed as qualitative ceiling-order-of-
  magnitude inference for SPEC_DRAFT, with a note that Phase 4 prototype
  benchmarking is needed before hard ceiling assertions.

  Verdict: bridge is [BULK-IMPORT-THEN-READ] [BATCHING-ABSENT] with three
  identifiable bottlenecks (connection-pool size, lack of prepared statements,
  lack of batching/group-commit). Quantitative ceiling: TBD via Phase 4
  prototype benchmarks.

Q11. ATOM-IDENTITY STABILITY (L1 prerequisite)
  CONSENSUS [VERIFIED across all three]: Atoms are content-addressed via
  add_node(TYPE, name) and add_link(TYPE, [outgoing]) (standard AtomSpace
  semantics). For row data: stable across re-reads of the SAME row state, but
  any column-value change creates a NEW atom alongside the old (no GC). PRIMARY
  KEYS ARE NOT FLAGGED in the SignatureLink — all columns are treated as equal
  TypedVariables (SQLResponse.h:140-152).

  L1 round-trip: NOT supported. Agent edits cannot be reliably translated to
  UPDATE SQL because (1) PK columns aren't identified, (2) update-vs-insert
  recognition logic doesn't exist, (3) stale EdgeLinks accumulate without GC.

  Verdict labels: [CONTENT-ADDRESSED-IDENTITY] [PK-COLUMN-NOT-FLAGGED]
  [NO-UPDATE-VS-INSERT-RECOGNITION] [NO-STALE-ATOM-GC].

Q12. CONFLICT-RESOLUTION SEMANTICS
  CONSENSUS [VERIFIED]: Zero version-counter, sync-token, optimistic-lock, MVCC
  interaction, sequence-number, ON CONFLICT, FOR UPDATE/FOR SHARE anywhere in
  the codebase. Currently the model is effectively (b) Sole-Decko-writer (bridge
  reads only); naive Phase 4 extension would put us in (d) Last-writer-wins with
  no detection (silent corruption risk).

  The (B) co-canonical promise is structurally undeliverable on bridge as-is;
  Phase 4 (B) requires either a new sync layer above bridge (advisory locks /
  version columns / change-data-capture) or a different write substrate.

  Verdict label: [NO-CONFLICT-DETECTION].

Q13. SCHEMA-EVOLUTION TOLERANCE
  CONSENSUS [VERIFIED]: get_row_desc (SQLReader.cc:143-157) throws if
  `1 != sigs.size()` (line 151-154). Schema evolution mid-session creates a 2nd
  SignatureLink (because add_link is content-addressed and content differs),
  permanently breaking get_row_desc until cog-bridge-close + cog-bridge-open
  (full reset, which destroys in-flight AtomSpace state).

  Phase 4b durability concern confirmed: every Decko module migration would
  force a bridge restart.

  Verdict label: [SCHEMA-EVOLUTION-BREAKS-RUNTIME].
