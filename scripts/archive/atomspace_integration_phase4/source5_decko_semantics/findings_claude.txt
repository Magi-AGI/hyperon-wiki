Q9. WRITE-PATH TOPOLOGY — IPC CONTRACT GUARD
--------------------------------------------------------------------------------

[VERIFIED] delayed_event.rb:57-65 + :99-106 (read at HEAD; cited
above in Q1) confirm:
  - The integrate_with_delay job runs IN-PROCESS in a Ruby worker
    (Cardio::Job inherits ActiveJob::Base).
  - The job arguments are Ruby objects: card, card_attribs (hash),
    env (Card::Env.serialize), auth (Card::Auth.serialize),
    method_name (symbol).
  - These arguments are SERIALIZED BY ACTIVEJOB into delayed_jobs.
    handler as YAML-encoded ActiveJob state (Ruby-class-and-instance-
    variable structure).

CONSEQUENCE: Codex's V5-PROTOCOL-3 [DELAYED-JOBS-IS-NOT-AN-IPC-
CONTRACT] is ARCHITECTURALLY VERIFIED. A non-Rails sidecar parsing
delayed_jobs.handler would have to:
  (i) parse YAML;
  (ii) decode ActiveJob's "_aj_globalid" / "_aj_serialized" hash
       structure;
  (iii) reconstruct Card / Card::Env / Card::Auth state from
       ActiveJob's representation;
  (iv) track Decko gem version compatibility (ActiveJob serialization
       format is stable per Rails contract but Decko-side serialize
       methods can evolve).
This is exactly the brittleness Codex warned about.

[VERIFIED] Gemini Q9 said "Sidecar via Delayed::Job" without
specifying the wire protocol; the v1 reconciliation correctly notes
Gemini does NOT explicitly conflict with Codex on close reading
(Gemini's MirrorJob.new framing is a Ruby-worker code pattern, not
an "external sidecar reads delayed_jobs.handler" pattern).

RESOLUTION: v1 reconciliation Q9 verdict stands. Architecture =
integrate_with_delay job runs in Ruby worker; worker emits stable
JSON payload to Hyperon sidecar via HTTP / Unix-socket; outbox table
is backup; delayed_jobs is the Rails-side queue, NOT the IPC contract.
