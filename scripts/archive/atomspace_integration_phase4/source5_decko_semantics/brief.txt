This is the DECKO-SIDE source. Sources 1-4 audited the Hyperon /
classical / MORK / DAS / atomspace-rocks substrate landscape and locked
in:  PHASE 4 SUBSTRATE = HYPERON SPACE (in-memory mirror), with
PATCH-1 + PATCH-2 as required code additions, D3-1 metadata as the
SpaceEvent envelope discipline.

Source 5 closes the OTHER half of the Phase 4 architecture: the Decko
side. The Phase 4 mirror service must subscribe to Decko's
write-events, translate Decko card state into Hyperon atoms, and emit
SpaceEvents tagged with D3-1 metadata. To specify that translation
precisely, we need:

  (i)   the canonical Decko event/stage hooks (which fire after the
        DB write commits, and which signal pre-commit);
  (ii)  the canonical Decko card schema (cards / card_acts /
        card_actions / card_changes / card_revisions / card_references
        — what gets persisted, with what granularity);
  (iii) the cardtype primitive inventory (RichText / Pointer / Phrase /
        Date / etc. — and which ones the wiki currently uses);
  (iv)  the +convention layer (compound names like
        `Parent+ai_draft`, `+tag`, `+description`, `+expert_reviewed`,
        `+links`, `+roadmap` — how these encode editorial workflow
        on top of cardtypes);
  (v)   the atom encoding scheme — given a Decko card, what's the
        canonical Hyperon expression? what's the identity policy
        (Decko card_id as primary key vs structural-by-name)?
  (vi)  the D3-1 envelope finalization — what fields, populated from
        which Decko tables/columns, with what semantic meaning;
  (vii) the trash / soft-delete / restore semantics — when does the
        mirror remove an atom vs keep it with a tombstone?
  (viii) the auth/actor layer — how does `Card::Auth.as(...)` flow
        into the D3-1 `agent_id` field?

This source feeds Source 6 (agent-facing MCP surface design) — once
we know the atom encoding, MCP can specify what agent-writable
operations look like — and Source 7 (Magus plan reconciliation) —
which checks D3-1 + atom encoding against the existing Magus 4-phase

Q1. DECKO DIRECTOR PHASE MODEL + CANONICAL EVENT HOOK SELECTION.
    The Phase 4 mirror needs to fire AFTER the Decko/Postgres write
    commits successfully. Decko's 9-stage Director model
    (decko/card/lib/card/director/stages.rb:5-7) gives 3 phases:
    validation, storage, integration.
      (a) Walk the canonical stage list. For each stage, document
          (i) when it fires relative to the DB COMMIT, (ii) whether

Q5. SPACEEVENT GRANULARITY: PER-CARD vs PER-FIELD-CHANGE.
    A Decko card_action can have multiple card_changes (e.g., one
    "rename + retype" act = one action with field=name + field=
    type_id changes). SpaceEvent (Source 2) is enum {Add, Remove,
    Replace} per atom.
      (a) Option-PER-CARD: one card_action → one SpaceEvent.
            For action_type=create: Add(card_atom).
            For action_type=update: Replace(old_atom, new_atom).
            For action_type=delete: Remove(card_atom).
          The mirror reads the FINAL card state after all changes
          are applied and emits one SpaceEvent.
      (b) Option-PER-FIELD: one card_change → one SpaceEvent.
            Each field change emits a Replace where the old/new
            atoms differ ONLY in that field.
          More granular but introduces ordering dependencies and
          intermediate states that are not actually committed.
      (c) Phase 4 implication: Option-PER-CARD is correct.
          Intermediate field states are not real database states.
          Field-level diffs belong in D3-1 envelope, not in
          SpaceEvent enum.
          Predicted answer: SpaceEvent granularity = card_action.
          D3-1 envelope can carry field-level diff list as metadata.

Q6. ATOM ENCODING SCHEME FOR DECKO CARDS — DESIGN.
    Source 5 must propose the canonical encoding. Constraints:
      (i) Identity-stable: card renames must not change the atom
          identity (because rename is a single TRACKED_FIELDS=name
          change action, and PostgreSQL rows keep their id).
      (ii) Structurally-meaningful: cardtype, parent/child via
           left_id/right_id, references should be queryable by
           Hyperon-side reasoning.
      (iii) Round-trippable: reading the atom back can fully
            reconstruct the card (or at least the audit-trail-
            relevant subset).
      (a) Proposed shape (orchestrator low-confidence):
            (DeckoCard
              (Id <numeric-id>)
              (Name "<full+name>")
              (Type <type-codename-or-id>)
              (Content "<db_content>")
              (LeftId <numeric-or-NoLeft>)
              (RightId <numeric-or-NoRight>)
              (Trash <bool>))
          Identity policy: atom-identity is keyed on the (Id N)
          field. The Name field can change across edits without
          breaking identity.
      (b) Alternative (compact): single grounded atom with a struct.
            (DeckoCard <id> <name> <type> <content> <left> <right>
                       <trash>)
          Compact but loses the ability to query by individual
          field via MeTTa pattern matching.
      (c) Alternative (typed): nested expression with (Card (Id ...))
          as the wrapper, carrying typed-leaf atoms.
            (Card
              (Id 7445)
              (Name "About Hyperon+Vision")
              (Type Draft)
              (Content (Markdown "...long content..."))
              (Trash False))
      (d) Phase 4 implication: pick (a) for v1 — explicit-field
          shape is debuggable in MeTTa and pattern-matches cleanly.
          Phase 5+ can revisit.
      (e) The Replace semantics (from PATCH-2 / Source 2):
            replace-atom! old-atom new-atom
          For an update, the mirror calls
            replace-atom!
              (DeckoCard (Id 7445) (Name "...") (Content "old") ...)
              (DeckoCard (Id 7445) (Name "...") (Content "new") ...)
          The Hyperon Space matches by structural equality. This
          requires the OLD atom to be available at update time.
          Mirror service responsibility: cache the previous-state
          atom keyed on id, OR query the Hyperon Space for it
          before emitting Replace.
