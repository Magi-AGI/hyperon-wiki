Q5. SPACEEVENT GRANULARITY.
--------------------------------------------------------------------------------

CONSENSUS [VERIFIED + INFERRED]: Phase 4 mirror emits ONE SpaceEvent per
non-draft Card::Action (NOT per Card::Act, NOT per card_change).

  Codex reasoning:
    - Card::Act has 1:N Card::Action (mod/history/lib/card/act.rb:3-15).
    - Card::Action has super_action_id / sub_actions for subcard
      relationships (mod/history/lib/card/action.rb:29-33).

    - The "atomic unit" of Decko mutation is the Action, not the Act
      or the Change.

  Gemini reasoning: rename emits Replace(Old, New) keyed by stable
                    card_id; mirroring at the Card level is required.

  CONSENSUS event mapping:
    create  → Hyperon Add(new_atom)
    update  → Hyperon Replace(old_atom, new_atom) (with old-state
              caching or Hyperon-side query for old)
    delete  → Hyperon Replace(old_atom, new_atom_with_trash_true)
              under POLICY-B trash (see Q8 reconciliation)

  Codex adds: shared act_id grouping; idempotency keyed by action_id
  (and optionally act_id) so re-runs of the same delayed job do not
  double-emit.

Q8. TRASH SEMANTICS — DISSENT RESOLVED IN FAVOR OF CODEX/POLICY-B.
--------------------------------------------------------------------------------

V5-Q8-X-TRASH-POLICY-DISSENT:

  Brief prediction: POLICY-B (Replace ... Trash=true).
  Codex [CONFIRM POLICY-B]: 5 file:line citations supporting Replace.
  Gemini [PROPOSED Remove(CardAtom)]: ungrounded preference for "lean
                                       mirror"; conflicts with Decko's
                                       actual delete-state model.

  Codex evidence (DECISIVE):
    - Decko identifies delete actions as `trash && trash_changed?`
      (card/lib/card/director/all.rb:19-22). i.e., Decko's "delete"
      IS a trash-flag transition, not a row removal.
    - cards.trash is a non-null boolean column (card/db/schema.rb:92).
      Trashed cards remain in DB.
    - Standard card queries implicitly add `trash is false`
      (card/lib/card/query/sql_statement/where.rb:72-76). Filtering
      is at query-time, not at row-delete time.
    - Reference cleanup treats trashed cards as missing for reference
      ids (card/lib/card/reference.rb:62-68). Application-level
      handling, not DB removal.
    - History revision-before-first-action returns { trash: true }
      (mod/history/set/all/history/revision.rb:21-25). Decko's model:
      nonexistence/precreate ≈ trash state. The atom-presence
      invariant matches POLICY-B.

  Gemini's "lean mirror = Remove" framing is a generic-design heuristic
  not constrained by the Decko data model. POLICY-B (atom-always-present,
  Trash field reflects state) is the faithful mirror choice.

  RESOLUTION: ADOPT POLICY-B.
    - Decko delete (trash=true) → Hyperon Replace(old_atom,
      new_atom_with_trash_true).
    - Decko restore (trash=false) → Hyperon Replace(old_atom_with_trash_true,
      new_atom_with_trash_false).
    - Hard delete / empty-trash maintenance, if performed, is an
      administrative resync path NOT in the Phase 4 event path. (At
      hard-delete the row is gone — the mirror can either emit a
      true Hyperon Remove via a separate admin tool, or rely on a
      backfill reconciliation.)

  MeTTa-side query convention: queries default-filter (Trash False)
  unless explicitly probing trash state. Document in SPEC_DRAFT.md.

VERDICT: [POLICY-B-FAITHFUL-MIRROR-CONFIRMED].

  Codex [REFINE STRONGLY]: do NOT have a non-Rails sidecar parse

                                                  No conflict with Codex
                                                  on close reading.

    (a) Rails delayed event calls the sidecar over local HTTP / Unix
        socket.
    (b) Rails delayed event writes to a small explicit outbox table
        (decko-mirror-outbox or similar; non-card ActiveRecord table
        to avoid recursion).
    (c) Rails delayed event writes a line-delimited JSON queue file
        in a controlled location.

  RESOLUTION: ADOPT (a) HTTP / Unix-socket as primary, (b) outbox
  table as backup if sidecar is unavailable. (c) is operationally
  fragile (file permissions, log rotation interference) — defer
  unless network/socket is unviable.

  Architecture (RECONCILED):

    Decko Card.save!
      → Director phases
      → integrate stage
      → integrate_with_delay job enqueued (ActiveJob → delayed_jobs)

      → Worker constructs DeckoCard atom + DeckoProvenance atom +

  mirror MUST filter them defensively. Cardspace mirror writing
  triggering more cardspace mirror writes is a feedback loop.

  Idempotency (Codex): job retries are normal in delayed_jobs failure
  scenarios. The sidecar MUST treat (event_id) — i.e.,
  "decko:action:<action_id>" — as idempotency key. Re-applying a
  Replace with the same event_id is a no-op.
