Q2. CHOICE OF MCP SERVER STACK.
    Three real candidates after Source 3 ruled out cpp-mcp:
      (a) [RUBY-EXTEND]: extend hyperon-wiki-mcp with new ::MCP::Tool
          subclasses for atom-CRUD. Same gem, same auth, same
          deployment. Agents see one MCP server with both Decko-CRUD
          and atom-query tools.
            Pros: single auth surface; reuses 40-tool registration
                  pattern; same agents already authenticated;
                  deployment is one Rack app.
            Cons: Ruby has no native Hyperon binding (PATCH-1 is
                  pybind11/Python, not Ruby); atom queries must go
                  via Hyperon-side service over HTTP/Unix-socket
                  (echoing Source 5 sidecar IPC); 2 hops for atom
                  reads (agent → Ruby MCP → Hyperon sidecar) where
                  1 hop would suffice.
      (b) [PYTHON-PARALLEL]: new Python MCP server using upstream
          `mcp` SDK or `fastmcp`, in-process with the Hyperon Python
          bindings (PATCH-1 enables `space_register_observer`; the
          Python server binds atoms directly via the existing
          hyperon Python module).
            Pros: 1 hop for atom reads; PATCH-1 already required by
                  Source 2; matches the in-process subscriber model
                  (agent → Python MCP → SpaceMut).
            Cons: separate auth; agents must authenticate twice
                  (one for wiki-CRUD via hyperon-wiki-mcp, one for
                  atom-CRUD via the Python server); two deployment
                  units; auth bridging needed.
      (c) [RUST-PARALLEL]: new Rust MCP server using upstream `rmcp`
          crate, linked into the hyperon Rust crate directly. Skips
          PATCH-1 entirely (Rust-to-Rust SpaceObserver).
            Pros: simplest performance path; no Python intermediary;
                  no PATCH-1 dependency for the atom surface.
            Cons: separate auth (worse than Python — Rust likely
                  needs JWT verification logic re-implemented vs
                  reusing the Ruby auth gems); two languages to
                  maintain; SpaceObserver lifetime/aliasing in Rust
                  is non-trivial (Source 2's Rc/RefCell single-thread
                  invariant must be respected).
      Decision criteria (reviewers choose):
        (i)  PATCH-1 dependency. PYTHON-PARALLEL inherits PATCH-1's
             scope; RUST-PARALLEL avoids it but adds rmcp+JWT cost.
        (ii) Agent authentication overhead. RUBY-EXTEND wins.
        (iii) Deployment complexity. RUBY-EXTEND wins; PARALLEL
              options need a second service.
        (iv) Latency. PYTHON-PARALLEL or RUST-PARALLEL wins (1 hop).
        (v)  Recursion-guard simplicity. Same across choices —
             D3-1 origin-marking handles agent feedback regardless.

      Reviewers must identify the load-bearing constraint and rank.
      Predicted Phase 4 default (orchestrator low-confidence):
      RUBY-EXTEND for Phase 4 read-only (lower deployment cost,
      reuses auth); PYTHON-PARALLEL for Phase 5+ if direct
      SpaceMut writes need to be exposed.

          (Q3) marks the originating agent. The agent's subscription

          per-agent write rate limit (Phase 5+ when writes are

      (d) Mitigation 3 [LOOP-DETECTION]: Phase 5+ heuristic: if an
          agent writes 10 times within 1 second to the same card_id,

          tool-call invocation through to the SpaceEvent metadata
