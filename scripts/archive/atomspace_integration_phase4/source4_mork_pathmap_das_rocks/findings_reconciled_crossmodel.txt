Q2. MORK SPACE TYPE vs HYPERON SPACEMUT
--------------------------------------------------------------------------------

[CONSENSUS] [NOT-A-PHASE-4-SUBSTRATE]. MORK is architecturally distinct
from Hyperon SpaceMut.

Verified divergences:
  - MORK uses concrete `impl Space` blocks; no trait abstraction for
    mutation.
  - MORK's `Expr` is a raw byte-pointer/buffer-based representation
    (Gemini cites space.rs:450; not re-verified).
  - Hyperon Space uses an atom-oriented `Atom` enum.
  - Identity translation: structural Hyperon atoms <-> MORK path-based
    trie keys would require a custom mapper.
  - Observer translation: Hyperon's per-instance SpaceObserver vs MORK's
    global `static mut` counters would require an emit-on-write shim
    inside any wrapper.

Phase 4 implication (consensus):
  - Adapter cost is significant. Phase 4 will not absorb it.
  - Phase 5+ MORK-PATH would require a fresh adapter project (NOT
    mork_ffi, which is a reference checkpoint per Q4).
