Q2. CLASSICAL HANDLE/UUID IDENTITY MODEL.
--------------------------------------------------------------------------------

[CO-VERIFIED] Classical Handle is std::shared_ptr<Atom>-derived; pointer
identity for ==, content identity for <.

  - Handle.h:51   `typedef uint64_t ContentHash;`
  - Handle.h:54   `typedef std::shared_ptr<Atom> AtomPtr;`
  - Handle.h:57   `class Handle : public AtomPtr`
  - Handle.h:69   `static const ContentHash INVALID_HASH = std::numeric_limits<size_t>::max();`
  - Handle.h:125  `inline bool operator==(const Handle& h) const noexcept {`
  - Handle.h:126     `    return get() == h.get();`           (POINTER ==)
  - Handle.h:132-134 doc: "Defer to Atom::operator< thus is content based."
  - Handle.h:135  `bool operator< (const Handle& h) const noexcept;`
                                                              (CONTENT <)

[INFERRED] Two Handles to atoms with identical content but allocated
separately compare unequal under operator==. Content identity is via
Atom::operator< (or get_hash via the Atom layer per Codex's Atom.h:340/443/510-514
citations). UUID is a STORAGE-LAYER concept (TLB), not a CORE-LAYER concept.

[CO-VERIFIED] Persistent UUID lives in atomspace-pgres SQLUUID.cc per Codex
(SQLUUID.cc:37-51, :77-117, :211-231). The TLB (Transaction Lookaside Buffer)
maps RAM Handle ↔ DB UUID; not a property of Handle itself.

VERDICT: [POINTER-IDENTITY-AT-RAM] [CONTENT-HASH-AT-DB] [UUID-IS-STORAGE-CONCERN]

Phase 4 implication: do NOT borrow classical UUID into the Phase 4 mirror.
Decko already provides stable card_id / action_id; encode those as atom
metadata (Source 5 territory).
