          signature (file:line).
      (b) Compare to atomspace-bridge (Source 1): bridge had storeAtom/
          removeAtom/storeValue/updateValue/storeAtomSpace as empty stubs
          at SQLReader.cc:395-425. atomspace-pgres claim: REAL impls. Verify.
      (c) Transaction model: BEGIN/COMMIT/ROLLBACK present? Prepared

          atomspace-pgres has NO knowledge of Decko's card_actions /
          card_changes / card_revisions semantics — those come from the
          Decko Rails layer. Even if write-capable, atomspace-pgres
          replaces ONLY the Hyperon-side persistence, NOT Decko durability.

Q2. CLASSICAL ATOMSPACE IDENTITY MODEL.
    Source 2 Q7 found Hyperon Space has structural-only identity (no UUID,
    no stable handle). Classical AtomSpace has Handle (a `std::shared_ptr`-
    like type to Atom) — does this provide stable identity that Phase 4
    could borrow?
      (a) Audit opencog/atoms/base/Handle.h. Is Handle a UUID-like stable
          identifier, or is it a pointer-like dynamic reference?
      (b) Audit opencog/atomspace/AtomTable.h. Does AtomTable maintain a
          UUID-to-Atom registry? Is the UUID stable across process restarts
          (i.e., persistence-aware)?
      (c) atomspace-pgres SQLUUID.cc — is the UUID-to-Atom binding stored
          in Postgres? If yes, Phase 4 could optionally borrow this for
          Decko round-trip stability.
      (d) Phase 4 implication: if classical AtomSpace's Handle/UUID is
          stable AND borrowable, it could enable a Phase 4 extension where
          Decko cards have stable atom IDs across mirror-service restarts.
          If not, Phase 4 must continue using Decko's card_id as the
          identity carrier.
