Bottom line
-----------

[STAY-HYPERON-SPACE]

Source 3 refines, but does not override, Source 2. Phase 4 should stay on
Hyperon Space / SpaceMut / SpaceObserver as the write-side mirror substrate.
Classical AtomSpace and atomspace-pgres contribute useful design constraints
for durability, identity, and future L1 introspection, but they are not a
better immediate substrate for Decko-to-Hyperon write-back.

Recommended Phase 4 stance:

- Durability authority: Decko card/action tables, not atomspace-pgres.
- Mirror write API: Hyperon SpaceMut add/remove/replace.
- Mirror identity: Decko card_id / action metadata encoded in atoms or side
  metadata, not classical Handle/UUID grafted into Phase 4.
- Agent eventing: SpaceObserver -> event bus / MCP surface after the Source 2
  Python observer binding gap is closed.
- MCP implementation: language-matched Python/Rust service. cpp-mcp is only
  relevant if a future service is explicitly C++.

[VERIFIED-PERSISTENT-UUID-LIVES-IN-STORAGE-LAYER]

Evidence:

Q12. synthesis / recommendation
-------------------------------

Answers to the Source 3 decision questions:

- Q12.a Switch to classical AtomSpace + pgres? No. Stay Hyperon Space.
  Classical pgres is a real writer, but deprecated, classical-only, and not
  Decko-aware.

- Q12.b Borrow Linas's identity model? No for Phase 4. Use Decko card_id/action
  identity. Classical UUIDs are StorageNode/TLB persistence details; core
  Handle is not a durable UUID.

- Q12.c Borrow Linas's write-back semantics? Only as design precedent.
  StorageNode store/remove/fetch shows the shape of durable APIs, but Phase 4
  should stay with SpaceMut add/remove/replace plus a clean event envelope.

- Q12.d Use cpp-mcp? No unless the service is C++. Use a Python or Rust MCP
  stack aligned with the mirror/event service language.

Proposed Phase 4 refinement:

1. Decko remains the durable source of truth and emits card/action events.
2. Mirror worker resolves each event into `{action_type, card_id, before_atom,
   after_atom, action_id, timestamp}`.
3. Hyperon Space update:
   - create -> `space.add(after_atom)`
   - update -> `space.replace(before_atom, after_atom)`
   - delete -> `space.remove(before_atom)`
4. SpaceObserver receives the event synchronously after Source 2 PATCH-1
   exposes observer registration to Python or an equivalent Rust/C bridge.
5. Agent-facing MCP publishes resources/tools over the mirror/event state in
   the implementation language of the service.

Residual risks / next-source handoff
------------------------------------

- Phase 4 still needs an explicit canonical atom encoding for Decko `card_id`
  and action identity.
- Phase 4 still needs the Source 2 Python observer binding (`space_register_observer`)
  or a Rust/C event-forwarder.
- No throughput ceilings should be claimed from Source 3. pgres docs provide
  historical performance numbers, but they are deprecated-backend numbers, not
  benchmarks for the Phase 4 mirror.
- Source 5 should decide whether identity metadata lives inside atoms, in a
  side table, or both.
