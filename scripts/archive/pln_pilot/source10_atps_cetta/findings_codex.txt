1. ai-agents top-level architecture

C-1-1. ai-agents is not a narrow PLN repository. Its top level includes AIrxiv, atps, c-projects, hyperon, isabelle, lean-projects, megalodon, metta-tools, mizar-projects, private-ref, Ramsey36, tests, and tools. PLN-relevant material is concentrated in atps/ and lean-projects/mettapedia/.

C-1-2. The atps/ directory is the adjacent executable line for PLN premise selection. It contains Python selectors and evaluation scripts, and its README explicitly frames the work as "Premise selection experiments using Probabilistic Logic Networks on the extended MPTP 5k dataset."

C-1-3. lean-projects/mettapedia/ is the formal-proof line. It contains the PLN-kNN-NB paper source, theorem modules, conformance artifacts, and many additional .tex papers.

C-1-4. ai-agents/hyperon/ is not the cited PeTTa tree. In this checkout the cited hyperon/PeTTa/demos/ path is absent; the brief's scope check found only hyperon/metamath/ and hyperon/translators/.

C-1-5. This repo should be described as a multi-formalism benchmark/formalization workspace by Zarathustra Goertzel, not as a production Hyperon PLN repo.

3. lean-projects/mettapedia Lean theorems

C-3-1. The headline NB theorem is real. Mettapedia/Logic/PLNBayesNetInference.lean:290 defines:
`theorem toStrength_nbEvidence_eq_nbPosterior`
with statement that BinaryEvidence.toStrength of nbEvidence equals nbPosterior over prior and likelihood evidence. The proof body at lines 296-298 is `classical`, `unfold`, and `simp`.

C-3-2. The headline kNN theorem is real. Mettapedia/Logic/PremiseSelectionKNN_PLNBridge.lean:125 defines:
`theorem PLN_hplusPos_eq_knnRelevance`
with statement that `(plnKnnEvidence ...).pos = knnRelevanceENN ...`. The proof at line 130 delegates to `plnKnnEvidence_pos_eq_knnRelevanceENN`.

C-3-3. A targeted search for `sorry` in the key theorem files PLNBayesNetInference.lean, PremiseSelectionKNN_PLNBridge.lean, and PremiseSelectionBestPLNDraft.lean returned no matches. This does not prove the whole Lean tree is sorry-free, but the two headline bridges are not placeholder theorems.

C-3-4. EvidenceClass.lean declares the abstract evidence type-class layer. It states the key insight that PLN quantale structure is on evidence, not truth values, and defines EvidenceType at line 53, SequentialEvidence at line 61, InterpretableEvidence at line 72, and BinaryContext at line 99.

C-3-5. EvidenceKind.lean defines an EvidenceKind enum at line 34 for epistemic classification of evidence sources, including structured expert judgment and model-derived evidence.

C-3-6. EvidenceModel.lean is an applied SUMO repair evidence model. It defines EvidenceAtom at line 131 and BinEvNat at line 289, tying heterogeneous source evidence into binary evidence counts.

C-3-7. PLNWorldModelGeneric.lean generalizes the binary world-model interface. It defines AdditiveWorldModel at line 38, extract_add' at line 50, GMQueryEq at line 57, queryObservationCount at line 85, and bridges the generic interface back to BinaryWorldModel at line 119.

C-3-8. PainEvidenceWM.lean is a concrete evidence-driven ontology repair proof of concept. It defines SourceItem at line 60, painEvidence at line 66, aggregate at line 124, multiple by-decide ranking theorems at lines 134-190, and an AdditiveWorldModel instance at line 262.

C-3-9. The Lean side is not just paper prose. It contains formal theorem statements and domain-specific proof artifacts that are linked to the PLN-kNN-NB claims.

10. Resolution: IMPLEMENTED vs PROTOTYPE

Source 10 verifies substantial real implementation: pure-Python mixture selector, kNN/NB Python drivers, MaSh baselines, Lean bridge theorems, conformance artifacts, CeTTa evidence runtime, and biomedical WM-PLN tests.

C-10-3. The best label is PROTOTYPE-ALMOST-COMPLETE-WITH-GAP:
- present: Python orchestration, key Lean proofs, CeTTa evidence runtime, CeTTa PLN runtime, conformance intent, benchmark claims.
- missing: lib_pln_xi.metta, PeTTa demos selectors, cited PeTTa pln_evidence path, dataset/deps artifacts for reproducing reported numbers.

C-10-4. For wiki status fields, avoid both "fully implemented" and "paper-only." Use: "adjacent prototype with verified Python + Lean + CeTTa components, but missing cited PeTTa selector layer and unreproducible local benchmark inputs."
