Summary
- MORK is a high-performance hypergraph execution kernel, not a checked-in FactorGraph or probabilistic-inference engine.
- The repo provides a real executable substrate for `.mm2` programs and benchmark workloads, including the `chaining` Metamath benchmark interface.
- The PLN FactorGraph / MORK paper stream therefore appears to be a code-to-paper proposal bridge, not an already-realized implementation in this repo.
- MORK belongs to a distinct contributor ecosystem from the Geisweiller-centered PLN line; the connection is inter-ecosystem benchmarking/proposal, not a single unified implementation lineage.

1. Architecture
- The codebase is organized as a Rust workspace with `kernel/` as the default member, plus `expr/`, `frontend/`, `interning/`, and several `experiments/*` crates.
- `kernel/src/space.rs` is the architectural center. Its `Space` type combines:
  - a `PathMap`-backed storage surface,
  - shared symbol mapping,
  - zipper-driven expression traversal,
  - interpretation and execution loops over `exec`-style expressions,
  - optional integrations for JSON transcription, z3, wasm, and sink/source plumbing.
- The architecture reads as a graph/term execution substrate:
  - hypergraph-like storage and retrieval,
  - zipper-based structural traversal,
  - unification and substitution over expressions,
  - MeTTa-calculus execution via `metta_calculus`.
- `kernel/src/main.rs` exposes the repo as a CLI runtime rather than a library-only component. The core subcommands are `bench`, `test`, `run`, and `convert`.
- `expr/src/lib.rs` is where low-level expression and unification logic lives. This supports the claim that MORK’s core competency is fast symbolic structure processing rather than domain-specific probabilistic inference.

2. FactorGraph audit
- I found no checked-in FactorGraph implementation surface in this repo.
- Repository-wide searches turned up no meaningful code for:
  - `FactorGraph`
  - message passing / belief propagation
  - marginalization / posterior / prior inference pipelines
  - explicit probabilistic graphical model engines
- This matches the brief’s pre-check: MORK currently has no FactorGraph code.
- The strongest conclusion is negative but important:
  - MORK may be a plausible target substrate for the FactorGraph/MORK research direction,
  - but that direction is not realized here in code at this commit.
- This means the wiki should not present MORK as already implementing the FactorGraph machinery proposed in the PLN FactorGraph/MORK papers.

3. `.mm2` format and chaining benchmark interface
- `.mm2` is a real executable program format in this repo, not just an archival artifact.
- `kernel/resources/` contains multiple checked-in `.mm2` programs, including:
  - `ancestor.mm2`
  - `counter_machine_5.mm2`
  - `decision_tree_learning.mm2`
  - `decision_tree_learning_without_min_sink.mm2`
  - `grounding.mm2`
  - `ip_sudoku.mm2`
  - `odd_even_sort.mm2`
  - `std.mm2`
  - `string_convert.mm2`
  - `transitive.mm2`
  - `zip_add.mm2`
- `kernel/src/main.rs` includes explicit `.mm2`-focused routines and comments such as `work_mm2_run`, `mm2_bc`, and `mm2_bc_v3`, including comments about “MM2 inference control”.
- The `Run` and `Convert` command paths show that MORK can ingest, execute, and transform these program representations directly.
- The `chaining` benchmark interface is real:
  - `chaining/experimental/metamath/benchmarks/benchmark.sh` invokes `MORK/target/release/mork run ...`
  - this makes MORK an external runtime target for benchmarked `.mm2` workloads.
- So the repo materially closes one part of the earlier paper/code gap:
  - there is real MORK code in the cluster,
  - and it is used from another cluster repo,
  - but the realized surface is execution/runtime infrastructure, not FactorGraph PLN.

4. `experiments/` deep dive
- The `experiments/` subtree is substantial and meaningful, but it does not change the FactorGraph conclusion.
- `experiments/eval`
  - provides a `no_std` / nightly evaluation engine with function registry, stack-frame machinery, evaluation scope, and quoting/pure-eval support.
  - This looks like experimental execution infrastructure, not probabilistic graph inference.
- `experiments/eval-ffi`
  - exposes a small FFI layer around the experimental eval stack.
- `experiments/eval-examples`
  - contains small example exported functions like arithmetic helpers, showing how the eval machinery can be embedded or extended.
- `experiments/unification_test_laws`
  - is especially important for recent development direction.
  - It compares MORK unification against SWI-Prolog and diffs results, which aligns with the brief’s note that PR #49 (“Unification laws test”) is the latest merged work.
- Net interpretation:
  - experiments focus on evaluation, embedding, and unification correctness,
  - not on turning MORK into a probabilistic inference engine.

- Taken together, `kernel/resources/` presents MORK as a programmable symbolic execution platform with benchmark/demo workloads.

7. PathMap dependency
- `PathMap` is a hard architectural dependency and a practical build blocker in this environment.
- The root `Cargo.toml` references `pathmap = { path = "../PathMap/" ... }`.
- `kernel/src/space.rs`, `kernel/src/sinks.rs`, and related code use `PathMap` pervasively.
- This is not an optional integration; it is a core substrate dependency.
- As a result:
  - I did not run a local build,
  - and any future reproduction/build instructions in the wiki or repo docs should clearly state that MORK is not self-contained without the sibling `PathMap` checkout.
