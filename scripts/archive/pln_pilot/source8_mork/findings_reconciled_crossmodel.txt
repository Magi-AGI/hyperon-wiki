## 3. The `.mm2` format — what it actually is

**[X1 Claude — DEFINITIVE FINDING]** `.mm2` files in MORK are **NOT Metamath**. They are **MeTTa S-expressions with `(exec <loc> <pat> <tpl>)` rewrite-rule syntax**. `mork run X.mm2` just parses S-expressions and calls `metta_calculus(steps)`. **No `.mm2`-specific parser exists; the file extension is naming convention only.**

This **corrects a likely wiki misreading**: the chaining repo's `experimental/metamath/` directory uses MORK to run `.mm2` files for the Metamath propositional calculus benchmarks. One could naturally infer "`.mm2` is Metamath 2 format." That inference is wrong. The `.mm2` files happen to encode Metamath proofs as MeTTa rewrite rules, but the format itself is generic MeTTa.

**[X1 Codex agreement]** Codex independently noted: "`.mm2` is a real executable program format in MORK, not just a benchmark artifact." Both agents agree the format has runtime semantics; it's how you write MORK programs.

**[X1 Claude — additional evidence]** `kernel/resources/decision_tree_learning.mm2` (**549 lines, by Remy Clarke**) is a **complete Gini-impurity decision-tree learner** written entirely in MM2 rewrite rules — including type-checking, macro-expansion stages, `pure`-sink f64 arithmetic, and prediction verification. **Demonstrates MM2 is a usable Turing-complete-ish programming language inside MORK, not just a config format.** This significantly upgrades how the wiki should characterize MORK: it's not just a hypergraph DB + zipper VM; it's a *programming environment* with non-trivial worked programs.

## 4. Cross-model unanimous findings

### 4.1 [X3] What MORK actually IS (in code terms)

- **High-performance hypergraph processing kernel** (CLAUDE.md / README explicit).
- **Rust workspace** (Edition 2024, nightly required) with 7+ crates: `kernel/`, `expr/`, `frontend/`, `interning/`, `experiments/eval/`, `experiments/eval-ffi/`, `experiments/eval-examples/` (dylib), `experiments/unification_test_laws/`.
- **CLI subcommands**: `bench`, `test`, `run`, `convert` (per `kernel/src/main.rs`).
- **Multiple frontend parsers**: CZ2, CZ3, HE (Hyperon Experimental), Rosetta, bytestring, JSON.
- **Triemap-based graph database** with relational algebra (union, intersection, subtraction). Subtraction is being added in PR #50 (open).
- **External hard dependency**: PathMap at `../PathMap/` (NOT cloned locally; build blocker).

### 4.2 [X3] What MORK is NOT

- **NOT a probabilistic reasoner.** No TruthValue, no Beta posteriors, no Bayes operations.
- **NOT a FactorGraph engine.** No factor-graph data structures, no message-passing protocols, no marginalization machinery.
- **NOT a tensor framework.** PR #48 (sparse tensor primitives + einsum) was **CLOSED, not merged** — explicit scope rejection by maintainers.
- **NOT a benchmark-only artifact.** Despite the `chaining/experimental/metamath/benchmarks/` integration that brought MORK into the cluster pilot, MORK is a real engine with worked programs (decision trees, unification tests, etc.).

### 4.3 [X3] Strategic-scope signal: PR #48 closure

PR #48 by Igor Malovitsa proposed *"Add sparse tensor primitives with einsum contraction"*. **CLOSED, not merged.** All three models flag this as the strategic-scope signal: maintainers explicitly chose NOT to broaden MORK into tensor/numeric/numerical-linear-algebra territory. **MORK is intentionally a symbolic hypergraph kernel** — even though the FactorGraph papers might naturally invite tensor extensions, the team rejected that direction.

## 5. Performance characterization (X1 Gemini, useful for wiki)

**Wiki implication**: when describing MORK's performance, the answer is workload-dependent. Full FC: dramatic speedup. Sparse BC: not yet competitive. Don't over-generalize the "millions of times" claim; it depends on traversal pattern.

## 8. Wiki implications

### 8.1 The FactorGraph MORK paper series (Source 9 priority)

Now that we know MORK is a substrate, the paper series can be characterized precisely. Suggested wiki framing:

> *"The PLN FactorGraph MORK paper series proposes protocols for implementing factor-graph reasoning on top of MORK's hypergraph processing primitives (zipper traversals, Triemap-based graph DB, multi-threaded VM). MORK itself does not currently implement these protocols. Papers should be read as design-stage proposals rather than documentation of existing MORK functionality. Cross-ecosystem implementation work (PLN side: Geisweiller; MORK side: Vandervorst) is the path forward; no integrated implementation exists today."*

### 8.2 Specific wiki updates from this source

1. **MORK is in the cluster** but in a **separate ecosystem** with a different team. Wiki should note this.
2. **`.mm2` files are MeTTa S-expressions**, not Metamath. Wiki must not conflate "Metamath benchmarks via .mm2 files" with "Metamath format support."
3. **Performance is workload-dependent**: forward-chaining-friendly, sparse backward chaining still less efficient. The "millions of times speedup" claim is asymptotic / best-case.
4. **PR #48 closure documents intent**: MORK is symbolic, not numerical. Tensor / einsum / sparse-tensor extensions are explicitly out of scope.
5. **`distr-combinators` dependency unused** — another "dead infrastructure" signal. Probabilistic primitives are imported as deps but not wired in. Same pattern as chaining's unused `BetaDistribution`.
6. **MORK has the substrate primitives** that *could* implement probabilistic / FactorGraph reasoning — but the PLN cluster team (Geisweiller et al.) is not the team building MORK, and the MORK team has explicitly de-prioritized numerical extensions.

### 8.3 Per-repo orientation doc refresh
- `MORK/CLAUDE.md`: surgical update — ZAM is documentation-only (actual zippers in pathmap), `.mm2` is MeTTa-S-expr-with-(exec), decision_tree_learning demonstrates MM2-as-language, no FactorGraph code, PR #48 closure as scope signal.
