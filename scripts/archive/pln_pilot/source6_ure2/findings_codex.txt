§1 - Architecture map and core C++ surface

C-1-1. `opencog/ure/` is a self-contained C++ rule-engine surface with explicit control, rule, forward-chaining, backward-chaining, Scheme glue, and type-definition modules, not just a thin wrapper around AtomSpace APIs. Evidence: top-level files `ActionSelection.*`, `BetaDistribution.*`, `MixtureModel.*`, `Rule.*`, `ThompsonSampling.*`, `UREConfig.*`, `URESCM.cc`, `Utils.*`, plus `backwardchainer/`, `forwardchainer/`, and `types/`. [opencog/ure/]

C-1-2. The backward-chaining side is architecturally first-class: it has dedicated `BackwardChainer`, `BIT`, `ControlPolicy`, `Fitness`, and `TraceRecorder` units, which means the search/control model is implemented in code rather than left to external scripts. [opencog/ure/backwardchainer/BackwardChainer.h; opencog/ure/backwardchainer/BIT.h; opencog/ure/backwardchainer/ControlPolicy.h]

C-1-3. The forward-chaining side is also first-class but simpler in its public structure: `ForwardChainer`, `SourceSet`, `SourceRuleSet`, and `FCStat` organize source/rule population and inference tracing. [opencog/ure/forwardchainer/ForwardChainer.h]

C-1-4. URE remains natively Atomese/AtomSpace-oriented: rule application is expressed over `Handle`, `BindLink`, `ExecutionOutputLink`, `EvaluationLink`, and `AtomSpace`, not MeTTa modules or standalone mathematical kernels. [opencog/ure/Rule.h; opencog/ure/backwardchainer/BIT.h; opencog/ure/URESCM.cc]

C-1-5. The core Bayesian/control files are all explicitly authored by Nil Geisweiller, which is strong local evidence that Geisweiller, not just Linas Vepstas, designed the advanced inference-control layer. Evidence: authorship headers on `ActionSelection.h`, `ThompsonSampling.h`, `BetaDistribution.h`, `MixtureModel.h`, `ControlPolicy.h`, `Fitness.h`, and `BIT.h/cc`. [opencog/ure/ActionSelection.h; opencog/ure/ThompsonSampling.h; opencog/ure/BetaDistribution.h; opencog/ure/MixtureModel.h; opencog/ure/backwardchainer/ControlPolicy.h; opencog/ure/backwardchainer/Fitness.h; opencog/ure/backwardchainer/BIT.h]

C-1-6. This repo is a legacy engine with repo-local orientation docs already acknowledging that status: local `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md` all describe `ure2` as a mirror/legacy copy rather than an active current implementation line. [AGENTS.md; CLAUDE.md; GEMINI.md]

§2 - Inference control: MAB / ControlPolicy / ActionSelection

C-2-1. URE implements explicit Bayesian action selection, not just ad hoc heuristic weighting. `ActionSelection` stores `HandleTVMap action2tv`, a `TruthValueSeq`, and a `ThompsonSampling` instance, and its docstring defines the action probability as `Pi = ∫ pdf_i(x) * Π_j!=i cdf_j(x) dx / nt`. [opencog/ure/ActionSelection.h, class `ActionSelection`; opencog/ure/ActionSelection.cc]

C-2-2. `ThompsonSampling` is the arm-selection algorithm, and the code confirms it uses beta-posteriors rather than softmax over scalar scores. `ThompsonSampling::distribution()` builds per-action CDFs from `BetaDistribution(tv).cdf(_bins)` and approximates the selection weights; `ThompsonSampling::operator()` samples a posterior draw for each arm and picks among maxima. [opencog/ure/ThompsonSampling.h, class `ThompsonSampling`; opencog/ure/ThompsonSampling.cc, `distribution()` and `operator()`]

C-2-3. `BetaDistribution` is the TV-to-posterior bridge. It maps a `TruthValuePtr` to `(alpha, beta)` using `tv->get_mean() * tv->get_count()` and exposes `mean()`, `variance()`, `pdf()`, `cdf()`, and posterior sampling via `ibeta_inv`. [opencog/ure/BetaDistribution.h; opencog/ure/BetaDistribution.cc, constructors and `operator()`, `mean()`, `variance()`]

C-2-4. `MixtureModel` goes beyond arm selection and tries to infer the truth value of a control decision from multiple partial models. Its comments explicitly frame this as a Universal Operator Induction-style mixture with complexity priors and a Kolmogorov-style estimate `K(D) = |D|^(1-c)` for unexplained data. [opencog/ure/MixtureModel.h, class comment and `operator()()`; opencog/ure/MixtureModel.cc, `prior_estimate()`, `kolmogorov_estimate()`, `weighted_average()`]

C-2-5. `ControlPolicy` is the control-rule orchestrator for backward chaining. It takes `UREConfig`, `BIT`, `target`, and optional `control_as`; computes valid rules for a selected `AndBIT`/leaf; queries active expansion-control rules; builds `MixtureModel` TVs when such rules exist; and otherwise falls back to default rule TVs. [opencog/ure/backwardchainer/ControlPolicy.h; opencog/ure/backwardchainer/ControlPolicy.cc, constructor, `get_valid_rules()`, `expansion_success_tvs()`, `rule_weights()`]

C-2-6. The control rules are themselves Atomese patterns over inference-tree state, not hard-coded policy branches. `ControlPolicy` looks for implication-scope rules whose antecedent contains `Evaluation(Predicate "URE:BC:preproof-of", ...)` plus an `Execution` of `TraceRecorder::expand_andbit_schema_name`, optionally with extra pattern clauses. [opencog/ure/backwardchainer/ControlPolicy.h, comments on `get_antecedent_preproof()`, `get_expansion()`, `fetch_expansion_control_rules()`; opencog/ure/backwardchainer/ControlPolicy.cc, `mk_expansion_control_rules_query()`]

C-2-7. Backward search control is richer than a plain BIT plus random rule choice. `BackwardChainer::expand_bit()` selects an `AndBIT`, then a `BITNode` leaf, then calls `_control.select_rule(andbit, *bitleaf)` to get both a rule instance and a probability-of-success estimate used in expansion complexity and tracing. [opencog/ure/backwardchainer/BackwardChainer.cc, `expand_bit()`]

C-2-8. The repo therefore contains a materially richer inference-control surface than a simple confidence-ranked queue: explicit BIT state, per-leaf fitness, Thompson-sampled rule weights, mixture-modeled control-rule success TVs, and complexity-aware pruning. Even before any cross-repo comparison, those components are all visible in the local C++ path. [opencog/ure/backwardchainer/BIT.h; opencog/ure/backwardchainer/Fitness.h; opencog/ure/backwardchainer/ControlPolicy.cc; opencog/ure/ActionSelection.cc; opencog/ure/MixtureModel.cc]

§3 - Rule representation and rule families

C-3-1. URE's `Rule` abstraction is generic Atomese, not a built-in PLN formula table. A rule is structurally a `BindLink` whose body contains premises/clauses and whose implicand is an `ExecutionOutputLink`; the header comments explain variable declarations, premises, conclusions, and `ExecutionOutputLink` wrapping. [opencog/ure/Rule.h, class `Rule` and rule-format comments]

C-3-2. Rules are attached to rule bases as weighted members. The header comment and implementation describe `MemberLink <TV> <rule alias> <rbs>`, and `ControlPolicy` later uses those member TVs as default rule priors. [opencog/ure/Rule.h; opencog/ure/backwardchainer/ControlPolicy.cc, constructor]

C-3-3. Meta-rules are part of the intended surface. `RuleSet::expand_meta_rules()` instantiates derived rules from meta-rules, and both backward and forward chainer paths invoke meta-rule expansion as a temporary hack before full native support. [opencog/ure/Rule.cc, `RuleSet::expand_meta_rules()`; opencog/ure/backwardchainer/BackwardChainer.cc, `expand_meta_rules()`; opencog/ure/forwardchainer/ForwardChainer.cc, `expand_meta_rules()`]

C-3-4. The checked-in local rule/test surface is broader than the recent MeTTa PLN repos. Under `tests/ure/rules/` there are files for deduction, modus ponens, fuzzy conjunction introduction, implication instantiation, implication introduction, implication-scope direct evaluation, implication-and-lambda factorization, conditional direct evaluation, and intensional inheritance direct introduction. [tests/ure/rules/*.scm]

C-3-5. That breadth is mostly expressed as external Scheme/Atomese rule files, not specialized C++ rule classes. The C++ engine is generic; the concrete inference families live in SCM rule definitions and test fixtures. [opencog/ure/Rule.h; tests/ure/rules/*.scm; opencog/ure/README.md]

C-3-6. I found no checked-in local evidence of FactorGraph, MORK, temporal/causal rule families, or distributional-truth-value implementations in this repo. The engine is generic enough to host more families, but the visible local rule surface is centered on classical PLN/OpenCog families and utility rules. [repo-wide read of `opencog/ure/`, `tests/ure/rules/`, `tests/ure/backwardchainer/scm/`, `tests/ure/forwardchainer/scm/`]

§4 - `forwardchainer/` and `backwardchainer/` deep dive

C-4-1. The BIT data structure is fully implemented here. `BITNode` stores a target/body, fitness, available rule instantiations, complexity, and exhaustion status; `AndBIT` stores an FCS (`BindLink`) plus leaf-to-BIT-node mapping and expansion logic; `BIT` stores a sorted population of `AndBIT`s. [opencog/ure/backwardchainer/BIT.h]

C-4-2. Search complexity is explicit and probabilistic. `BITNode::complexity` is initialized as `-log(operator()())`; `AndBIT::expand_complexity()` adds parent complexity, selected leaf complexity, and `1 - log(prob)` from the selected rule's success estimate. [opencog/ure/backwardchainer/BIT.cc, `BITNode::BITNode`, `BITNode::operator()`, `AndBIT::expand_complexity()`]

C-4-3. Backward chaining is a three-phase loop: expand BIT, fulfill BIT, reduce BIT. `BackwardChainer::do_step()` calls `expand_bit()`, `fulfill_bit()`, and `reduce_bit()` each iteration. [opencog/ure/backwardchainer/BackwardChainer.cc, `do_step()`]

C-4-4. `fulfill_bit()` runs selected FCS programs in a temporary child atomspace and writes back inferred results to the knowledge base, with a warning in comments that this may unintentionally mutate TVs of pre-existing atoms. That is a nontrivial operational caveat, not just a missing polish item. [opencog/ure/backwardchainer/BackwardChainer.cc, `fulfill_fcs()`]

C-4-5. BIT reduction is probabilistic and explicitly tied to inference-control assumptions. `remove_unlikely_expandable_andbit()` computes each tree's probability of never being expanded for the remaining iterations and removes trees according to that distribution. [opencog/ure/backwardchainer/BackwardChainer.cc, `remove_unlikely_expandable_andbit()`]

C-4-6. Forward chaining also uses Thompson sampling, but in a narrower way. It maintains `SourceSet` / `SourceRuleSet`, supports an alternative "source rule producer implementation" (`_srpi`), Thompson-samples among valid rules for a source, and computes source-rule TVs by scaling beta means/variances by source weight. [opencog/ure/forwardchainer/ForwardChainer.h; opencog/ure/forwardchainer/ForwardChainer.cc, `mk_source_rule()`, `select_rule()`, `calculate_source_rule_tv()`]

C-4-7. The forward-chainer docs are themselves unfinished. `opencog/ure/forwardchainer/README.md` says it is "utterly a draft" and contains only a tiny specification note about `SetLink` sources, which matches the code's relative lack of documented high-level control policy compared with the backward chainer. [opencog/ure/forwardchainer/README.md]

§10 - Lineage and successor map

C-10-1. URE's enduring contribution is architectural, not just algorithmic: it combines a generic Atomese rule engine with explicit search objects (`BIT`, `AndBIT`, `BITNode`), explicit Bayesian control (`ThompsonSampling`, `BetaDistribution`, `MixtureModel`), and configurable control rules in the AtomSpace. Those are the big lineage-bearing ideas in this repo. [opencog/ure/backwardchainer/BIT.h; opencog/ure/ActionSelection.h; opencog/ure/MixtureModel.h; opencog/ure/backwardchainer/ControlPolicy.h]

C-10-2. The most plausible survivors into later systems are conceptual rather than code-level: rule application as search, probability-weighted rule selection, and the distinction between structure-level control policy and individual rule formulas. The concrete C++/Scheme implementation here is tightly coupled to AtomSpace handles and SCM glue, which makes straight code carryover unlikely. [opencog/ure/Rule.h; opencog/ure/URESCM.cc; opencog/ure/backwardchainer/ControlPolicy.h]

C-10-3. The repo also shows what was easy to abandon: the entire BIT/control-rule/MAB stack is concentrated in specialized C++ modules and disabled tests, which makes it much easier for successors to retain "rule firing" while dropping the full inference-control layer. That abandonment risk is visible locally even without reading successor repos. [opencog/ure/backwardchainer/*.h/*.cc; tests/ure/CMakeLists.txt; README.md]

C-10-4. The local evidence strongly supports Claude's broader cluster interpretation that URE went beyond the 2008 book on inference control rather than merely implementing it. The code contains Bayesian Thompson sampling, a mixture-model over active control rules, and probabilistic BIT pruning machinery that are research extensions in their own right, not just a direct transcription of a simple softmax loop. [opencog/ure/ActionSelection.h; opencog/ure/ThompsonSampling.cc; opencog/ure/MixtureModel.h/.cc; opencog/ure/backwardchainer/BackwardChainer.cc]

C-10-5. The likely historical authority split is visible from local evidence: Linas Vepstas dominates long-term maintenance and API-churn commits, but Nil Geisweiller is the named author on the advanced control/selection machinery. Any wiki claim about inference-control design intent should therefore not collapse those roles into a single maintainer narrative. [`git shortlog -sn HEAD`; authorship headers in `ActionSelection.*`, `ThompsonSampling.*`, `MixtureModel.*`, `ControlPolicy.*`, `BIT.*`]

C-10-6. The successor signal is now bifurcated: master presents URE as a bit-rotting historical artifact, while branch/doc surfaces point to `trueagi-io/chaining` as the future line. For wiki purposes, that means URE should be documented as a historically important but no-longer-authoritative reasoning engine whose control ideas matter precisely because later lines simplified or replaced them. [README.md; GEMINI.md; `git log -1 origin/update-readme`]
