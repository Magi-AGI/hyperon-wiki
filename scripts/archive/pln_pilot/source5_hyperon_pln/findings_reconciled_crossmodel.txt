## 1. Headline finding: the MP 0.2-hack genealogy is more interesting than we thought

**[X1 Claude → VERIFIED in this reconciliation]** The 2023 seed contains **one** Modus Ponens rule, in `metta/common/formula/ModusPonensFormula.metta`:

```
(= (modus-ponens-formula (STV $As $Ac) (STV $ABs $ABc))
   (let* (($notABs 0.2) ; temporary hack
          ($notABc 1))
       (STV (+ (* $ABs $As) (* $notABs (- 1 $As)))
            (min (min $ABc $notABc) $Ac))))
```

Strength: `s_AB · s_A + 0.2 · (1 − s_A)` (with the symmetric MP shape — `$notABs` plays the role of an assumed `s(¬A → B)`).
Confidence: `min(min($ABc, $notABc), $Ac)`.

**Downstream structurally diverges into TWO rules**:
1. **`Truth_ModusPonens`** in PLN3 / PLN/ uses `c = 0.02` (book §5.7.1 default term probability) — this is a **NEW rule added downstream**, faithful to the 2008 PLN book. **It did not exist in the 2023 seed.**
2. **`Truth_SymmetricModusPonens`** in PLN3 / PLN/ uses `$snotAB = 0.2` — this **inherits the 2023 seed's MP behavior**, just renamed and slotted under a different rule schema (gated by `SymmetricModusPonensRuleGuard` for `Similarity`/`IntentionalSimilarity`/`ExtensionalSimilarity` link types).

This **recontextualizes Source 3's reconciliation §1.2** — that doc framed `c = 0.02` (asymmetric MP) as inherited from the book and `0.2` (Symmetric MP) as a downstream-specific datum. The truer story:
- The `0.2` is **inherited from the 2023 hyperon-pln seed**, written by Manhin Leung (leungmanhin) in PR #23 (drafted 2023-07-13 as *"Draft the Modus Ponens formula and DTL"*; comments added 2023-07-31 explicitly labeling it *"temporary hack"*).
- The `c = 0.02` is **a later book-faithful rule introduced separately** in the production line.
- So the 2023 MP became downstream's *Symmetric* MP, while a fresh book-faithful asymmetric MP was added next to it.

**Wiki implication:** when documenting PLN's Modus Ponens, distinguish the "0.2 symmetric heuristic" (provenance: leungmanhin 2023, hyperon-pln) from the "0.02 asymmetric heuristic" (provenance: book-faithful, introduced post-2023 in production line, exact author TBD). They have different lineages and different stated rationales.

### 2.2 [X3] PrCnt is real but redundant with STV (Claude's bijection clarification)

`TruthValue.metta:L23-L32` defines four constructors: `Bl (Bool)`, `Pr (Number)`, **`PrCnt (Number Number)`**, `STV (Number Number)`.

- **Both Codex and Gemini** flagged PrCnt as a "real transitional TV type."
- **Claude refined this**: PrCnt and STV are **isomorphic via `count ↔ confidence` with lookahead=1** (`count->confidence` and `confidence->count` defined in `TruthValue.metta:L60-L69`). Same beta-distribution semantics; different storage. The `mean` is computed identically through the Beta-posterior.
- **Resolution**: PrCnt is *more substantive than `Bl`/`Pr`* (which are degenerate count=1e9 wrappers) but *redundant with STV* (bijective). Both views compatible.
- Codex's C-2-4: PrCnt is **actively used** in this seed's generic `ded_formula` and equality/entailment test premises — not dead. Source 4's `equal/` and `match/` tree still uses `ETV Empty (PrCnt 1 0)` while production PLN3/PLN/ simplifies to `STV` only.

**Wiki implication**: PrCnt is *historical, not a missing implementation that was abandoned for capability reasons*. It was correctly dropped downstream as redundant. No information was lost.

### 2.3 [X3] The synthesis engine is central here, did NOT survive into production

`metta/synthesis/Synthesize.metta` implements depth-limited proof search over an axiom supplier and a rule supplier. Supports backward chaining, forward chaining, mixed chaining, type checking, and type inference. Used by the dependent-types tests directly (`!(import! &self ../synthesis/Synthesize.metta)`).

**Status checks (X3):**
- `unify`/`unify*` are buggy — README acknowledges. Synthesizer falls back to `let*`.
- Synthesis engine **survives experimentally into Source 4's `experimental/early-pln/`** but is not central there.
- **Production PLN3/PLN/ replaced it entirely** with the bounded-priority-queue forward-chaining `Deriver` — fundamentally different control regime.

This makes Source 5 the **surviving local archive of the original "PLN as program synthesis over typed proof constructors" architecture**.

### 2.4 [X3] Consistency conditions: stubbed in seed, activated downstream

**[X2 Codex + Claude, VERIFIED]** Two divergent paths within `DeductionFormula.metta`:
- Generic `ded_formula` (consumes any `TruthValue`): `conditional-probability-consistency` is **stubbed to `True`** with the actual inequality checks **commented out**.
- STV-specialized `deduction-formula`: uses the **real consistency checks**.

**Downstream behavior:** PLN3/PLN/'s `lib_pln.metta` **uncommented and reactivated** the consistency-condition checks. This is a downstream stabilization choice, not seed behavior. The production code we documented in Sources 2 and 3 inherited the *idea* (constraint enforcement) but had to *re-enable* it from the inert seed code.

**Wiki implication**: when Source 1's V1 "consistency condition" was confirmed at the code level in Sources 2-3, the actual code path used by production was the STV-specialized version, not the generic one. The generic "type-polymorphic deduction" never enforced book-page-74 constraints.

### 2.6 [X3] Idris prototyping was design-pattern source, code abandoned

`idris/StrictlySorted.idr` is **a 105-line 2-implementation Idris2 prototype** of strictly-sorted-list dependent type with type-level `<`-proofs (Codex C-5-1; Claude verified). The README explicitly frames Idris as *"more mature than MeTTa"* for prototyping dependent-type concepts.

**[X1 Gemini, X1 Claude]**: This Idris work is **the formal origin of the sorted-evidence-ID double-counting prevention** later seen in MeTTa code (Source 4's IDIRecursive uses sorted-ID checks; production code uses `StampDisjoint` + sorted union). The pattern survived; the Idris code didn't.

**[X1 Claude]**: MeTTa's 2023 type checker couldn't supply the sorted-list proof, so `Set.metta` uses `believe_me` as an anti-pattern workaround. The Idris file documents what the MeTTa code *should* have been able to prove if the type checker were stronger.

**Downstream**: Idris was carried into Source 4's `experimental/early-pln/`, then **explicitly deleted on 2025-12-18** in PR #49 ("Remove Idris experiments"). Code dead; design lives on.
