# C-1-4. Methodological/formal differences (verified across files):
#   - dependent-types (metta/dependent-types/DeductionDTL.metta L20-29): rules are TYPE CONSTRUCTORS. The rule body is the line `(: Deduction (-> ... (≞ (→ $p $r) (deduction-formula ...))))`. A proof is a Hyperon Atom built from these constructors. Truth-value computation is embedded as the type-level expression `(deduction-formula ...)`.
#   - entail (metta/entail/DeductionEntail.metta L23-30): rules are atoms whose head is `⊢`. Body is `(⊢ premise1 ... premiseN conclusion)`. Premises and conclusion are listed as siblings under the same `⊢` constructor; consumed via `match &self (⊢ $p1 ... $pN $res) $res`.
#   - equal (metta/equal/DeductionEqual.metta L24-40): rules use bare MeTTa `=`. Forward direction: `(= (, prem1 ... premN) conclusion)` rewrites a tuple of premises into the conclusion. Backward direction defined symmetrically but with an explicit author comment on L31-33: "Unfortunately this won't work, the interpreter cannot unify the left hand side of the equality because it contains function calls, as opposed to just constructors." So equal is broken in BC direction by interpreter limitation, not by design.
#   - match (metta/match/DeductionMatch.metta L32-51): rules are functions; matching is hand-written via `(match $kb (, prems...) (deduction prems...))`. The simplest, most explicit, but does not chain.

# §2 — Truth-value type hierarchy (PrCnt characterization)
# ============================================================
#
# C-2-1. `metta/common/truthvalue/TruthValue.metta` L10 declares: `(: TruthValue Type)`.
#
# C-2-2. Four constructors declared (verbatim, file metta/common/truthvalue/TruthValue.metta L17-32):
#   L17-18:  ;; Boolean TV constructor
#            (: Bl (-> Bool TruthValue))
#   L20-21:  ;; First order probability TV constructor, i.e. mere probability.
#            (: Pr (-> Number TruthValue))
#   L23-27:  ;; Second order probability TV constructor, i.e. probability and
#            ;; count.  The second order probability is distributed according to a
#            ;; beta distribution.  The first argument of the constructor
#            ;; represents the mode of the second order distribution.
#            (: PrCnt (-> Number Number TruthValue))
#   L29-32:  ;; Simple Truth Value.  A Second order probability TV constructor,
#            ;; i.e. probability and confidence.  The probability is in fact the
#            ;; mode of the corresponding beta distribution.
#            (: STV (-> Number Number TruthValue))
#
# C-2-3. **PrCnt vs STV — the critical lineage finding** (TruthValue.metta L23-32 + L107-126):
#   - PrCnt = `(probability_mode, count)` — second-order probability with **explicit raw count** of evidence.
#   - STV   = `(probability_mode, confidence)` — second-order probability with **confidence** in [0,1).
#   - Both reference the SAME beta-distribution semantics (Jeffreys prior α=β=0.5, file L40-43).
#   - They are **isomorphic via `count→confidence` (L60-61)** and `confidence→count` (L66-69).
#   - confidence = count / (count + lookahead), where lookahead = 1.0 (file L46-47).
#   - confidence→count = (confidence·lookahead) / (1 - confidence), with a `if approxEq 1.0 1e-9 → max-count=1e9` clamp.
#   So PrCnt and STV are **two encodings of the same evidence-count beta-distribution**, transformed by a lookahead-1 monotonic bijection. PrCnt stores raw count; STV stores normalized confidence. **PrCnt is NOT a different theory.** It is the **count-form sibling** of STV.
#
# C-2-4. `pos_count`, `neg_count`, `post_alpha`, `post_beta`, `mean` (file L128-161): all defined uniformly across `Bl`, `Pr`, `PrCnt`, `STV`, falling back through `count` and `mode`. The four TV types share one polymorphic interface.
#
# C-2-5. **Bl and Pr are degenerate fixed-certainty wrappers** (verified TruthValue.metta L96-126):
#   - mode (Bl True) = 1.0, mode (Bl False) = 0.0
#   - mode (Pr $pr) = $pr
#   - count (Bl _) = max-count = 1e9 (constant, L51)
#   - count (Pr _) = max-count = 1e9
#   - confidence (Bl _) = 1.0
#   - confidence (Pr _) = 1.0
#   So `Bl b` ≅ `STV (if b 1 0) (1 - 1e-9)` and `Pr p` ≅ `STV p (1 - 1e-9)`. **Confirms Source 4's Bl/Pr-as-degenerate-STV-wrappers finding** at the seed.
#
# C-2-6. **PrCnt does have evidence-accumulation that Pr lacks** — its second argument is a raw count, so accumulation is via count addition (e.g., `inc-count` on L72-73 = `+1`). Pr has no count parameter, so cannot accumulate. **This makes PrCnt the only one of {Bl, Pr} that supports incremental learning.**
#
# C-2-7. **PrCnt is NOT the missing link between Pr and STV** — it is a parallel encoding of STV's underlying evidence count. The "missing link" framing in the §2 brief is the wrong shape; PrCnt and STV are equivalent up to a monotonic invertible transformation.
#
# C-2-8. The brief asks: was PrCnt carried into Source 4's `experimental/early-pln/`? Source 4 reconciliation §2.2 says only STV is destructured by rules in pln-experimental2. PrCnt as a constructor presumably exists in that repo's TruthValue.metta (since pln-experimental2 inherits this code), but is unused by the {dependent-types, entail, equal, match} encodings there.
#
# C-2-9. **Was PrCnt carried into PLN3 / PLN/ at all?** Per Source 2 and Source 3 reconciliations, PLN3 and PLN/ are STV-only — no PrCnt constructor. **PrCnt was DROPPED downstream in the production fork.**
#
# C-2-10. **PrCnt remains active inside hyperon-pln itself** — used by:
#   - `ded_formula` (DeductionFormula.metta L40-57) returns `(PrCnt ...)`.
#   - DeductionFormulaTest.metta L14: `!(ded_formula (PrCnt 1 10) (PrCnt 1 4) ...)`.
#   - DeductionEntailTest.metta L4-8: all premises `(PrCnt 1 0.1)` / `(PrCnt 1 0.5)`.

# C-4-1. `metta/synthesis/Synthesize.metta` L37-80: the `synthesize` function definition is **5 separate `=` rules** for nullary/unary/binary/trinary/quaternary/quintenary rule arities. Verbatim header comment L34-36: "TODO: recurse over curried rules instead of duplicating code over tuples." **This is a hand-written variadic dispatch via overloading, not a recursive Curry-style implementation.**
#
# C-4-2. Synthesizer signature (file L37): `(: synthesize (-> Atom Nat (-> Atom) (-> Atom) Atom))`. Inputs:
#   - $query: an atom of form `(: TERM TYPE)` with optional free vars.
#   - $depth: a Nat (Peano numeral) bounding search depth.
#   - $axiom: nullary thunk that nondeterministically yields axioms.
#   - $rule: nullary thunk that nondeterministically yields rules.
#
# C-4-3. Comment block L11-19 documents 5 query modes:
#   1. Backward chaining: `(: $term (Inheritance $x Mammal))`
#   2. Forward chaining: `(: ($rule $premise AXIOM) $type)`
#   3. Mixed chaining: `(: ($rule $premise AXIOM) (Inheritance $x Mammal))`
#   4. Type checking: `(: TERM TYPE)`
#   5. Type inference: `(: TERM $type)`
#   So **the synthesizer supports backward, forward, AND mixed chaining as a single primitive** — a sophisticated unifier-driven design.
#
# C-4-4. Implementation core (binary rule case, L48-53):
#       (= (synthesize $query (S $k) $axiom $rule)
#          (let* (((: $ructor (-> $premise1 $premise2 $conclusion)) ($rule))
#                 ((: ($ructor $proof1 $proof2) $conclusion) $query)
#                 ((: $proof1 $premise1) (synthesize (: $proof1 $premise1) $k $axiom $rule))
#                 ((: $proof2 $premise2) (synthesize (: $proof2 $premise2) $k $axiom $rule)))
#                $query))
#   The recursive descent on premises is depth-first, depth-bounded by Peano `S` decrement. **Unification is via MeTTa `let*`** (variable binding); not via the custom `unify` in Unify.metta.
#
# C-4-5. The synthesis README.md L8-11 explicitly notes: "It is meant to be using `unify` and `unify*`, however due to some issues it still relies on `let` and `let*`." **This is the AGENTS.md-flagged "unifier-related tests are still unstable" subtopic.** The Synthesize.metta in production uses `let*` instead of `unify` because `unify` is buggy (or runtime non-supportive). **The `unify`-based synthesizer lives only in `experiments/synthesize-via-unify.metta` and remains experimental.**
#
# C-4-6. `Unify.metta` is 25 lines (file metta/synthesis/Unify.metta) defining `unify` and `unify*` via `case` (not `let`). The intended distinction: `case` filters out branches that haven't fully reduced; `let` keeps them. `unify`'s definition is one-liner: `(case $rterm (($lterm $rewrite)))`. `UnifyTest.metta` is 14 lines, 3 tests; the README admits these "are not working yet."
#
# C-4-7. **Backward chaining**: supported as one of the 5 query modes (per C-4-3). The synthesizer recursion is bidirectional: it unifies the conclusion with $query and then recursively solves premises against the same kb/rb. **Backward and forward are not separate algorithms; the same recursion handles both via free-variable placement.**
#
# C-4-8. **Did the synthesizer survive into Source 4 / PLN/**?
#   - Source 4 (pln-experimental2) reconciliation §5 says experimental/early-pln/ contains 4-encoding rules covering 3 rules — does not mention a synthesizer module surviving. The brief and reconciliation list `experimental/early-pln/dependent-types/` as descendant of this repo's dependent-types tree.
#   - PLN/ uses a forward-chaining bounded-priority-queue Deriver (Source 3 §5.1, §3.1) — **a completely different control regime**. Not derived from this synthesizer.
#   - PLN3 uses the same Deriver as PLN/ (Source 2 §1.2). Same conclusion.

# C-4-9. **Why was it abandoned?** Likely speed: depth-bounded recursive synthesis with `let*` is exponential in depth × rule-count. The PLN3/PLN/ priority-queue Deriver is iterative and has explicit beam-size knobs (TaskQueueSize, BeliefQueueSize). For production, PQ-Deriver wins on scaling. **The synthesizer was a research tool, not a production engine.**

Each represents an evolutionary step toward the final `Synthesize.metta`. Most have inline notes like "It fails because the type checker is static" (per experiments/README.md L7-9). **A clear archive of failed-or-superseded approaches.**

They all import `../synthesis/Synthesize.metta` and call `(synthesize ...)` in their test bodies. **Without the synthesizer, this repo's dependent-types encoding has no way to generate proofs.** The synthesizer's abandonment downstream is what forced PLN3/PLN/ to replace dependent-types-style proof construction with PQ-Deriver-style forward-chaining beam search.
