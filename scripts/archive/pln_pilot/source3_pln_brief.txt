### §3 — PLN3 bug / deviation verification

Source 2 flagged concrete code-level issues in PLN3. For each, determine if `PLN/` fixes, carries, or sidesteps it:

**V7 — Inversion strength bug.** PLN3's `Truth_inversion` returned `(stv $ABs ...)` unchanged — strength was NOT updated via Bayes (`sBA = sAB·sA/sB`). Only a `0.6·cB·cAB` confidence discount was applied. Code comment: *"not according to OpenCOG classic"*. Does `PLN/` fix this?

**V8 — Revision confidence saturation.** PLN3's `Truth_Revision` clamps confidence with `max($c, $c1, $c2)` so it monotonically approaches 1.0. Does `PLN/` still clamp this way, or has revision been re-derived?

**V9 — `TransitiveSimilarity` zero-division.** PLN3's `TransitiveSimilarityStrength` uses raw `/` (not `/safe`) in 4 spots — will crash on `$sA=0`, `$sB=0`, `$sC=0`. Fixed in `PLN/`?

**V10 — `sentence` / `Sentence` capitalization.** PLN3 had this inconsistency bridged only via a Python-injected rewrite in `PLN.py`. Pure-MeTTa tests without the Python shim silently broke. Does `PLN/` have consistent capitalization, or does it still require a shim?

**V11 — Python shim dependency.** Does `lib_pln.metta` require Python-side compilation like PLN3's `PLN.Init` did, or is it self-contained MeTTa?
