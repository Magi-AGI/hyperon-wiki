## 1. Architecture delta: monolith vs. modular

C-1-1: `PLN/` is a self-contained, PeTTa-oriented upstream surface built around one checked-in library file, `lib_pln.metta`, plus `examples/`, `ruletests/`, and a repo-root `test.sh`. There is no `src/` tree, no `build.sh`, and no checked-in `PLN.py` translator shim in this repo. [AGENTS.md:L4-L12] [CLAUDE.md:L7-L18] [README.md:L39-L63]

C-1-2: `lib_pln.metta` is structurally a flattened successor to the former modular layout, with clear in-file boundaries for utilities, consistency constraints, truth functions, inference rules, config, and derivation/query control. The file progresses in that order from helper math and dedup (`/safe`, `PushUnique`, `ConcatUnique`), to constraints, then formulas, then rule schemas, then config and deriver/query logic. [lib_pln.metta:L1-L34] [lib_pln.metta:L38-L64] [lib_pln.metta:L65-L208] [lib_pln.metta:L210-L430]

C-1-3: The public API is narrower and cleaner than the PLN3 path: there is no `PLN.Init`. The repo exposes `PLN.Derive` and `PLN.Query` directly from `lib_pln.metta`, and every example imports the repo by `git-import!` and then imports `(library PLN lib_pln)`. [lib_pln.metta:L377-L430] [README.md:L57-L64] [examples/FlyingRaven.metta:L1-L3]

C-1-4: The Python/transpiler path from PLN3 is gone from the repo surface. The only remaining evidence of a former translator is a comment saying negation introduction is "handled by Translator", but there is no `Translator.metta`, no `PLN.py`, and no `PLN.Init` symbol in the checked-in tree. [lib_pln.metta:L283-L284]

C-1-5: This means the Source 2 Python-shim dependency does not carry over as a repo-local runtime path. `PLN/` is intended to be imported and executed directly inside a PeTTa workspace through `git-import!`, not through generated intermediate files. [README.md:L24-L64] [AGENTS.md:L15-L27]

C-1-6: `test.sh` is also different in shape from the PLN3 local-build flow. It expects the repo under `PeTTa/repos/PLN`, does `cd ../../`, and runs every `./repos/PLN/*/*.metta` file in parallel via `run.sh`, excluding only a few named examples. That makes the whole example and ruletest corpus part of the default regression surface. [test.sh:L1-L39]

C-1-7: The controller defaults changed materially from the Source 2 configuration. `PLN/` now sets `MaxSteps=100`, `TaskQueueSize=10`, and `BeliefQueueSize=100`, i.e. a deeper horizon but tighter active queues. [lib_pln.metta:L325-L332]

C-1-8: `CLAUDE.md` is stale on config: it still claims `MaxSteps=20`, `TaskQueueSize=20`, and `BeliefQueueSize=200`, while the code now defines `100/10/100`. The doc is reliable on the monolithic layout, but not on the current runtime knobs. [CLAUDE.md:L46-L52] [lib_pln.metta:L325-L332]

## 2. Rule-level delta

C-2-1: The core rule surface is still the same compact STV subset seen in Source 2: revision, asymmetric MP, symmetric MP, deduction, induction, abduction, three evaluation-over-inheritance bridge rules, negation elimination, inheritance inversion, implication inversion, two equivalence-to-implication directions, transitive similarity, evaluation implication, and member deduction. No new high-level rule families appear in `lib_pln.metta`. [lib_pln.metta:L210-L323]

C-2-2: The book-facing formulas for deduction, induction, abduction, MP, symmetric MP, revision, negation, inversion, equivalence-to-implication, transitive similarity, and evaluation implication are still present verbatim in the monolith. This is not a conceptual rewrite of the Source 2 math; it is mostly a packaging/consolidation move. [lib_pln.metta:L82-L204]

C-2-3: Standard Modus Ponens still hardcodes the familiar fallback constant `0.02`. That part of Source 2 carries forward unchanged. [lib_pln.metta:L123-L126]

C-2-4: Symmetric Modus Ponens still uses the multiplication form, but it also still uses the distinct local constant `$snotAB = 0.2` rather than the book's `c = 0.02`. The `cnotAB` binding remains present but unused. [lib_pln.metta:L128-L133]

C-2-5: Revision is still the same weighted-average STV merge with converted weights and an upward clamp by `max(max(c,c1),c2)`. There is no new overlap-sensitive or trail-sensitive revision derivation in the core library. [lib_pln.metta:L135-L143]

C-2-6: Inversion is still the same simplified rule that preserves the original strength and only discounts confidence by `0.6 * target_confidence * link_confidence`. This remains a major semantic shortcut relative to classical/OpenCog-style Bayes inversion. [lib_pln.metta:L150-L153]

C-2-7: Transitive similarity still uses raw `/` in the four `T1..T4` terms and only falls back to `/safe` in the later nested correction terms. The zero-division hazard is still present in the main similarity-strength calculation. [lib_pln.metta:L165-L171]

C-2-8: The predicate bridge rules are still first-class runtime schemas, not just documentation claims: unary inheritance-to-predicate substitution, plus binary argument-1 and argument-2 substitutions. [lib_pln.metta:L261-L281]

C-2-9: `Direct.metta` does not add a new core inference rule family. It is a user-space macro layer that compiles implication-like clauses into direct MeTTa reductions using `Truth_ModusPonens` and then merges alternate derivations with `Truth_Revision`. That is an example-level extension, not a new engine primitive. [examples/Direct.metta:L5-L15]

C-2-10: The repo still cites older OpenCog rule files and the `hyperon-pln` dependent-types deduction file for provenance, but those citations do not correspond to recovered rule coverage. The executable library remains a compact subset. [lib_pln.metta:L38-L40] [lib_pln.metta:L82-L83] [lib_pln.metta:L128-L129] [lib_pln.metta:L155-L196] [lib_pln.metta:L235-L317]

C-2-11: The only meaningful rule-surface additions over the practical Source 2 experience are the three new example programs and the wider top-level test sweep, not new implemented inference families inside `lib_pln.metta`. [test.sh:L1-L39] [examples/DeductionRevision.metta:L1-L18] [examples/Direct.metta:L1-L33] [examples/RavenInduction.metta:L1-L17]

C-2-12: `README.md` and `GEMINI.md` describe "multi-component truth-values", but the actual code surface remains plain STV `(strength, confidence)` throughout. There is no second truth-value hierarchy beyond that pair. [README.md:L4-L21] [GEMINI.md:L9-L15] [lib_pln.metta:L67-L76] [lib_pln.metta:L82-L204]

## 3. PLN3 bug / deviation verification

C-3-1: V7 persists. `Truth_inversion` is still exactly the simplified form that leaves strength unchanged and only applies a `0.6` confidence discount: `(stv $ABs (* $Bc (* $ABc 0.6)))`. The Source 2 inversion bug is carried forward, not fixed. [lib_pln.metta:L150-L153] [ruletests/inversion.metta:L13-L18]

C-3-2: V8 persists. `Truth_Revision` still computes `$c = Truth_w2c($w)` and then clamps the returned confidence with `min 1.0 (max (max $c $c1) $c2)`, so the monotone upward pressure on confidence remains part of the implementation. [lib_pln.metta:L135-L143]

C-3-3: V9 persists. `TransitiveSimilarityStrength` still contains raw divisions by `$sA`, `$sB`, and `$sC` in the `T1..T4` terms, so the zero-division risk is still in the code path. [lib_pln.metta:L165-L171]

C-3-4: V10 is effectively sidestepped. The repo-local files use `Sentence` consistently, and a repo-wide search finds no lowercase `sentence` symbol at all. The specific capitalization bug seen in PLN3 is not reproduced in this tree. [lib_pln.metta:L362-L420] [examples/FlyingRaven.metta:L12-L37] [examples/DeductionRevision.metta:L10-L18]

C-3-5: V11 is fixed at the repository-surface level. `PLN/` no longer depends on a checked-in Python shim or generated translator artifact. The runtime path is direct `git-import!` plus `lib_pln`, and the test harness runs `.metta` files directly through PeTTa's `run.sh`. [README.md:L39-L64] [test.sh:L1-L39]

## 4. New content: examples not in PLN3

C-4-1: `DeductionRevision.metta` demonstrates revision through duplicated deduction paths, not a new revision formula. It sets up `A -> B`, `A -> C`, `B -> D`, and `C -> D`, then queries `A -> D` and expects a single revised result with combined evidence `(1 2 3 4)` and truth value `(stv 0.125 0.30429942952553224)`. This exercises "deduce twice, then revise", but it does not change the underlying revision logic. [examples/DeductionRevision.metta:L5-L18]

C-4-2: `Direct.metta` is an example-level clause-compilation convenience layer. It defines a custom `=>` operator that folds antecedents through `Truth_ModusPonens`, a `?` query helper that collapses and revises multiple derived variants, and then demonstrates a small `father/mother/grandfather/old` toy theory. This is orthogonal to the core PLN rule inventory. [examples/Direct.metta:L5-L33]

C-4-3: `RavenInduction.metta` is a clean induction example. It introduces two raven instances, one positively black and one negatively black, then queries `(Inheritance raven (IntSet black))` and expects `(stv 0.38000000000000006 0.6541049798115747)` with combined evidence `(1 2 3 4)`. That complements `FlyingRaven.metta`, which is deduction-driven, by making the class-property inference direction explicit. [examples/RavenInduction.metta:L4-L17] [examples/FlyingRaven.metta:L12-L49]

C-4-4: Because `test.sh` now sweeps all `.metta` files under `examples/` and `ruletests/`, these three added examples are part of the default regression run, not just illustrative extras. [test.sh:L19-L39]

## 6. Canonical current surface

C-6-1: The real current entrypoints are `PLN.Derive` and `PLN.Query`, each with multiple arities. `PLN.Derive` returns `($Tasks $Beliefs)` state, and `PLN.Query` selects the best-confidence matching sentence from the derived belief set. [lib_pln.metta:L377-L430]

C-6-2: The controller remains a confidence-ranked forward derivation loop, but the current defaults show a different operational posture than Source 2: longer derivation horizon (`100`) with tighter active buffers (`10` task queue, `100` belief queue). [lib_pln.metta:L325-L332] [lib_pln.metta:L377-L430]

C-6-3: The active dedup/order substrate has been modernized inside the monolith.
