## Executive summary (read first)

**PLN/ is ~99 % a pure repackaging of PLN3 into a single file, with ONE semantic change and TWO config changes.** The 18 558-byte `lib_pln.metta` is a concatenation of (nearly) every line of `Utils.metta`, `Constraints.metta`, `Formulas.metta`, `Rules.metta`, `Config.metta`, and `Deriver.metta` from PLN3 — reordered into sections, with inline comments preserved. The only **semantic** delta in the rule/formula body is that `Truth_SymmetricModusPonens` now has a **different confidence formula** (see C-2-4 below). All five V7–V11 bugs carry over unchanged. No missing families (quantifiers, temporal, intensional, distributional, FactorGraph/MORK) are recovered. The Python shim is **gone** (not replaced — just deleted along with `Translator.metta`, `build.sh`, and `PLN.Init`). Three new examples were added; `DeductionRevision.metta` is a plain two-path deduction-then-revision test, NOT a revision-confidence-saturation probe. The repo is explicitly positioned as a **PeTTa-coupled library** (PeTTa is the SWI-Prolog MeTTa runtime), not self-contained — tests require an external PeTTa workspace.

## §1 — Architecture delta: monolith vs. modular

### C-1-1 — `src/` directory entirely deleted
`CLAUDE.md:7` states verbatim: *"The codebase was recently restructured: the `src/` directory was removed and all core modules consolidated into a single `lib_pln.metta` at the repository root."* No `src/` exists. No `Translator.metta`, `Utils.metta`, `Constraints.metta`, `Formulas.metta`, `Rules.metta`, `Config.metta`, or `Deriver.metta` files.

### C-1-2 — `lib_pln.metta` is a sectioned concatenation, not a rewrite
Visible section headers inside `lib_pln.metta` (via blank-line separators and comments) map 1-to-1 to PLN3's former files:
- Utils helpers (`and5`, `min5`, `/safe`, `clamp`, `negate`, `invert`, `PushUnique`, `ConcatUnique`) at `lib_pln.metta:L1-L34` ← was PLN3 `src/Utils.metta`
- Consistency conditions at `lib_pln.metta:L38-L63` ← was `src/Constraints.metta`
- Truth functions (`Truth_c2w`, `Truth_or`, `Truth_Deduction`, `Truth_Induction`, `Truth_Abduction`, `Truth_ModusPonens`, `Truth_SymmetricModusPonens`, `Truth_Revision`, `Truth_Negation`, `Truth_inversion`, `Truth_equivalenceToImplication`, `TransitiveSimilarityStrength`, `Truth_transitiveSimilarity`, `simpleDeductionStrength`, `Truth_evaluationImplication`) at `lib_pln.metta:L65-L204` ← was `src/Formulas.metta`
- Inference rules (the `|-` patterns) at `lib_pln.metta:L208-L323` ← was `src/Rules.metta`
- Config constants at `lib_pln.metta:L326-L332` ← was `src/Config.metta`
- Stamp + PQ + `PLN.Derive` + `PLN.Query` at `lib_pln.metta:L335-L430` ← was `src/Deriver.metta`

### C-1-3 — `Translator.metta` is ABSENT; translator-era comment retained as vestigial
No translator module, no `(= (Translate ...) ...)` code, no pre-expansion step anywhere in the repo. However, `lib_pln.metta:L283` comment reads *"negation elimination rule (introduction now handled by Translator)"* — this is stale text carried over from when Translator did exist. **No code in `PLN/` implements negation-introduction.** This is a documentation-to-code mismatch: the comment asserts a Translator exists but the repo has none. Two ruletests (`evaluationImplicationRuleA.metta:L1`, `evaluationWithNegationAndInheritanceInversion.metta:L1`) start with `!(translatePredicate (retractall ...))` — another dangling reference to a runtime feature not present in `lib_pln.metta`.

### C-1-4 — `build.sh` is gone; `test.sh` is a PeTTa-workspace runner
PLN3 had a self-contained MeTTa `build.sh` that concatenated `src/*.metta` into `PLN.metta` (excluding `Translator.metta`). `PLN/` has no `build.sh`. `test.sh` is a thin shell harness that `cd ../../` then calls `sh run.sh ./repos/PLN/*/*.metta` — it is **only runnable inside a parent PeTTa workspace**, never standalone. `test.sh:L28` globs `./repos/PLN/*/*.metta`, confirming the assumed layout.

### C-1-5 — No Python shim at all
Grep-verified: zero `*.py` files in the repo (confirmed via `Glob **/*.py` → no results). No `PLN.Init`. No `metta-morph` reference. No `examples/PLN.py`. This is a *removal*, not a replacement. Compare PLN3's `examples/PLN.py` which registered `PLN.Init` and did offline KB compilation.

### C-1-6 — Import surface changed
PLN3 examples started with one line: `!(import! &self PLN)` (after its build.sh-produced `PLN.metta`). PLN/ examples use a **three-line** PeTTa-specific sequence:
```

!(git-import! "https://github.com/trueagi-io/PLN.git")
!(import! &self (library PLN lib_pln))
```
(verbatim from `examples/Smokes.metta:L1-3`, `examples/FlyingRaven.metta:L1-3`, etc.)

### C-1-7 — Entry points preserved, defaults changed
`PLN.Derive` and `PLN.Query` exist at `lib_pln.metta:L377-L430` with the same four overloads (full args, drop `$steps`, drop queue sizes, drop max). Return contract unchanged: `PLN.Derive` returns `($Tasks $Beliefs)` pair. `PLN.Query` unchanged.

### C-1-8 — Config default constants doubled/halved
`lib_pln.metta:L326-L332`:
```
(= (PLN.Config.MaxSteps) 100)
(= (PLN.Config.TaskQueueSize) 10)
(= (PLN.Config.BeliefQueueSize) 100)
```
Compare PLN3 `src/Config.metta`: `MaxSteps=20, TaskQueueSize=20, BeliefQueueSize=200`.
Net changes: MaxSteps **5x** (20 → 100), TaskQueueSize **halved** (20 → 10), BeliefQueueSize **halved** (200 → 100).
**Behavioral implication:** deeper but narrower search per invocation. This is a material configuration delta, not a refactor. Note: `CLAUDE.md:L49-L51` lists `MaxSteps=20, TaskQueueSize=20, BeliefQueueSize=200` — meta-doc **contradicts** actual code (confirmed mismatch).

### C-1-9 — No visible `PLN.Init` entry point, no pre-expansion phase
PLN3 (per Source 2) had `PLN.Init` as the primary user-facing lifecycle atom that loaded files, compiled, and pre-expanded translator rules. In PLN/, there is no such atom defined anywhere in `lib_pln.metta`. The init-phase semantics have been fully elided; derivation is invoked directly on a user-supplied KB list.

### C-2-3 — Symmetric Modus Ponens: strength IDENTICAL, **CONFIDENCE FORMULA CHANGED**
`lib_pln.metta:L129-L133`:
```
(= (Truth_SymmetricModusPonens (stv $sA $cA) (stv $sAB $cAB))
   (let* (($snotAB 0.2)
          ($cnotAB 1.0))
         (stv (+ (* $sA $sAB) (* (* $snotAB (negate $sA)) (+ 1.0 $sAB)))
              (* (* $cA $cAB) (Truth_or $sA $sAB)))))
```
PLN3's confidence line was:
```
(min (min $cAB $cnotAB) $cA)
```
(per reconciled Source 2 §2 V1).
**PLN/ replaces `min(min($cAB, $cnotAB), $cA)` with `(* (* $cA $cAB) (Truth_or $sA $sAB))`** — multiplication of confidences times the `Truth_or` of the two strengths. This is a **semantic change**, not a refactor. `$cnotAB` is defined (1.0) but no longer used. Strength formula (`sA·sAB + snotAB·(1-sA)·(1+sAB)`) is bit-identical.

### C-2-4 — Symmetric MP confidence delta implications
The new formula gives more-nuanced confidence (scaled by an or-combined strength term) rather than a flat min. For inputs like `sA=0.99, sAB=0.99, cA=cAB=0.9`: old form = `min(min(0.9, 1.0), 0.9) = 0.9`; new form = `0.9·0.9·(1 - 0.01·0.01) ≈ 0.81`. The new form is generally **lower** and depends on strengths. This is a **meaningful behavioral change** that impacts any inference through symmetric MP (used for Similarity / IntentionalSimilarity / ExtensionalSimilarity). Test values in `RuleTester.metta` do not exercise symmetric MP, so this delta is unvalidated by the included tests.

## §4 — New content: examples not in PLN3

### C-4-1 — `examples/DeductionRevision.metta` — two-path deduction with revision
Full verbatim (`DeductionRevision.metta:L5-L18`):
```
(= (STV A) (stv 0.5 0.9))
(= (STV B) (stv 0.25 0.9))
(= (STV C) (stv 0.25 0.9))
(= (STV D) (stv 0.5 0.9))

(= (kb)
   ((Sentence ((Inheritance A B) (stv 0.25 0.9)) (1))
    (Sentence ((Inheritance A C) (stv 0.25 0.9)) (2))
    (Sentence ((Inheritance B D) (stv 0.5 0.9)) (3))
    (Sentence ((Inheritance C D) (stv 0.5 0.9)) (4))
   ))

!(test (PLN.Query (kb) (Inheritance A D) 100 30 100)
       ((stv 0.125 0.30429942952553224) (1 2 3 4)))
```
Inference exercised: Deduction(A→B, B→D) and Deduction(A→C, C→D), then Revision of the two conclusions.

Expected TV: `(stv 0.125 0.30429942952553224)` over evidence IDs `(1 2 3 4)`. **The test answer reveals:**
- Strength `0.125` matches `$sAB·$sBC = 0.25·0.5 = 0.125` via both paths (short-circuit path since the independence formula with consistent inputs converges).
- Confidence `~0.304` is below 1.0 — so **Revision did NOT saturate** on this single-merge case with four-element disjoint stamps. This is consistent with the `StampDisjoint` check preventing the trivial overlap; revision *structure* works correctly for the first merge.

### C-4-2 — `DeductionRevision.metta` is NOT a probe of V8 (contra Gemini's flag)
The test uses two **disjoint** evidence stamps `(1 3)` and `(2 4)` — so `StampDisjoint` allows the merge, and Revision runs with `(c=?, c1=?, c2=?)` where the derived `c` from `w1+w2` addition is already large. The `max($c, $c1, $c2)` clamp is never tested against a pathological re-revision with overlapping stamps. **This example does not demonstrate revision-confidence-saturation improvement — the bug is untested.** Gemini's hypothesis was structurally plausible but not borne out by the test content.

### C-4-3 — `examples/Direct.metta` — macro-style `=>` compilation extension
Full verbatim of the extension (`Direct.metta:L6-L15`):
```
(: => (-> Expression Expression Atom Atom))
(= (=> (cons , $args) $C $stvImp)
       (progn (cut) (add-atom &self (= $C (Truth_ModusPonens (foldl-atom $args (stv 1.0 1.0) Truth_ModusPonens) $stvImp)))))
(= (=> $A $C $stvImp)
   (add-atom &self (= $C (Truth_ModusPonens $A $stvImp))))

(: ? (-> Expression Atom))
(= (? $term)
   (unique-atom (collapse ($term (progn (reduce $term) ;ensures binding to specific answer, then combines evidence for all derived variants:
                                        (foldl-atom (collapse (reduce $term)) (stv 0.5 0.0) Truth_Revision))))))
```
Exercised rules:
```
!(=> (, (father $a $b) (father $b $c)) (grandfather $a $c) (stv 1.0 0.9))
!(=> (, (father $a $b) (mother $b $c)) (grandfather $a $c) (stv 1.0 0.9))
!(=> (grandfather $a $x) (old $a) (stv 1.0 0.9))
```
**This is a parallel inference surface**, not using `PLN.Derive` at all. It defines `=>` as a macro that compiles user-given implications into `Truth_ModusPonens`-driven MeTTa equations, and `?` as a query combinator that `reduce`s then `Revision`-folds results. **This is a new pattern-matching front-end that bypasses the PQ-based Deriver** in favor of native MeTTa reduction + Revision aggregation. Expected outputs: `(grandfather a c) → (stv 1.0 0.729)`, `(old a) → (stv 1.0 0.6561)`. The `0.729 = 0.9^3` confidence chain suggests the macro is doing three-step MP chaining without a PQ.

### C-4-4 — `Direct.metta` relationship to `hyperon-pln` direct-introduction rules: inconclusive from code

### C-4-5 — `examples/RavenInduction.metta` — induction over IntSet
Full verbatim (`RavenInduction.metta:L1-L17`):
```
!(import! &self (library lib_patrick))
!(import! &self (library lib_pln))

(= (STV rv1) (stv 0.25 0.9))
(= (STV rv2) (stv 0.25 0.9))
(= (STV raven) (stv 0.25 0.9))
(= (STV (IntSet black)) (stv 0.25 0.9))

(= (kb)
   ((Sentence ((Inheritance rv1 raven) (stv 0.9 0.9)) (1))
    (Sentence ((Inheritance rv2 raven) (stv 0.9 0.9)) (2))
    (Sentence ((Inheritance rv1 (IntSet black)) (stv 0.8 0.9)) (3))
    (Sentence ((Inheritance rv2 (IntSet black)) (stv 0.0 0.9)) (4))
   ))

!(test (PLN.Query (kb) (Inheritance raven (IntSet black)))
       ((stv 0.38000000000000006 0.6541049798115747) (1 2 3 4)))
```
Inference pattern: given two individual ravens (rv1, rv2) with disagreeing IntSet-black membership (0.8 and 0.0), **induce** `(Inheritance raven (IntSet black))`. This inverts the FlyingRaven case (which is forward-deduction on Sam being a raven → flies). Expected strength `0.38` is consistent with induction's averaging-style formula over partially-disagreeing evidence. Uses **different import header** (`lib_patrick` instead of PeTTa's `lib_import`) — suggesting experimental harness not integrated with the main git-import flow. This is a **red flag**: the example's import won't work under the standard PeTTa runtime as documented in README.

### C-4-6 — `RavenInduction.metta` vs. `FlyingRaven.metta` directionality
`FlyingRaven.metta` exercises Deduction + ModusPonens forward (Sam → Raven → Bird → flies) and resolves the penguin exception case with asymmetric-MP blending. `RavenInduction.metta` exercises Induction (two individuals' property membership to class property membership). They are **complementary directional tests**, not redundant — the new one fills the induction-coverage gap the Raven scenario previously had.

## Final observations

1. **This is a 99% packaging PR with 1% semantic delta.** The only semantic change to formulas is the `Truth_SymmetricModusPonens` confidence formula (C-2-3). Everything else is line-identical or rearranged.
2. **Bug fixes opportunity missed.** V7, V8, V9 are all unchanged — the consolidation was a perfect chance to fix them but didn't. The comments *calling out* the bugs (`"not according to OpenCOG classic"`, `"Hack to work around the lack of distributional TV"`) are preserved, which at least shows awareness.
3. **Python shim removal is architecturally significant.** Without `PLN.Init` / metta-morph / offline compilation, the system is purely interpreted-at-query-time. This is a simpler mental model but potentially slower for large KBs. Compensates via `MaxSteps=100` default.
4. **Config changes are behaviorally important.** 5x MaxSteps + halved queues will change which queries converge and which time out. Any wiki citation using PLN3 timing numbers should NOT be applied to PLN/.
5. **The three new examples are modest.** `DeductionRevision` tests 2-path+revision (doesn't probe V8); `Direct` introduces a macro front-end (bypasses Deriver); `RavenInduction` adds an induction case to balance FlyingRaven's deduction coverage. Collectively they're incremental coverage, not architectural demonstrations.
6. **PeTTa coupling is the real architectural shift.** The consolidation is downstream of a decision to make PLN a PeTTa-workspace library rather than a standalone MeTTa codebase. `movedPLNfromPeTTa` branch name in the remote refs hints this may have oscillated.
